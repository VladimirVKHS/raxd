# raxd — базовая безопасность (обязательный контракт)

> `raxd` по назначению выполняет произвольные команды по сети — это мощный инструмент
> (как SSH). Поэтому этот чеклист — НЕ пожелание, а контракт. `security` владеет им,
> `developer`/`system-dev`/`devops` обязаны его выполнить, `reviewer` и `security-guardian`
> проверяют по пунктам. Любое отступление — через эскалацию и фиксацию в `threat-model.md`.

## 1. Аутентификация по API-ключам

- [ ] Генерация: `crypto/rand`, ≥ 128 бит энтропии (16+ байт). Никогда `math/rand`.
- [ ] Формат: префикс для опознания (`rax_live_…`) + base64url/hex тела.
- [ ] Хранение: НЕ в открытом виде. Хранить `sha256(key + per-key-salt)` + сам salt.
      (SHA-256 достаточно при высокой энтропии ключа; bcrypt/argon2 — избыточны на каждый запрос.)
- [ ] Сравнение: ТОЛЬКО constant-time — `crypto/subtle.ConstantTimeCompare` / `hmac.Equal`.
      Никаких `==`/`strings.EqualFold` по секретам (timing-атака).
- [ ] Ключ показывается пользователю один раз при `key create`; в `key list` — только id/label/метаданные.
- [ ] Отзыв (`key delete`) мгновенный: помечаем revoked, чистим кэш, дальнейшие запросы → 401/403.

## 2. Транспорт

- [ ] TLS обязателен. `MinVersion: tls.VersionTLS13` (допустимо 1.2 как нижняя граница для legacy).
- [ ] Self-signed серт генерируется при установке/первом запуске (`crypto/x509`); ключ `0600`.
- [ ] mTLS — опционально, для machine-to-machine (`tls.RequireAndVerifyClientCert`).
- [ ] Для HTTP/MCP-эндпоинта: валидация заголовка `Origin` (защита от DNS-rebinding).
- [ ] По умолчанию слушать осознанный интерфейс; если только локально — bind `127.0.0.1`.

## 3. Выполнение команд (самое опасное)

- [ ] Запуск через `exec.Command(bin, args...)` БЕЗ shell-интерполяции. Никогда `sh -c <user-input>`.
- [ ] Таймаут на каждую команду (например, 30s по умолчанию, конфигурируемо) через `context`.
- [ ] Опциональный allowlist команд: строгое сопоставление (не regex), по умолчанию выключен,
      но архитектурно предусмотрен и документирован.
- [ ] Демон работает НЕ от root: выделенный системный пользователь; при необходимости порта <1024 —
      Linux capabilities (`CAP_NET_BIND_SERVICE`), а не setuid root.
- [ ] Рабочая директория/окружение команд ограничены и предсказуемы.

## 4. Аудит и устойчивость

- [ ] Аудит-лог КАЖДОГО действия: timestamp, fingerprint ключа (не сам ключ), команда+аргументы,
      exit code, длительность, удалённый адрес. Структурно (JSON), с ротацией.
- [ ] Логировать неудачные аутентификации и аномалии (всплеск отказов).
- [ ] Rate limiting: per-key и per-IP (`golang.org/x/time/rate`, token bucket), 429 при превышении.
- [ ] Graceful restart сервиса при падении (`Restart=on-failure` в unit/plist).
- [ ] Никаких секретов в логах, в выводе CLI по умолчанию, в commit-истории.

## 5. Дистрибуция

- [ ] Install-скрипт: `set -euo pipefail`, тело в функции (защита от обрыва закачки), `trap` на очистку temp.
- [ ] Проверка `SHA256SUMS` скачанного бинаря (минимум; GPG — плюсом).
- [ ] macOS: подпись + нотаризация для «правильной» дистрибуции; минимум — снятие quarantine
      (`xattr -d com.apple.quarantine`) и понятная инструкция пользователю.

## 6. Среда разработки, тестирования и запуска — только Docker

- [ ] Сборка, тесты (`go build`/`go test`), запуск демона `raxd` и любые ручные прогоны утилиты —
      ТОЛЬКО внутри Docker-контейнера. На хосте `raxd` не запускается: он исполняет произвольные
      команды, его место — изолированный контейнер.
- [ ] Проверка install-flow (`curl | sh`) прогоняется в чистом Linux-контейнере (debian/ubuntu) —
      имитация свежего сервера.
- [ ] В репозитории есть `Dockerfile` (и/или `docker-compose.yml`/devcontainer) для воспроизводимой
      dev/test-среды; команды запуска тестов задокументированы как docker-команды.
- [ ] CI прогоняет сборку и тесты в контейнере.
- Примечание: тесты сервисной интеграции (systemd) — в контейнере с systemd; на хост-машину
      разработчика сервис не ставится.

## Эскалация

Если выполнение пункта невозможно в выбранной архитектуре — НЕ игнорировать молча: завести запись
в `threat-model.md` (риск + почему + смягчение) и эскалировать пользователю до релиза.
