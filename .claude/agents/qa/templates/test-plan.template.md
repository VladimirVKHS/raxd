## Стратегия
Какие уровни тестирования применяем и зачем (1-2 строки на уровень):
- **Unit** — логика модулей (генерация ключа, парсинг, sha256+salt).
- **Integration** — сборка модулей: БД ключей, TLS-хендшейк, middleware аутентификации.
- **E2E** — CLI-команды и `raxd serve` + клиент по сети (happy path сценариев пользователя).
- **Install-flow** — `curl | sh`: детект ОС/арки, выбор архива, проверка SHA256, регистрация сервиса.

## Security-тесты
По `SECURITY-BASELINE.ru.md` — для raxd обязательно:
- Провал аутентификации → 401/403 (неверный ключ, отозванный ключ).
- Rate-limit per-key и per-IP → 429 при превышении.
- Сравнение секретов только constant-time (нет timing-разницы).
- `exec.Command(bin, args...)` без shell-интерполяции (нет `sh -c <user-input>`).
- Отзыв ключа (`key delete`) → последующие запросы отклонены.

## Install-flow тест
- Детект `uname -s`/`uname -m` → выбор корректного архива.
- Проверка `SHA256SUMS`: подмена бинаря → установка падает.
- `set -euo pipefail`, тело в функции (защита от обрыва закачки), `trap` на очистку temp.
- macOS: снятие quarantine; права на бинарь `0755`.

- Install-скрипт: `shellcheck install.sh` + прогон в контейнере/песочнице.
