# CLAUDE.md — правила работы с агентами

В `.claude/agents/` лежат 4 субагента — `pm-spec`, `architect`, `implementer`, `reviewer`. Это канонический pipeline для разработки новых фич: каждая роль имеет свой системный промпт, свой scoped `tools` и свой output. Главный агент (этот контекст) координирует pipeline, но сам код не пишет — он делегирует ролям.

## Правило 1. Когда использовать pipeline

Для **feature-задач** — pipeline в полном составе, по порядку:

```
pm-spec → architect → implementer → reviewer
```

Feature-задача — это:
- новый функционал (новый endpoint, новая команда CLI, новый экран)
- значимое изменение поведения существующего модуля
- любая задача, где результат не очевиден на этапе формулировки

## Правило 2. Когда pipeline НЕ нужен

Для **тривиальных правок** работаем напрямую, без субагентов:
- багфикс с понятной причиной и понятным фиксом
- typo, переименование переменной, форматирование
- мелкий refactor без изменения поведения
- обновление зависимости / конфига

Если задача укладывается в ≤15 минут и не требует обсуждения архитектуры — pipeline избыточен.

## Правило 3. Handoff через `specs/<task-id>/`

Все артефакты pipeline живут в `specs/<task-id>/`:

- `spec.md` — пишет pm-spec
- `plan.md` — пишет architect (читает `spec.md`)
- код в ветке `feat/<task-id>` — пишет implementer (читает `spec.md` + `plan.md`)
- `review.md` — пишет reviewer (читает spec, plan, diff)

Каждая роль читает предыдущие файлы и пишет следующий. Не пропускай шаги — pipeline ломается на следующем звене.

## Правило 4. Ветка — `feat/<task-id>`

Implementer создаёт ветку `feat/<task-id>` (тот же slug, что у папки `specs/<task-id>/`). Atomic commit, тесты до коммита.

## Минимальный пример

Пользователь: «добавь экспорт пользователей в CSV».

Это feature-уровень (новый функционал) — запускаем pipeline:

1. Делегирую `pm-spec` с задачей и `<task-id>` (`export-users-csv`). Получаю `specs/export-users-csv/spec.md`.
2. Если в `Open Questions` остались вопросы — спрашиваю пользователя, дополняю `spec.md`.
3. Делегирую `architect`. Получаю `plan.md`.
4. Делегирую `implementer`. Получаю ветку `feat/export-users-csv` с коммитом.
5. Делегирую `reviewer`. Получаю `review.md` с verdict.
6. Если verdict = `needs-changes` — возвращаюсь к `implementer` с списком issues.
