# Как это работает: от пустого каталога до увиденного эффекта

> Этот документ — практический проход за десять минут. Цель — не объяснить концепцию (для этого есть `GUIDE.md`), а **увидеть** живой pipeline. Берётся пустой каталог, делается простая CLI-программа сначала без кита (один агент), потом с китом (четыре роли) — и параллельно показано, как убедиться, что субагенты реально работают, а не имитируют.

## Что такое субагент с точки зрения процесса

Сначала картинка в голове, без которой остальное не сложится.

Когда вы запускаете `claude` в каталоге, у вас поднимается **один процесс** Claude Code. Этот процесс держит окно контекста — буфер, в который попадают системный промпт, ваши сообщения, ответы модели, результаты вызовов инструментов. Всё, что Claude «помнит» в текущей сессии, — это содержимое этого окна.

Субагент — это **другой процесс** того же Claude Code с **другим окном контекста**. Главный Claude не таскает свой контекст в субагента: он передаёт короткий бриф (один-два параграфа задачи), субагент работает в чистом окне со своим системным промптом из файла `.claude/agents/<name>.md`, и возвращает главному только результат. Главный видит **результат**, а не весь внутренний путь субагента. Это и есть изоляция контекста.

Аналогия — кабинет переводчика-синхрониста. Вы говорите главному ассистенту: «нужен перевод этого документа с японского». Он не лезет в японский сам — стучит в дверь переводчику, передаёт документ, ждёт перевода, отдаёт вам. Переводчик у себя в кабинете делает свою работу со своими словарями. Вы не видите, как он искал термины, — видите готовый перевод. А если у переводчика на столе стоит табличка «без права писать в исходник», он физически не сможет править ваш документ — только переводить и комментировать.

Вот этот переводчик с табличкой — это субагент с tools scoping.

## Сценарий А: чистый каталог, без кита, один агент

Чтобы потом стало ясно, чем кит лучше, начнём с того, что бывает **без** него.

```bash
mkdir hello-cli && cd hello-cli
git init
claude
```

Открылся интерактивный Claude Code. Дальше вы пишете в чат:

> Сделай маленький CLI на Python, который при запуске печатает «Hello world» и плюсом то, что я передал аргументом. Например, `python hello.py Eli` должно напечатать `Hello world Eli`.

Что произойдёт — один процесс Claude:
1. Прочитает каталог (Glob/Read), увидит, что он пустой.
2. Решит, что нужен один файл `hello.py`.
3. Напишет его через Write.
4. Возможно, предложит `python hello.py Test` для проверки.
5. Если согласитесь — прогонит через Bash.

Готово, секунд тридцать. **Здесь субагентов нет**, всё делает главный Claude в одном окне. И это **правильно**. На задаче из десяти строк собирать pipeline из четырёх ролей — это как нанимать четырёх человек, чтобы повесить полку. Кит существует не для таких задач. Кит существует для случаев, когда у вас в голове путаются требования, архитектура, имплементация и проверка, и одного агента уже не хватает.

Запомните эту точку отсчёта: **простую задачу делает один агент, и это нормально**.

## Сценарий Б: тот же каталог, но с китом

Теперь представьте, что задача чуть жирнее. Не «Hello world», а так:

> CLI на Python, который читает имя из аргумента. Если аргумент не передан — спрашивает интерактивно. В имени запрещены символы, ломающие терминал (управляющие коды, например `\x1b`). При пустом вводе — выходим с кодом 1 и понятным сообщением. Есть флаг `--shout`, который превращает результат в заглавные.

Уже не десять строк. Уже есть **требования** (что такое «запрещённые символы»), **выбор подхода** (как валидировать ввод — regex или whitelist), **реализация** (argparse vs raw sys.argv) и **проверка** (что должно тестироваться). Если запустить это одним агентом — он за вас примет все четыре решения по дороге, не спрашивая. Скорее всего получится работающее, но местами не такое, как вы хотели. Edge cases он либо пропустит, либо реализует на свой вкус.

Теперь с китом. Поехали с самого начала.

### Шаг 1. Распаковать кит в проект

Допустим, у вас есть архив `S08-subagents-starter-kit.zip`. Распаковываете и копируете нужное в свой пустой проект:

```bash
mkdir hello-cli && cd hello-cli
git init

# Распаковали кит куда-то рядом, например в ~/Downloads/S08-subagents-starter-kit/
cp -r ~/Downloads/S08-subagents-starter-kit/.claude .
cp    ~/Downloads/S08-subagents-starter-kit/CLAUDE.md .
cp    ~/Downloads/S08-subagents-starter-kit/.gitignore .
```

Если у вас уже был свой `CLAUDE.md` в репо — не копируйте поверх, откройте оба и перенесите блок про субагенты из шаблона кита к себе. Иначе `cp` молча перетрёт ваши собственные правила.

Что теперь в репо:

```
hello-cli/
├── .claude/
│   └── agents/
│       ├── pm-spec.md
│       ├── architect.md
│       ├── implementer.md
│       └── reviewer.md
├── CLAUDE.md
└── .gitignore
```

`.claude/agents/` — это место, куда Claude Code сам смотрит при старте. Каждый `.md` файл с правильным YAML наверху — отдельный субагент. Имена в поле `name` — это идентификаторы, по которым их можно звать.

`CLAUDE.md` — это правила для **главного** Claude. Он его автоматически загружает при старте сессии в этом каталоге. Внутри написано: «у тебя есть pm-spec, architect, implementer, reviewer, для новых фич используй pipeline в порядке pm-spec → architect → implementer → reviewer, для багфиксов — работай напрямую».

### Шаг 2. Проверить, что субагенты подхватились

```bash
claude
```

В интерактивном Claude Code напечатайте:

```
/agents
```

Это встроенная команда, она показывает список доступных субагентов. Вы должны увидеть четыре строки — `pm-spec`, `architect`, `implementer`, `reviewer` — каждая со своим описанием из поля `description`.

Если их нет — значит файлы лежат не в `.claude/agents/`, либо YAML битый. Если есть — кит установился, всё в силе.

### Шаг 3. Первый запрос — главный Claude сам делегирует

Пишете в чат точно так же, как раньше, без всякой магии:

> Хочу CLI на Python. Принимает имя как аргумент или интерактивно, если аргумента нет. Печатает «Hello world `<имя>`». Есть флаг `--shout`, который делает результат заглавными. На пустое имя — выход с кодом 1 и сообщением «name required». Распиши, что нужно сделать.

Обратите внимание на **последнюю фразу**: «распиши, что нужно сделать». Это семантический триггер. Главный Claude видит описание `pm-spec` в его frontmatter — «превращает запрос или идею в техническое задание со списком acceptance criteria» — и понимает, что это работа именно для PM-Spec. Он **сам** инициирует вызов субагента, вы ничего специально не пишете.

В выводе вы увидите блок типа:

```
Это feature-задача, начинаю с pm-spec. Делегирую...

[Task(pm-spec): создаёт specs/HELLO-001/spec.md]
```

Дальше — **результат**. PM-spec возвращает заполненный `spec.md` с секциями Goal / User Stories / Acceptance Criteria / Out of Scope / Open Questions. Файл реально появляется на диске в `specs/HELLO-001/`. Вы можете открыть его в редакторе и почитать.

Это первое доказательство, что субагент работает: **на диске есть файл, который написал не главный Claude, а его помощник**.

### Шаг 4. Дальше по pipeline

Прочитали `spec.md`, всё устраивает. Пишете главному Claude:

> Окей, теперь продумай архитектуру.

Опять семантика — слово «архитектуру» матчится с `description` у architect. Главный Claude делегирует:

```
[Task(architect): читает specs/HELLO-001/spec.md, пишет plan.md]
```

В `specs/HELLO-001/plan.md` появляется план: один выбранный подход (например, использовать стандартный `argparse`, без сторонних зависимостей), какие функции нужны, какие edge cases как обрабатываются, какие компромиссы выбраны.

Дальше:

> Реализуй.

Это implementer. У него в `tools` есть `Write`, `Edit`, `Bash` — он создаст `hello.py`, прогонит через `python hello.py Test`, сделает atomic commit. Если в репо настроены pre-commit hooks или есть `pytest` — он их прогонит тоже.

И финал:

> Проверь, что получилось, против spec и plan.

Это reviewer. У него **нет** `Write`/`Edit`/`Bash`. Он откроет `hello.py` через Read, сверит с `spec.md` и `plan.md`, и напишет `specs/HELLO-001/review.md` со списком найденных проблем. Например:

- «На вводе `--shout` без аргумента имя запрашивается интерактивно, но shout не применяется к интерактивному вводу — не покрыто acceptance criterion».
- «Нет теста на пустой `--shout ""`».

Reviewer **не правит код сам**. Он показывает список. Вы передаёте его обратно Implementer'у:

> Вот замечания от reviewer. Поправь.

И implementer закрывает issues.

## Что вы увидите физически, чтобы поверить

Три осязаемых сигнала, что субагенты реально работают, а не имитируют.

**Первый — папка `specs/HELLO-001/` появилась сама.** Никто из людей её не создавал. Файлы внутри написаны разными ролями, и это видно по стилю: `spec.md` про требования, `plan.md` про модули и контракты, `review.md` про проблемы. Откройте их в редакторе — содержимое разное по структуре, разное по словарю. Это не один и тот же текст, просто переименованный.

**Второй — попытка дать reviewer'у написать что-то возвращает ошибку.** Можно проверить руками. После того как reviewer закончил, напишите главному Claude:

> Попроси reviewer'а сразу исправить найденные проблемы.

Главный Claude попытается делегировать в reviewer, у того в `tools` нет Edit. В выводе вы увидите примерно:

```
[reviewer attempted to call Edit — tool not allowed]
```

или агент сразу скажет «не могу, у меня нет такого инструмента, я только читаю». Это не вежливость и не запрограммированный отказ. Это уровень разрешений — попытка вызвать инструмент проверяется **до** самого вызова. Прямой аналог HTTP 403 в API: «у тебя нет такого permission».

**Третий — `/agents` показывает их с метаданными.** Покажите команду в Claude Code ещё раз:

```
/agents
```

Увидите примерно:

```
pm-spec     (sonnet)  Превращает запрос в техзадание со списком AC...
architect   (opus)    Превращает spec.md в архитектурный план...
implementer (sonnet)  Пишет код по spec + plan, прогоняет тесты...
reviewer    (opus)    Проверяет код против spec + plan, не правит сам...
```

У каждого свой `model` — это значит **разные модели работают над разными ролями**. Architect на opus, потому что ему думать важнее, чем писать. Implementer на sonnet — быстрее и дешевле для рутины. Это видно сразу, без копания в файлах.

## Когда писать «расскажи», а когда `@pm-spec`

Выше я дважды показал, что главный Claude **сам** определяет, какого субагента звать, по семантическому совпадению с `description`. Это **автоматический режим**. Работает в 80% случаев, если писать нормальным языком.

Иногда не срабатывает. Например, вы сказали «продумай», а главный Claude решил, что «продумывание» — это implementer (он же может сам всё продумать и сделать). Тогда **явный режим**:

```
@architect продумай архитектуру для этой задачи
```

`@<name>` — это явный pin. Главный Claude обязан позвать именно эту роль, даже если description не идеально совпадает.

Третий способ — фразой, без `@`:

```
Используй architect, чтобы продумать архитектуру.
```

Это работает чуть хуже `@`, но в большинстве случаев тоже выруливает.

Правило, которое имеет смысл применять: **сначала пишите нормальным языком, без `@`. Если главный Claude позвал не ту роль или вообще не позвал — переключайтесь на `@`.** Со временем накопится наблюдение, какие формулировки у вас триггерят какие роли, и явный режим понадобится всё реже.

## Что должно быть в `CLAUDE.md`, чтобы всё это работало

Один файл, который вы копируете в корень репо вместе с китом. Внутри — несколько правил для главного Claude:

```markdown
# CLAUDE.md — правила работы с агентами

В .claude/agents/ лежат четыре субагента: pm-spec, architect, implementer, reviewer.

Это pipeline для новых фич. Используй его так:

- Для feature-задач (новый функционал, новая команда CLI, новый endpoint):
  pm-spec → architect → implementer → reviewer.
  Handoff-файлы пиши в specs/<task-id>/.

- Для багфиксов и тривиальных правок (typo, переименование, мелкий refactor):
  pipeline не нужен. Работай напрямую.

- Ветки именуй feat/<task-id> (это инструкция для implementer).

Когда сомневаешься — спроси, использовать pipeline или нет.
```

Главный Claude его автоматически читает при старте. Без этого файла он **знает**, что у него есть субагенты, но **не знает**, что хотите делать вы — будет звать их непоследовательно или вообще игнорировать. Полный шаблон лежит в корне кита, скопируйте к себе и при необходимости подправьте под свой проект.

## Что попробовать прямо сейчас

Прогон, чтобы реально увидеть pipeline за десять минут:

1. `mkdir hello-cli && cd hello-cli && git init`
2. Распакуйте кит туда (или копируйте из репо проекта)
3. Запустите `claude`, наберите `/agents` — увидьте четыре роли
4. Напишите: «Сделай CLI на Python с флагом `--shout`. Принимает имя как аргумент. Распиши требования.»
5. Откройте в редакторе файл `specs/<id>/spec.md`, который появился — это работа pm-spec, в чистом виде
6. Вернитесь в чат, напишите: «Теперь продумай архитектуру»
7. Посмотрите на `plan.md` — это уже architect
8. «Реализуй» — implementer пишет `hello.py`, делает commit
9. «Проверь» — reviewer пишет `review.md`
10. Финал — `git log` показывает один коммит от implementer'а, и три markdown-файла в `specs/<id>/` рядом

Этих десяти минут хватает, чтобы понять: субагенты — это не магия и не теория, а **четыре файла в `.claude/agents/`**, на которые Claude Code смотрит при старте и которые он умеет звать сам.

Когда вы увидите, что `review.md` написан в другом тоне, чем `spec.md` — потому что у них разные системные промпты — это будет момент, когда теория станет ощущением.

## Что дальше

Дальше — открыть `GUIDE.md` и прочитать его подряд. Там разобрано, **почему** именно такие четыре роли, что такое tools scoping глубже, какие бывают типичные затыки и как их чинить. После этого документа `GUIDE.md` будет ложиться легче, потому что у вас уже есть тактильный опыт.

И когда захочется попробовать на реальной задаче из своего проекта — откройте `specs/EXAMPLE-001/` в этом ките и посмотрите, как выглядит pipeline на содержательной фиче (экспорт пользователей в CSV). Это ваш ориентир по уровню детализации.
