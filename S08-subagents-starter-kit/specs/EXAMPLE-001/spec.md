# Экспорт пользователей в CSV

## Goal

Дать администраторам возможность выгружать список активных пользователей в CSV-файл для использования в отчётах и аналитике. Экспорт должен работать в один клик, без настроек со стороны пользователя в первой итерации — фокус на быстром закрытии запроса финансового отдела и команды поддержки.

## User Stories

- Как администратор биллинга, я хочу экспортировать список активных пользователей в CSV, чтобы передать его финансистам для сверки подписок и MRR.
- Как руководитель отдела поддержки, я хочу получить файл со всеми активными аккаунтами, чтобы рассчитать нагрузку на команду и спланировать смены.
- Как product analyst, я хочу одной кнопкой получить актуальный snapshot активных пользователей, чтобы загрузить его в Google Sheets для ad-hoc анализа без обращения к разработчикам.

## Acceptance Criteria

- [ ] Endpoint доступен по `GET /api/users/export.csv` с авторизацией через стандартный Bearer-токен.
- [ ] Доступ только у пользователей с ролью `admin` — остальные роли получают `403 Forbidden`.
- [ ] В CSV колонки в порядке: `id, email, name, created_at, status`. Заголовки в первой строке.
- [ ] Возвращаются только пользователи со статусом `active` — записи со статусом `suspended` или `deleted` отфильтрованы.
- [ ] Поля, содержащие запятые, кавычки, переводы строк, экранируются по RFC 4180 (двойные кавычки вокруг поля, удвоение внутренних кавычек).
- [ ] Response имеет header `Content-Disposition: attachment; filename="users-export-YYYY-MM-DD.csv"`, где дата — текущая в UTC.
- [ ] При ошибке базы данных endpoint возвращает `500` с логированием полного stack trace на сервере; CSV-файл не отдаётся клиенту частично.

## Out of Scope

- Фильтрация по диапазону дат регистрации, по тарифу, по тенанту — откладывается до следующей итерации, когда соберём реальные запросы от админов.
- Экспорт в форматах JSON или Excel (.xlsx) — только CSV; альтернативные форматы будут отдельной задачей при наличии спроса.
- Асинхронный экспорт через очередь с email-уведомлением — текущая база ~50k активных пользователей, синхронный streaming-ответ укладывается в timeout и работает без очереди.
- UI-кнопка в админ-панели — бэкенд endpoint поставляется первым, UI делается отдельной задачей фронтенд-командой после готовности API.

## Open Questions

- Нужно ли логировать сам факт экспорта в audit log (кто экспортировал, когда, сколько строк)? Спросить у legal/security перед merge — возможно есть GDPR-обязательства.
- Какой timezone использовать для `created_at` в CSV — текущий тенант или UTC? В первой итерации пишем UTC и в comment к endpoint помечаем это; при необходимости добавим параметр позже.
