# Пример: экспорт пользователей в CSV

Это пример выхода 4-ролевого pipeline (pm-spec → architect → implementer → reviewer) на одной конкретной задаче. Контекст: SaaS-приложение, в админке есть таблица активных пользователей, и финансовому отделу понадобилась выгрузка для сверки и отчётности. Бизнес-владелец сформулировал короткое требование «дайте админам кнопку экспорта в CSV» — дальше задача прошла через четыре роли, и здесь видно, как она оформилась в spec, plan и review.

Стек примера — TypeScript + Express + PostgreSQL для конкретности. В Python (FastAPI/SQLAlchemy), Go (chi/pgx) или Rust (axum/sqlx) модули и имена файлов будут другими, но структура артефактов и сам pipeline идентичны: spec описывает «что», plan описывает «как», implementer пишет код, reviewer ищет дыры.

## Как читать

1. **`spec.md`** — что делаем. Цель, user stories, acceptance criteria. Без кода и без «как».
2. **`plan.md`** — как делаем. Один выбранный подход с обоснованием, модули, контракты, trade-offs.
3. **`review.md`** — что не так с реализацией. Конкретные issues с указанием места, причины и фикса.

## Замечания

- В `review.md` найдены 2-3 проблемы. Это нормально и хорошо: задача reviewer'а — находить упрощения и риски, а не штамповать accept. Если первая итерация review всегда даёт «всё ок» — у вас сломан reviewer, а не идеальный код.
- Implementer не оставил здесь markdown-артефакта — его выход это git commits в коде проекта, которые в этом примере не воспроизводятся.
- **Важно про ссылки на код.** Конкретные `file:line` ссылки в `plan.md` и `review.md` (например `src/routes/users/export.ts:28`) — **иллюстративные**. Соответствующего кода в этом репозитории нет, и попытка открыть файл по такому пути ничего не найдёт. Это пример того, как Reviewer формулирует претензии — где, почему, что делать. В реальном прогоне на вашем проекте эти ссылки будут указывать на настоящие файлы в вашем исходнике.
- Open Questions в `spec.md` и issues «для обсуждения» в `review.md` — это не баги pipeline, а нормальные точки эскалации к человеку. Subagent не должен их решать самостоятельно.
