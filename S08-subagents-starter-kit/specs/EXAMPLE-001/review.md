# Review: экспорт пользователей в CSV

## Summary

Реализация в основных моментах следует плану: streaming через `csv-stringify`, keyset-пагинация по `id`, авторизация через существующий middleware `requireRole('admin')`. Найдено два места, требующих изменений до merge, и одно открытое для обсуждения с командой. Покрытие тестами оставляет gaps по edge cases экранирования и пустой выборке. Структура кода чистая, имена понятные, хорошо разделены route handler и сервисный слой. Verdict: `needs-changes`.

> **Примечание читателю примера**: `file:line` ссылки в Issues ниже — иллюстративные, реального кода в этом репозитории нет. Этот файл показывает **формат** вывода Reviewer'а на реалистичной задаче. См. `README.md` рядом для контекста.

## Issues

- [ ] **Issue: Нет теста на пустой результат (нет активных пользователей в БД)**
  - Где: `src/routes/users/export.spec.ts`
  - Почему: Acceptance criteria требует, что endpoint возвращает 200 с корректным CSV в любом случае. На пустой выборке мы должны вернуть файл с одним заголовком и нулём строк данных. Сейчас этот сценарий не покрыт тестом, и из кода неочевидно, поведёт ли streaming-логика себя корректно — возможно, заголовок не запишется до первого batch, и клиент получит пустой response без header'а.
  - Что делать: добавить test case `returns CSV with header row only when no active users exist`. Проверить, что body ровно `id,email,name,created_at,status\r\n` и Content-Type выставлен.

- [ ] **Issue: Тест экранирования покрывает только запятые и обычные кавычки**
  - Где: `src/routes/users/export.spec.ts:67`
  - Почему: В продовой БД у части старых записей `name = NULL`, у трёх пользователей в name встречается `\r` (артефакт импорта из старой CRM), и потенциально могут быть переводы строк `\n` внутри полей. `csv-stringify` в strict mode корректно обрабатывает эти случаи, но если завтра кто-то решит заменить библиотеку или переписать на ручное экранирование — текущий тест не поймает регрессию.
  - Что делать: расширить тест экранирования на cases: `null` в name, name с `\r`, name с `\n`, name с двойными кавычками, name с unicode-символами. Каждый case — отдельный it-блок с понятным названием.

- [ ] **Issue: Connection не закрывается явно при ошибке посередине стрима**
  - Где: `src/routes/users/export.ts:28`
  - Почему: Если БД-запрос упадёт после того как мы уже отдали клиенту 5000 строк, текущий код пишет `res.status(500).json(...)` в catch-блоке — но статус уже отправлен (200), header'ы зафиксированы, и Express в лучшем случае проигнорирует второй write, в худшем — выбросит `ERR_HTTP_HEADERS_SENT`. Под HTTP/2 multiplexing stream может остаться в half-closed state, держа соединение открытым до timeout.
  - Что делать: в catch-блоке после логирования вызвать `res.destroy(error)` чтобы явно оборвать stream. Клиент получит truncated response и поймёт, что что-то не так. Дополнительно — добавить unit-тест с моком БД, который кидает ошибку после 100 строк, и проверить, что соединение закрыто.

## Looks good

- Авторизация через существующий middleware `requireRole('admin')` — без копи-пасты логики проверки роли, переиспользование того, что уже протестировано.
- Keyset-пагинация по `id` реализована корректно: WHERE-clause использует `id > :last_id`, ORDER BY стабильный, batch_size конфигурируемый. Индекс на `id` существует как PRIMARY KEY.
- Header `Content-Disposition` с датой в имени файла соответствует AC и удобен в работе — админ скачает несколько раз и не перезапишет предыдущие выгрузки.
- Service-метод `streamActiveUsers` возвращает `AsyncIterable`, что даёт чистую интеграцию с `for await` в route handler — никаких ручных колбэков и promise-цепочек.

## Verdict

`needs-changes` — два требующих обязательного изменения issue (empty result test и connection cleanup при ошибке) и одно для обсуждения с командой (расширение покрытия тестов экранирования). После исправления первых двух можно идти на повторный review; третий пункт можно закрыть в follow-up задаче, если команда согласится.
