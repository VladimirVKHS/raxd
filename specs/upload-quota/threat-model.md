# Threat Model: upload-quota — общий лимит объёма (disk quota) на `upload_file`

Автор продукта: Vladimir Kovalev, OEM TECH
Вход: `spec.md` (PM), `plan.md` (architect), `SECURITY-BASELINE.ru.md`, `MCP-INTEGRATION.ru.md`.

> Фича расширяет уже готовый `upload_file`, наследуя auth/Origin/rate-limit/аудит/атомарную запись
> (file-upload: SR-68…SR-82). Эта модель фокусируется на НОВОЙ поверхности — общем лимите объёма
> upload root: подсчёт обходом `os.Root` под мьютексом на root, deny ДО фиксации, fail-closed.
> Угрозы, уже закрытые file-upload, здесь не переписываются — фиксируется только то, что фича
> добавляет или обязана НЕ ослабить.

## Активы
- **Дисковое пространство хоста под upload root** — главный защищаемый ресурс этой фичи. Его
  исчерпание выводит из строя сам демон и соседние сервисы на хосте (хост-DoS).
- **Целостность существующих файлов в upload root** — новая загрузка/проверка квоты не должна
  повреждать или удалять ранее записанные файлы.
- **Аудит-лог** (наблюдаемость отказов по квоте) — должен честно фиксировать deny, не теряя записи и
  не пропуская обход лимита.
- **Конфиденциальность путей/чисел/секретов** — сообщение об отказе и аудит не должны раскрывать
  абсолютные пути ФС, точную раскладку/числа объёма, тело ключа или содержимое файла.
- **Доступность `upload_file`** — сама механика подсчёта (обход под мьютексом) не должна превращаться
  в вектор отказа в обслуживании самого инструмента.
- **Инвариант traversal-безопасности `os.Root`** (унаследован от SR-69) — обход для подсчёта не должен
  разыменовывать симлинки наружу или считать что-либо вне корня.

## Поверхность атаки
- **Сетевой вход**: MCP-tool `upload_file` поверх Streamable HTTP/TLS (MCP-INTEGRATION). Аутентифи-
  кация по ключу, Origin, rate-limit отрабатывают ДО tool (наследовано). Новая логика — внутри
  `fileupload.Write`.
- **Конфиг**: новый параметр `upload.max_total_bytes` (`config.yaml`). Невалидное значение (отрица-
  тельное) — вход для отказа сервиса на старте.
- **Файловая система upload root**: содержимое корня (число/размер файлов, симлинки, подкаталоги) —
  вход для логики обхода `currentBytes` (`fs.WalkDir` через `os.Root.FS()`).
- **Конкурентность**: одновременные вызовы `upload_file` в один root — вход для гонки (TOCTOU) между
  проверкой объёма и записью.
- **Кто атакует**: клиент с ВАЛИДНЫМ ключом (главная модель — внутренний/полу-доверенный клиент, как
  у SSH; цель — забить диск или обойти лимит). Внешний без ключа отсекается наследуемой auth.
  Локальный пользователь, кладущий файлы/симлинки в root мимо демона, — учитывается в подсчёте.

## Угрозы (STRIDE-ориентир)
- **S**poofing: вне scope фичи — auth/Origin/rate-limit наследованы без изменений (AC11). Новых
  путей аутентификации фича не вводит.
- **T**ampering:
  - подмена/повреждение существующих файлов при проверке квоты или при частичной/temp-записи на
    отказе (T-Q1);
  - подкладка симлинка наружу в upload root, чтобы обход `currentBytes` разыменовал/посчитал чужой
    объём или вышел за корень (T-Q2).
- **R**epudiation:
  - отказ по квоте без аудит-записи или с потерей записи → факт отказа ненаблюдаем (T-Q3).
- **I**nformation disclosure:
  - утечка абсолютного пути ФС / раскладки / точных чисел объёма / содержимого файла / секрета через
    сообщение об отказе или аудит-запись (T-Q4).
- **D**enial of service:
  - «забить диск» множеством мелких валидных файлов — целевая угроза, ради которой фича существует
    (T-Q5);
  - обход лимита через гонку (TOCTOU): параллельные загрузки проскакивают проверку и совокупно
    превышают лимит (T-Q6);
  - молчаливый обход лимита при ошибке обхода/подсчёта (fail-open) (T-Q7);
  - отказ в обслуживании самим обходом: O(N) `WalkDir` под мьютексом сериализует все upload в один
    root; большой/глубокий root или много параллельных вызовов → деградация/таймауты (T-Q8);
  - отказ сервиса на старте из-за невалидного `max_total_bytes`, либо паника/5xx при отказе по
    квоте (T-Q9);
  - регресс совместимости: жёсткий дефолт внезапно начнёт отклонять прежде проходившие загрузки
    после апгрейда (T-Q10).
- **E**levation of privilege: новых привилегий фича не вводит; запись по-прежнему только через
  `os.Root` относительными путями (SR-69 не ослаблен). Отдельной EoP-угрозы у фичи нет — `None`.

## Риски (риск → вероятность/влияние → смягчение)
- [ ] **R-Q1 (Tampering, T-Q1):** отказ по квоте или ошибка записи оставляет частичный/temp-файл или
      повреждает существующие. → вероятность средняя / влияние среднее → смягчение: проверка квоты
      выполняется ДО создания temp-файла; вся запись атомарна (temp→Sync→Rename→fsync-dir,
      наследовано SR-74); при deny ни temp, ни целевого файла не остаётся, существующие не меняются
      (SR-90, SR-95). (baseline §3 валидация ввода; AC3/AC10)
- [ ] **R-Q2 (Tampering, T-Q2):** симлинк/не-regular в upload root заставляет обход разыменовать
      ссылку, посчитать чужой объём или выйти за корень. → вероятность средняя / влияние среднее →
      смягчение: обход идёт ТОЛЬКО через `os.Root.FS()`; симлинки и не-regular файлы НЕ разыменовы-
      ваются и НЕ учитываются; суммируются только regular-файлы внутри корня (SR-93, не ослабляет
      SR-69). (baseline §3; AC6)
- [ ] **R-Q3 (Repudiation, T-Q3):** отказ по квоте не попадает в аудит или пишется не как deny → риск
      невидим. → вероятность низкая / влияние среднее → смягчение: на каждое отклонение по квоте —
      РОВНО одна основная аудит-запись `Result:"deny"` с reason про общий лимит, без потери и дубля
      (SR-94). (baseline §4; AC5)
- [ ] **R-Q4 (Information disclosure, T-Q4):** сообщение/аудит отказа раскрывает абсолютный путь,
      точные числа объёма, содержимое файла или секрет. → вероятность средняя / влияние высокое →
      смягчение: нейтральное сообщение «total upload quota exceeded» без путей/чисел; аудит — без
      абсолютного пути (допустим только относительный `Path`), без точных чисел, без содержимого, без
      тела ключа (fingerprint вместо ключа). Проверяемо grep по ответу и аудит-логу (SR-91, SR-94,
      не ослабляет SR-80). (baseline §4; AC4/AC5/Q6)
- [ ] **R-Q5 (DoS, T-Q5 — целевая угроза):** клиент с валидным ключом забивает диск множеством мелких
      файлов (per-file лимит не спасает от суммарного объёма). → вероятность высокая / влияние высокое
      → смягчение: при `max_total_bytes > 0` загрузка, выводящая суммарный объём за лимит,
      отклоняется ДО фиксации; общий лимит независим от и не ослабляет `max_file_bytes` (оба
      действуют) (SR-90, SR-97). ОСТАТОЧНО: лимит ОБЩИЙ, не per-key — см. остаточные риски ниже.
      (baseline §4 устойчивость/исчерпание ресурсов; AC3/AC9)
- [ ] **R-Q6 (DoS/обход, T-Q6 — TOCTOU):** параллельные загрузки между проверкой и записью совокупно
      превышают лимит. → вероятность средняя / влияние высокое → смягчение: проверка объёма И вся
      запись выполняются атомарно под ОДНИМ `sync.Mutex` на абсолютный `UploadRoot` (реестр
      `sync.Map`); Lock берётся ДО `os.OpenRoot`, Unlock через defer после фиксации/ошибки. При любом
      наборе одновременных вызовов итог НИКОГДА не превышает лимит (SR-92). (baseline §3/§4; AC7)
- [ ] **R-Q7 (DoS/обход, T-Q7 — fail-open):** ошибка обхода/`Info()`/`Stat` трактуется как «объём
      неизвестен → пропустить» → молчаливый обход лимита. → вероятность низкая / влияние высокое →
      смягчение: fail-closed — ЛЮБАЯ ошибка обхода/подсчёта возвращается наверх, запись НЕ выполняется,
      handler маппит её в `Result:"fail"`; молчаливого обхода нет (SR-96). (baseline §4; AC10/Q4)
- [ ] **R-Q8 (DoS самим обходом, T-Q8):** O(N) `WalkDir` под мьютексом сериализует все upload в один
      root; на большом/глубоком корне или при множестве параллельных вызовов — деградация/таймауты
      загрузок. → вероятность низкая / влияние низкое-среднее → смягчение: при `max_total_bytes == 0`
      (дефолт) обход/мьютекс НЕ задействуются — нулевая цена по умолчанию; при включённом лимите цена
      O(N) на загрузку и сериализация записи в один root приняты как осознанный trade-off (upload —
      редкая операция, root одиночного клиента скромен; см. plan «Trade-offs»). ПРИНЯТО как
      приемлемый компромисс, зафиксировано в остаточных рисках. (baseline §4; AC2/AC7)
- [ ] **R-Q9 (DoS, T-Q9):** невалидный `max_total_bytes` или отказ по квоте роняет сервис (паника/5xx)
      или валит старт неожиданно. → вероятность низкая / влияние среднее → смягчение: отрицательное
      значение отвергается на старте (как `max_file_bytes`); отказ по квоте не паникует и не отдаёт
      5xx — `isError:true`, следующий валидный вызов отрабатывает; edge-cases (ровно лимит, впритык,
      `0<max_total<max_file`) дают корректный исход (SR-98, SR-99). (baseline §3/§4; AC1/AC10)
- [ ] **R-Q10 (DoS/совместимость, T-Q10):** жёсткий дефолт после апгрейда внезапно отклоняет прежде
      проходившие загрузки. → вероятность средняя / влияние среднее → смягчение: дефолт
      `max_total_bytes=0` = лимит ОТКЛЮЧЁН; без явной настройки поведение `upload_file` не меняется,
      наследуемые auth/Origin/rate-limit/аудит-ветки не трогаются (SR-99, SR-100). (baseline §4; AC2/AC11)

## Остаточные риски / эскалации
- [ ] **RR-Q1 — нет per-key / per-fingerprint квоты (явный остаточный риск).** Лимит ОБЩИЙ на upload
      root: один клиент с валидным ключом всё ещё может выбрать ВЕСЬ общий лимит и тем лишить других
      клиентов возможности загрузки (DoS-«занять всю квоту»), хотя диск хоста уже не переполнит.
      Почему так: per-key квота — отдельная задача (Out of Scope spec). Смягчение/компенсирующий
      контроль: наследуемый per-key/per-IP rate-limit (baseline §4) ограничивает темп; общий лимит
      защищает диск хоста (главный актив). ЭСКАЛИРОВАТЬ пользователю: если модель угроз включает
      взаимно-недоверенных клиентов с ключами — нужна отдельная задача per-key квоты до релиза такого
      сценария.
- [ ] **RR-Q2 — нет FS/ОС-quota как замены (явный остаточный риск, baseline §6-контекст).** Защита —
      на уровне ПРИЛОЖЕНИЯ; обход хоста мимо демона (другой процесс/пользователь пишет в тот же том)
      приложением не контролируется. Почему так: filesystem quota / cgroup / `--storage-opt` —
      эксплуатационная мера ОС, Out of Scope spec. Смягчение/компенсирующий контроль: подсчёт
      авторитетно учитывает ВСЕ существующие файлы в root (включая записанные мимо демона, AC6), а
      запуск только в Docker (baseline §6) изолирует том. ЭСКАЛИРОВАТЬ как операционную рекомендацию:
      на проде дополнительно настроить FS/контейнерную квоту как defense-in-depth.
- [ ] **RR-Q3 — сериализация upload одним мьютексом на root (производительность, T-Q8).** При
      включённом лимите параллельные загрузки в один root исполняются строго последовательно (обход
      O(N) + запись под одним мьютексом). ПРИНЯТО как осознанный trade-off ради авторитетности и
      простоты корректности (см. plan «Trade-offs»); при `max_total_bytes=0` цена нулевая. НЕ требует
      эскалации до релиза; зафиксировано для наблюдения (если профиль нагрузки изменится — пересмотр
      на счётчик с durable-состоянием).
- Прочие пункты baseline §1/§2/§5 — вне поверхности этой фичи (наследованы без изменений от готового
  `upload_file`/сервиса); фича их НЕ ослабляет. Остаточных рисков по ним фича не добавляет — `None`.
