# Spec: upload-quota — общий лимит объёма (disk quota) на `upload_file`

Автор продукта: Vladimir Kovalev, OEM TECH

## Goal
Закрыть на уровне приложения остаточный риск `docs/production-readiness.md` §7: у `upload_file` есть
только per-file лимит (`upload.max_file_bytes`, дефолт 700 KiB), но НЕТ ограничения на СУММАРНЫЙ
объём в upload root — клиент с валидным ключом может забить диск множеством мелких файлов. Вводим
конфигурируемый общий лимит объёма upload root: при превышении загрузка отклоняется с понятной
ошибкой и аудит-записью, без падения сервера и без утечки секретов. Фича расширяет поведение уже
готового `upload_file` (наследует auth/Origin/rate-limit/аудит/атомарную запись, НЕ переписывая их) и
подчинена `SECURITY-BASELINE.ru.md` (валидация §3, аудит §4, Docker §6).

## User Stories
- Как администратор `raxd`, я хочу задать общий лимит объёма upload root через конфиг, чтобы один
  клиент с валидным ключом не мог заполнить диск множеством файлов.
- Как специалист по безопасности, я хочу, чтобы превышение общего лимита отклонялось с аудит-записью
  (без содержимого и без секретов) и не приводило к отказу сервера, чтобы риск был наблюдаемым.
- Как существующий пользователь, я хочу, чтобы без явной настройки лимита поведение `upload_file` не
  менялось, чтобы апгрейд не сломал текущие сценарии.

## Acceptance Criteria
- [ ] AC1 (новый конфиг-параметр): в секцию `upload` добавлен общий лимит (предлагаемое имя
      `upload.max_total_bytes`, целое, байты). Переопределяем; невалидное значение (напр.
      отрицательное) отвергается на старте, как `max_file_bytes`. Тест: дефолт применяется при
      отсутствии config.yaml; переопределение работает; отрицательное значение отклоняется на загрузке.
- [ ] AC2 (дефолт = без лимита, обратная совместимость): дефолт `max_total_bytes=0` означает «лимит
      ОТКЛЮЧЁН» (риск в §7 сейчас accepted; жёсткий дефолт мог бы внезапно отклонять загрузки после
      апгрейда). Тест: при `0` (или отсутствии параметра) ни одна загрузка не отклоняется по общему
      лимиту, сколько бы файлов ни было записано.
- [ ] AC3 (отклонение при превышении): при `max_total_bytes > 0` загрузка, после которой суммарный
      объём upload root превысил бы лимит, отклоняется до фиксации: `isError:true`, новый (в т.ч.
      частичный/временный) файл на диске не остаётся, существующие файлы не изменяются. Тест: загрузка,
      вписывающаяся в остаток лимита, проходит; загрузка, выводящая суммарный объём за лимит,
      отклоняется и файл не появляется.
- [ ] AC4 (понятное нейтральное сообщение): сообщение об отклонении объясняет причину (исчерпание
      общего лимита/квоты upload root) и НЕ раскрывает абсолютные пути ФС, раскладку и секреты (SR-80).
      Тест: тело ответа содержит понятную причину «общий лимит/квота», не содержит абсолютного пути и
      тела ключа/секрета как подстроки.
- [ ] AC5 (аудит отклонения = DENY): отклонение по общему лимиту пишет ровно одну основную
      аудит-запись с `Result:"deny"` (квота — это правило отказа, как traversal/exists/too-large, а не
      I/O-сбой `fail`): timestamp(UTC), fingerprint (НЕ ключ), `Tool:"upload_file"`, относительный
      `Path` (если известен), `Reason` про общий лимит, `RemoteAddr`. Содержимое файла в аудит НЕ
      пишется (SR-80). Тест: на одно отклонение — ровно одна deny-запись с reason про общий лимит;
      содержимое файла подстрокой в логе отсутствует.
- [ ] AC6 (учёт существующих файлов): при подсчёте суммарного объёма учитывается фактический объём
      всех ранее загруженных файлов в upload root (включая записанные до включения лимита и в
      подкаталогах), а не только файлы за время жизни процесса. Тест: при непустом upload root, уже
      близком к лимиту, новая загрузка, выводящая за лимит, отклоняется — учёт не «забывает» ранее
      существовавшие файлы.
- [ ] AC7 (конкурентная безопасность): при любом наборе одновременных вызовов `upload_file` итоговый
      суммарный объём upload root НИКОГДА не превышает `max_total_bytes` (с точностью до семантики
      границы, Q3). Тест: N параллельных загрузок, суммарно превышающих остаток квоты, оставляют
      upload root в пределах лимита (часть отклонена deny), без повреждённых/частичных файлов.
- [ ] AC8 (учёт при overwrite): при `overwrite:true` изменение учитывается как дельта (новый размер
      минус заменяемый), не как полный новый размер. Тест №1: при исчерпанной квоте перезапись файла на
      содержимое того же/меньшего размера проходит. Тест №2: перезапись на больший размер, выводящий
      суммарный объём за лимит, отклоняется (deny), исходный файл не изменён.
- [ ] AC9 (независимость от per-file лимита): общий лимит НЕ заменяет и НЕ ослабляет `max_file_bytes`;
      оба действуют. Тест: файл, проходящий per-file лимит, но выводящий суммарный объём за
      `max_total_bytes`, отклоняется; файл, нарушающий `max_file_bytes`, отклоняется как и раньше (без
      регресса).
- [ ] AC10 (сервер жив, edge-cases): отклонение по лимиту не паникует, не отдаёт 5xx; следующий
      валидный вызов отрабатывает. Края: лимит ровно достигнут; запись «впритык»;
      `0 < max_total_bytes < max_file_bytes` (Q2); ошибка подсчёта объёма не должна давать молчаливый
      обход лимита (Q4). Тест: каждый край → корректный исход (deny/ok), сервер жив, частичных/временных
      файлов не остаётся.
- [ ] AC11 (наследование без изменений): добавление лимита не меняет наследуемые
      auth/Origin/rate-limit и существующие upload-аудит-ветки
      (success/warn/deny(traversal/exists/isdir/too-large/bad-base64/bad-mode/deny_root)/fail); формат
      существующих записей не ломается. Тест: прежние сценарии `upload_file` ведут себя как раньше;
      добавлена только новая deny-причина (общий лимит).
- [ ] AC12 (тесты в Docker, офлайн из vendor): все AC зелёные в Docker; сборка/тесты офлайн из
      `vendor/` (`-mod=vendor`), без `go mod download`; запуск демона только в контейнере (baseline §6);
      новых внешних зависимостей нет. Тест: CI/локальный прогон в контейнере проходит из `vendor/`.

## Out of Scope
- Per-key / per-fingerprint квоты — отдельный риск/задача; здесь только ОБЩИЙ лимит на upload root.
- Content-inspection / антивирус / проверка типа содержимого — отдельный риск (§7).
- FS/контейнерная квота уровня ОС (filesystem quota, cgroup, `--storage-opt`) как замена — смежная
  эксплуатационная мера; задача закрывает риск на уровне ПРИЛОЖЕНИЯ (выбор app- vs FS-level — за
  architect, но FS-quota не артефакт этой задачи).
- Автоочистка/ротация/TTL старых файлов в upload root — отдельная задача.
- `download_file`, чтение/листинг ФС хоста, передача каталогов/архивов — вне scope (как в file-upload).
- Изменение `max_file_bytes`, модели сервиса, выбора системного пользователя — не здесь.

## Open Questions
Блокирующих нет: дефолт `0`=без лимита ради обратной совместимости (AC2) и результат аудита `deny`
(AC5) закрыты обоснованно. Делегированные architect/security (не блокируют контракт):
- [ ] Q1: финальное имя/единица параметра (предложено `upload.max_total_bytes`, байты) — AC1.
- [ ] Q2: трактовка `0 < max_total_bytes < max_file_bytes` (отвергать на старте vs допускать) — AC10.
- [ ] Q3: семантика границы — строгое `<` vs включающее `<=` при проверке превышения лимита — AC3/AC7.
- [ ] Q4: поведение при ошибке подсчёта объёма (fail-closed = отклонять vs иное); молчаливый обход
      недопустим — AC10.
- [ ] Q5: способ учёта (пересчёт vs счётчик) и механизм конкурентной синхронизации — AC6/AC7.
- [ ] Q6: раскрывать ли в сообщении/аудите точные числа текущего/предельного объёма или только факт
      исчерпания квоты — AC4/AC5.
