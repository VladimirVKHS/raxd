# Security Requirements: upload-quota — общий лимит объёма на `upload_file`

Автор продукта: Vladimir Kovalev, OEM TECH

> Каждое требование проверяемо (можно ответить «выполнено/нет» одним вопросом) и ссылается на пункт
> `SECURITY-BASELINE.ru.md`. Эти требования ОБЯЗАНЫ выполнить developer/system-dev/devops/mcp-engineer;
> reviewer и security-guardian проверяют по пунктам. Нумерация SR-90… продолжает file-upload
> (SR-68…SR-82) и command-exec — без коллизий.
>
> Фича расширяет готовый `upload_file`, наследуя SR-68…SR-82 БЕЗ изменений. Требования ниже относятся
> к НОВОЙ логике общего лимита и к инвариантам, которые фича НЕ имеет права ослабить (SR-69/SR-74/SR-80).

## Выполнение / валидация записи (baseline §3)

- [ ] **SR-90. Deny по квоте — ДО фиксации, без следов на диске.** При `max_total_bytes > 0`, если
      `current - replaced + len(data) > max_total_bytes`, загрузка отклоняется `ErrQuotaExceeded` ДО
      создания temp-файла; на диске не остаётся ни temp, ни целевого файла, существующие файлы не
      изменены. ПРОВЕРКА: тест «загрузка за лимит» → нет нового/temp-файла, существующие байт-в-байт
      прежние. (baseline §3; AC3/AC10; закрывает R-Q1/R-Q5)

- [ ] **SR-91. Нейтральное сообщение об отказе — без путей/чисел/секретов.** Текст ответа при deny по
      квоте сообщает только факт исчерпания общего лимита (напр. «upload denied: total upload quota
      exceeded») и НЕ содержит абсолютного пути ФС, раскладки, точных чисел текущего/предельного
      объёма, тела ключа/секрета или содержимого файла. ПРОВЕРКА: grep по телу ответа — нет абсолютного
      пути (нет ведущего `/`/корня хоста), нет тела ключа как подстроки, нет содержимого файла, нет
      числовых значений объёма. (baseline §4; AC4/Q6; не ослабляет SR-80; закрывает R-Q4)

- [ ] **SR-92. Атомарность «проверка+запись» под одним мьютексом на root (TOCTOU закрыт).** Проверка
      объёма И вся атомарная запись выполняются под РОВНО ОДНИМ `sync.Mutex`, привязанным к
      абсолютному `UploadRoot` (package-level реестр `sync.Map` / `LoadOrStore`); `Lock` берётся ДО
      `os.OpenRoot`, `Unlock` через `defer` срабатывает после фиксации ИЛИ любой ошибки. Один и тот же
      мьютекс на root независимо от вызывающего; в `Config` (value-тип) мьютекса нет (`go vet` чист).
      ПРОВЕРКА: тест с N параллельными загрузками, суммарно превышающими остаток квоты, оставляет root
      в пределах `max_total_bytes`, часть отклонена deny, повреждённых/частичных файлов нет; `go vet`
      без замечаний по копированию. (baseline §3/§4; AC7; закрывает R-Q6)

- [ ] **SR-93. Целостность обхода: только regular-файлы внутри корня, симлинки не разыменовываются.**
      `currentBytes` суммирует `Size()` только regular-файлов рекурсивно через `fs.WalkDir(root.FS(),
      ".", …)`; симлинки и не-regular (каталоги/устройства/сокеты) НЕ разыменовываются и НЕ учитываются;
      обход не выходит за корень (`os.Root`-инвариант SR-69 НЕ ослаблен). ПРОВЕРКА: тест — симлинк в
      root, указывающий на крупный файл снаружи, НЕ увеличивает учтённый объём и не приводит к выходу
      за корень. (baseline §3; AC6; не ослабляет SR-69; закрывает R-Q2)

- [ ] **SR-96. Fail-closed при ошибке подсчёта/обхода (молчаливого обхода нет).** ЛЮБАЯ ошибка
      `WalkDir`/`Info()`/`Stat`/открытия root при подсчёте возвращается наверх; `Write` НЕ выполняет
      запись и возвращает ошибку; ошибка трактуется как сбой (а не как «объём=0/неизвестен → пропустить»).
      ПРОВЕРКА: тест с инъекцией ошибки обхода → загрузка НЕ записывается (файла нет), возвращается
      ошибка. (baseline §4; AC10/Q4; закрывает R-Q7)

- [ ] **SR-97. Общий лимит независим и не ослабляет per-file `max_file_bytes`.** Оба лимита действуют
      одновременно; общий лимит НЕ заменяет и НЕ ослабляет per-file; допускается
      `0 < max_total_bytes < max_file_bytes` (тогда крупный файл штатно отклоняется по общему лимиту).
      ПРОВЕРКА: файл в пределах per-file, но выводящий суммарный объём за `max_total_bytes`, → deny по
      общему лимиту; файл сверх `max_file_bytes` → deny как раньше (без регресса). (baseline §3; AC9;
      закрывает R-Q5)

## Учёт и overwrite (baseline §3)

- [ ] **SR-95. Overwrite учитывается дельтой; цель-каталог и существующая цель без overwrite — до
      квота-арифметики.** Порядок после `root.Stat`: цель — каталог → `ErrIsDir` (до арифметики
      квоты); цель существует и `!overwrite` → `ErrExists` (до арифметики); `replaced = fi.Size()`
      ТОЛЬКО когда `overwrite==true` И цель — существующий regular-файл, иначе `replaced = 0`.
      ПРОВЕРКА: при исчерпанной квоте перезапись на тот же/меньший размер проходит; перезапись на
      больший размер за лимит → deny, исходный файл не изменён; перезапись каталога → `ErrIsDir`.
      (baseline §3; AC8; закрывает R-Q1)

## Аудит / устойчивость (baseline §4)

- [ ] **SR-94. Отказ по квоте = ровно одна deny-аудит-запись без секретов.** На каждое отклонение по
      общему лимиту пишется РОВНО одна основная аудит-запись `Result:"deny"`, `Tool:"upload_file"`,
      `Reason` про общий лимит, `timestamp` (UTC), `fingerprint` (НЕ тело ключа), относительный `Path`
      (если известен), `RemoteAddr`. Содержимое файла, абсолютный путь, точные числа объёма и тело
      ключа в записи ОТСУТСТВУЮТ. Существующие upload-аудит-ветки и формат записей не меняются.
      ПРОВЕРКА: на одно отклонение — одна (не ноль, не две) deny-запись с reason про общий лимит;
      grep по аудит-логу — нет содержимого файла, нет абсолютного пути, нет тела ключа. (baseline §4;
      AC5/AC11; не ослабляет SR-78/SR-80; закрывает R-Q3/R-Q4)

- [ ] **SR-96b. Ошибка обхода (fail-closed) аудируется как `fail`, сервер жив.** Ошибка подсчёта
      (НЕ `ErrQuotaExceeded`) маппится в `Result:"fail"` (I/O-семантика), не паникует, не отдаёт 5xx;
      следующий валидный вызов отрабатывает. ПРОВЕРКА: после инъецированной ошибки обхода — fail-запись
      в аудите, сервер отвечает на следующий валидный upload. (baseline §4; AC10; связано с SR-96)

- [ ] **SR-98. Невалидный `max_total_bytes` отвергается на старте; отказ по квоте не роняет сервис.**
      `upload.max_total_bytes < 0` → ошибка старта (как `max_file_bytes`); deny по квоте не паникует и
      не отдаёт 5xx; edge-cases (объём ровно `= max_total_bytes` разрешён по строгому `>`; запись
      «впритык»; `0<max_total<max_file`) дают корректный исход (deny/ok), сервер жив, частичных/temp
      файлов не остаётся. ПРОВЕРКА: старт с отрицательным значением падает с ошибкой; каждый edge-case
      → ожидаемый deny/ok без паники и без следов на диске. (baseline §3/§4; AC1/AC10; закрывает R-Q9)

- [ ] **SR-99. Дефолт `0` = лимит отключён (обратная совместимость, нулевая цена).** Дефолт
      `upload.max_total_bytes=0` означает «лимит ОТКЛЮЧЁН»; при `0` (или отсутствии параметра) обход и
      мьютекс НЕ задействуются и ни одна загрузка не отклоняется по общему лимиту, сколько бы файлов ни
      было. ПРОВЕРКА: при `0`/отсутствии config.yaml поведение `upload_file` идентично прежнему; нет
      вызова обхода/мьютекса. (baseline §4; AC2; закрывает R-Q8/R-Q10)

- [ ] **SR-100. Наследуемые контроли не изменены (без регресса безопасности).** Добавление лимита не
      меняет наследуемые auth/Origin/rate-limit и существующие upload-аудит-ветки
      (success/warn/deny(traversal/exists/isdir/too-large/bad-base64/bad-mode/deny_root)/fail); добавлена
      ТОЛЬКО новая deny-причина «общий лимит». ПРОВЕРКА: прежние сценарии `upload_file` (включая
      traversal/symlink-deny SR-69, no-secrets SR-80) зелёные без изменений; формат существующих записей
      не сломан. (baseline §1/§2/§3/§4; AC11; защищает SR-68…SR-82; закрывает R-Q10)

## Среда (baseline §6)

- [ ] **SR-101. Все проверки квоты — в Docker, офлайн из `vendor/`.** Все AC (включая конкурентный
      AC7 и симлинк-AC6) прогоняются в Docker; сборка/тесты офлайн из `vendor/` (`-mod=vendor`), без
      `go mod download`; запуск демона только в контейнере; новых внешних зависимостей нет (только
      stdlib `sync`/`io/fs`/`errors`). ПРОВЕРКА: CI/локальный прогон в контейнере зелёный из `vendor/`;
      `go.mod` без новых зависимостей. (baseline §6; AC12)

## Остаточные риски (см. threat-model.md)

- [ ] **SR-102. Остаточные риски зафиксированы и (при необходимости) эскалированы.** В `threat-model.md`
      явно зафиксированы: RR-Q1 (нет per-key квоты → DoS «занять всю общую квоту», смягчён rate-limit),
      RR-Q2 (нет FS/ОС-quota → рекомендация defense-in-depth на проде), RR-Q3 (сериализация upload под
      мьютексом — принятый trade-off). ПРОВЕРКА: каждый остаточный риск присутствует со смягчением;
      RR-Q1/RR-Q2 помечены к эскалации. (baseline §4 + Эскалация; AC: Out of Scope spec)
