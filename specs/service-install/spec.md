# Spec: service-install — регистрация raxd как управляемого системного сервиса (macOS + Linux), безопасно от не-root

Автор продукта: Vladimir Kovalev, OEM TECH

## Goal
Превратить `raxd` из foreground-процесса (`raxd serve`) в управляемый системный сервис на обеих
платформах продукта — Linux и macOS — с автозапуском при загрузке, перезапуском при сбое и
корректной остановкой, наследующей уже готовый graceful shutdown по сигналу. Сервис по умолчанию
исполняется под выделенным непривилегированным пользователем (не root), чем закрываются накопленные
остаточные риски `command-exec`/`file-upload` про запись/исполнение от root, а рост аудит-журнала
ограничивается системными средствами ротации. Дополнительно фиксируется требование кросс-сборки
рабочих бинарей под обе платформы и обе архитектуры. Инсталлятор `curl|sh`, goreleaser/CI и
нотаризация — НЕ здесь (задача `distribution`); здесь только регистрация сервиса и то, что нужно для
сборки бинарей под целевые платформы. Это спецификация ЧТО и проверяемых критериев; выбор механизма
регистрации, формата описания сервиса, способа выдачи сетевой capability и раскладки путей для
сервис-пользователя — за architect/system-dev/security.

## User Stories
- Как администратор хоста, я хочу одной командой `raxd` установить/запустить/остановить/удалить сервис
  и узнать его статус, чтобы `raxd` стартовал при загрузке и переживал сбои без ручного вмешательства.
- Как специалист по безопасности, я хочу, чтобы сервис по умолчанию работал под выделенным
  непривилегированным пользователем (не root) и чтобы рост аудит-журнала ограничивался системными
  средствами, чтобы сократить поверхность атаки и не дать журналу заполнить диск.
- Как релиз-инженер, я хочу воспроизводимо собирать рабочие бинари `raxd` под darwin/linux × amd64/arm64
  офлайн из вендоренных зависимостей, чтобы регистрация сервиса опиралась на корректные артефакты под
  обе платформы.

## Acceptance Criteria
- [ ] AC1 (поверхность управления сервисом): управление жизненным циклом сервиса доступно через CLI
      `raxd` и покрывает операции: установка (регистрация сервиса), удаление (дерегистрация), запуск,
      остановка, статус. Набор операций единообразен на обеих платформах (одинаковый пользовательский
      контракт команд; различия платформ скрыты внутри). Тест: на чистом Linux-контейнере с системным
      менеджером сервисов каждая операция доступна и возвращает осмысленный код выхода (0 при успехе,
      ненулевой при ошибке).
- [ ] AC2 (генерация валидного описания сервиса): установка генерирует синтаксически и семантически
      корректное описание сервиса для целевой платформы, которое принимается менеджером сервисов
      соответствующей платформы без ошибок разбора/валидации. (Linux и macOS используют разные форматы
      описания сервиса — конкретный формат на каждой платформе оставлен открытым, см. Q6; AC2 фиксирует
      только наблюдаемое поведение «менеджер принимает описание без ошибок».) Тест Linux: сгенерированное
      описание загружается менеджером сервисов целевой платформы без ошибок валидации (в контейнере).
      Тест macOS: сгенерированное описание сервиса проходит проверку валидности структуры и покрыто
      unit-тестами генератора (см. AC13 об ограничении среды проверки).
- [ ] AC3 (автозапуск при загрузке): после установки сервис помечен на автозапуск при старте системы.
      Тест: в Linux-контейнере с системным менеджером после установки и имитации старта системы сервис
      переходит в активное состояние без ручного запуска (проверяется статусом сервиса).
- [ ] AC4 (перезапуск при сбое): сервис настроен на автоматический перезапуск при аварийном завершении
      (поведение restart-on-failure). Тест: в Linux-контейнере процесс сервиса принудительно завершают
      аварийно (kill); в течение ограниченного интервала сервис повторно поднимается менеджером и
      возвращается в активное состояние (проверяется статусом и сменой PID).
- [ ] AC5 (корректная остановка наследует graceful shutdown): остановка сервиса через CLI/менеджер
      приводит к штатному завершению демона по сигналу — наследуется уже реализованный graceful
      shutdown (`raxd serve` завершается по SIGINT/SIGTERM с дренажом соединений). Остановленный сервис
      НЕ перезапускается автоматически (отличие штатной остановки от сбоя). Тест: остановка в контейнере
      завершает демон корректно (без принудительного kill по таймауту), активные соединения дренируются,
      сервис остаётся в остановленном состоянии и не поднимается заново.
- [ ] AC6 (не-root по умолчанию — явная проверка): по умолчанию сервис исполняет демон под выделенным
      непривилегированным пользователем; эффективный UID работающего процесса демона НЕ равен 0.
      Дефолтный порт `7822` непривилегированный, поэтому root для запуска не требуется. Тест: в
      Linux-контейнере после запуска сервиса эффективный UID процесса демона != 0 (root), и сервис
      обслуживает запросы на дефолтном порту без привилегий root.
- [ ] AC7 (привилегированный порт без полного root): если конфигурация требует привилегированный порт
      (<1024), сервису выдаётся ТОЛЬКО необходимая сетевая привилегия для биндинга такого порта (на
      Linux — точечная сетевая capability), а НЕ полный root и НЕ setuid-root. Конкретный механизм
      выдачи привилегии — Open Question (Q4). Тест: в Linux-контейнере сервис, настроенный на порт <1024,
      успешно слушает этот порт, при этом эффективный UID процесса демона != 0 (root не используется).
- [ ] AC8 (ротация/ограничение роста журнала — системными средствами): задокументирован и проверяемо
      работает механизм, ограничивающий неограниченный рост журнала сервиса (журналирование сервиса
      и/или ротация файлового вывода), закрывающий остаточный риск роста аудит-лога из
      `command-exec`/`file-upload`. Конкретный механизм/политика (системный журнал vs ротация файлов,
      пороги размера/срока) — Open Question (Q5). Тест: в Linux-контейнере порог объёма журнала намеренно
      занижается; журнал сервиса наполняется синтетическими записями до превышения порога; проверяется,
      что рост журнала ограничен (старые записи вытесняются/архивируются по настроенной политике, размер
      не растёт неограниченно), и механизм описан в сопровождающей документации. Конкретные числа порога
      и объём генерируемых записей — на усмотрение architect/QA; обязателен сам метод воспроизведения
      (заниженный порог + наполнение до превышения + проверка ограниченности роста).
- [ ] AC9 (идемпотентность установки): повторная установка при уже установленном сервисе НЕ создаёт
      дубликат и НЕ оставляет систему в противоречивом состоянии — операция либо безопасно
      завершается успехом (приводит к ожидаемому конечному состоянию), либо возвращает понятную ошибку
      «уже установлен», не повреждая существующую регистрацию. Тест: двукратная установка подряд в
      контейнере оставляет ровно одну корректную регистрацию сервиса; вторая попытка не ломает первую.
- [ ] AC10 (идемпотентность удаления): удаление при отсутствующем сервисе НЕ падает с невнятной
      ошибкой и не оставляет «осиротевших» артефактов — операция безопасно завершается (успех или
      понятное «не установлен»). После удаления установленного сервиса не остаётся артефактов
      регистрации (описание сервиса дерегистрировано/удалено). Тест: удаление неустановленного сервиса
      завершается предсказуемо; удаление установленного — снимает регистрацию полностью (повторный
      статус показывает «не установлен»).
- [ ] AC11 (безопасный откат при сбое установки — без «полуустановленного» сервиса): если установка
      прерывается на полпути (ошибка на любом шаге регистрации), команда НЕ оставляет частично
      зарегистрированный/неработоспособный сервис: либо доводит до рабочего состояния, либо откатывает
      изменения и сообщает об ошибке, оставляя систему в исходном (до установки) состоянии. Тест:
      смоделированный сбой на шаге установки (в контейнере) приводит к понятной ошибке и к отсутствию
      остаточных артефактов регистрации; повторная корректная установка после этого проходит штатно.
- [ ] AC12 (понятные сообщения и ошибки): операции управления сервисом печатают осмысленные сообщения
      об успехе и НЕЙТРАЛЬНЫЕ, действенные сообщения об ошибках (что произошло + подсказка), без утечки
      секретов и без сырых трасс. Типовые ошибки покрыты: недостаточно прав для регистрации сервиса,
      сервис уже установлен (AC9), сервис не установлен при остановке/удалении (AC10), менеджер сервисов
      недоступен/не поддержан. Тест: каждый перечисленный сценарий даёт ненулевой код выхода и
      сообщение в формате «ошибка + подсказка» (по стилю существующего CLI), без секретов в выводе.
- [ ] AC13 (ограничение проверки macOS зафиксировано — НЕ пропуск требования): требования AC1-AC12
      применимы к ОБЕИМ платформам как контракт; интеграционная проверка под macOS в Docker невозможна
      (Docker — Linux). Поэтому для macOS обязательны: (а) генерация валидного описания сервиса (AC2) и
      (б) unit-тесты генератора описания сервиса и логики выбора платформы. Это ОГРАНИЧЕНИЕ СРЕДЫ
      ПРОВЕРКИ, а не снятие требования; явно фиксируется в документации/плане. Тест: unit-тесты
      генератора описания сервиса зелёные; в документации/плане зафиксировано, что macOS-интеграция
      проверяется на реальном macOS вне Docker (ручной/CI-прогон), а в контейнере проверяется только
      генерация и unit-логика.
- [ ] AC14 (кросс-сборка под все 4 цели): сборка `raxd` даёт РАБОЧИЕ бинари под darwin/linux × amd64/arm64
      (4 цели). «Рабочий» минимально означает: бинарь успешно собирается под цель и под нативной для CI
      целью (linux/<arch>) запускается и отвечает на базовую неразрушающую проверку (например, вывод
      версии). Тест: матрица сборки даёт 4 артефакта без ошибок компиляции; нативный для контейнера
      бинарь исполняется и проходит базовую проверку (cross-target бинари — проверка факта успешной
      компиляции под цель).
- [ ] AC15 (без новых рантайм-зависимостей сверх baseline; офлайн-сборка из vendor): service-install НЕ
      вводит новых внешних рантайм-зависимостей сверх baseline-стека; зависимости вендорятся, сборка и
      тесты идут офлайн из `vendor/` (`-mod=vendor`, без `go mod download`). Тест: сборка/тесты в
      контейнере проходят офлайн из `vendor/`; список модулей не приобретает не-вендоренных внешних
      рантайм-зависимостей сверх уже зафиксированных в стеке.
- [ ] AC16 (проверяемость в Docker — baseline §6): все Linux-интеграционные критерии выше зелёные в
      Docker-контейнере с системным менеджером сервисов; сборка/тесты — офлайн из `vendor/`; запуск
      демона и сервисная интеграция — только в контейнере (baseline §6), не на хосте разработчика.
      Тест: контейнерный прогон сервисной интеграции (Linux) и unit-тестов (включая генератор описания
      сервиса) проходит целиком из `vendor/`.

## Scope
- CLI-операции жизненного цикла сервиса: установка/удаление/запуск/остановка/статус (AC1).
- Генерация валидного описания сервиса под Linux и macOS (AC2, AC13).
- Lifecycle: автозапуск при загрузке, перезапуск при сбое, корректная остановка с graceful shutdown
  (AC3-AC5).
- Безопасная раскладка по умолчанию: не-root исполнение; точечная сетевая привилегия для порта <1024
  вместо полного root (AC6, AC7).
- Системная ротация/ограничение роста журнала сервиса (AC8).
- Идемпотентность и безопасный откат install/uninstall; понятные ошибки (AC9-AC12).
- Кросс-сборка рабочих бинарей под darwin/linux × amd64/arm64 в объёме, нужном для регистрации сервиса
  под платформы (AC14).
- Вендоринг/офлайн-сборка и проверяемость в Docker (AC15, AC16).

## Out of Scope
- Инсталлятор `curl | sh`, конфигурация goreleaser и CI-пайплайн релизов — задача `distribution`
  (здесь только кросс-сборка как требование наличия рабочих бинарей под платформы, без дистрибуции).
- Подпись и нотаризация macOS-бинаря, снятие quarantine — задача `distribution`.
- Windows и любые не-{macOS, Linux} платформы — вне scope продукта.
- Контейнерный/Kubernetes-деплой как ПЕРВИЧНЫЙ режим запуска — не цель; Docker здесь только среда
  сборки/тестирования (baseline §6).
- Изменение функциональности самого демона (`serve`, MCP-инструменты, auth, TLS, rate-limit, формат
  аудит-записи) — берётся как есть; service-install лишь оборачивает запуск демона сервисом.
- Автоматическое обновление/самообновление установленного сервиса — отдельная задача.
- Многоэкземплярный запуск (несколько именованных сервисов на одном хосте) — не в v1.

## Open Questions
- [ ] Q1: имя выделенного сервис-пользователя (и группы) и способ его создания при установке (создавать
      ли пользователя командой установки, требовать ли заранее, политика при уже существующем
      пользователе) — для research/architect/security/system-dev (AC6).
- [ ] Q2: путь установки исполняемого бинаря сервиса (системный путь vs текущее расположение) и кто за
      него отвечает на стыке с задачей `distribution` (AC1, AC2).
- [ ] Q3: раскладка путей конфига/состояния/upload-root/аудита для сервис-пользователя — системные
      каталоги сервиса vs домашние `~/.config/raxd` и `~/.local/state/raxd`; как сервис-пользователю
      выдаётся доступ к этим путям с корректными правами (наследуется требование прав `0600`/`0700` из
      baseline и STACK) (AC6, AC8).
- [ ] Q4: конкретный механизм выдачи минимальной сетевой привилегии для порта <1024 (точечная
      capability vs альтернативы), как он применяется при установке и как проверяется в контейнере
      (AC7).
- [ ] Q5: механизм и политика ротации/ограничения роста журнала сервиса (системный журнал vs ротация
      файлового вывода; пороги размера/срока хранения), согласованные со структурным аудитом из
      `command-exec`/`file-upload` (AC8).
- [ ] Q6: формат и расположение генерируемого описания сервиса на каждой платформе (Linux и macOS
      используют разные форматы; где размещается описание, имя сервиса/лейбл, права на файл описания) —
      НЕ решать на уровне pm (AC2).
- [ ] Q7: семантика идемпотентной повторной установки — «привести к ожидаемому состоянию (upgrade in
      place)» vs «вернуть ошибку: уже установлен»; и критерий «полного» удаления артефактов (AC9, AC10,
      AC11).
- [ ] Q8: критерий приёмки «рабочего» cross-target бинаря для не-нативных под CI целей (только успешная
      компиляция vs дополнительная проверка через эмуляцию) (AC14).

## Примечание о размере задачи
Задача среднего размера (16 AC) и НЕДЕЛИМА по смыслу: «безопасно управляемый системный сервис» —
единый контракт, где регистрация/жизненный цикл, не-root раскладка, ограничение роста журнала и
наличие рабочих бинарей под обе платформы взаимозависимы (любой кусок в отрыве даёт небезопасную или
неполную установку). Часть AC обусловлена плотностью требований безопасности и кроссплатформенностью,
а не раздуванием scope. Если architect сочтёт кросс-сборку (AC14/AC15) отделимой от регистрации
сервиса — это допустимая развилка для обсуждения на следующем шаге, но в рамках текущего контракта
наличие рабочих бинарей под платформы — предусловие осмысленной регистрации.
