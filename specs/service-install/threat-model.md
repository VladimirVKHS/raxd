# Threat Model: service-install — регистрация raxd как управляемого системного сервиса (Linux systemd + macOS launchd), не-root

Автор: security (raxd). Дата: 2026-05-22. Язык: русский.
Автор продукта: Vladimir Kovalev, OEM TECH.

Вход: `specs/service-install/{spec.md (AC1–AC16, Q1–Q8), plan.md (architect-guardian pass), research.md (research-guardian pass)}`,
ADR-001 (ручная генерация unit/plist на stdlib `text/template` + нативные `systemctl`/`launchctl`),
ADR-002 (статический пользователь `raxd:raxd` + XDG-через-`Environment=` + `StateDirectoryMode=0700`),
ADR-003 (условный `AmbientCapabilities=CAP_NET_BIND_SERVICE` только при Port<1024; опуск `NoNewPrivileges` при ambient),

`.claude/reference/SECURITY-BASELINE.ru.md` (§1–§6, ОБЯЗАТЕЛЬНЫЙ контракт), `.claude/reference/MCP-INTEGRATION.ru.md`;
прочитанный напрямую код `internal/config/paths.go` (`Paths()` через XDG-env, `EnsureDirs` 0700);
УЖЕ ВЫПОЛНЕННЫЕ `specs/command-exec/{threat-model,security-requirements}.md` (ОР-1 root-исполнение, ОР-2 ротация)
и `specs/file-upload/{threat-model,security-requirements}.md` (ОР-U1 root-запись, ОР-U4 ротация) — накопленные
остаточные риски, которые ЭТА задача обязана закрыть не-root раскладкой + системной ротацией.

> **Это задача про привилегии и установку системного сервиса для инструмента, который по сети исполняет
> произвольные команды и пишет файлы (`raxd serve` = exec + upload поверх TLS/MCP).** Сама регистрация
> сервиса не добавляет новых сетевых поверхностей демона (serve/MCP/auth/TLS/rate-limit берутся как есть,
> spec Out of Scope), НО создаёт новый класс угроз УРОВНЯ ХОСТА: под каким UID и с какими привилегиями
> бежит демон, кто может подменить команду сервиса (unit/plist), что остаётся после удаления, и не утечёт
> ли секрет в журнал установки. Главный инвариант: **демон по умолчанию исполняется НЕ от root**
> (euid!=0), привилегии строго минимальны и условны, артефакты регистрации не подменяемы непривилегированным
> пользователем, а рост аудит-журнала ограничен системой. Каждый риск ниже имеет смягчение; невыполнимое/
> частичное вынесено в «Остаточные риски / эскалации»; отступления от буквы baseline — в «Принятые отклонения».

## Контекст наследования

`service-install` НЕ меняет функциональность демона и НЕ переоткрывает наследуемый сетевой периметр:

- **Транспорт `tls-transport`** (SR-1…SR-26): TLS 1.3, Bearer-auth, Host/Origin, rate-limit, bind
  `127.0.0.1`, аудит без секретов, graceful shutdown. **Не переопределяется** — сервис лишь оборачивает
  запуск `raxd serve`, который этот периметр уже несёт.
- **MCP `mcp-server`** (SR-27…SR-39) и инструменты **`command-exec`** (SR-40…SR-67) / **`file-upload`**
  (SR-68…SR-82): набор tools, схемы, лимиты, root-WARN/`deny_root` на уровне инструментов — как есть.
- **Graceful shutdown** (`internal/cli/serve.go`: `signal.NotifyContext(..., SIGTERM)` → return nil → код 0)
  наследуется как поведение остановки сервиса (AC5) — service-install лишь подаёт SIGTERM через менеджер.

Эта модель фиксирует ТОЛЬКО новое, что вносит service-install: **привилегии и UID процесса демона,
генерируемые артефакты регистрации (unit/plist/drop-in/каталоги/пользователь), вызов системного менеджера,
идемпотентность/откат install·uninstall, наследование окружения сервиса, и журнал установки.**

## Активы

- **Привилегии и UID процесса демона (особенно euid==0)** — ГЛАВНЫЙ актив этой задачи. raxd исполняет
  произвольные команды и пишет файлы; если демон бежит от root, КАЖДАЯ команда и КАЖДАЯ загрузка идут от
  root (это и есть остаточные риски command-exec ОР-1 / file-upload ОР-U1). Цель service-install — euid!=0
  по умолчанию и строго минимальная условная сетевая привилегия (CAP_NET_BIND_SERVICE только при Port<1024),
  никогда полный root и никогда setuid-root.
- **Исполняемый бинарь `raxd` (цель `ExecStart`/`ProgramArguments`)** — что именно запускает сервис.
  Подмена пути или самого бинаря = подмена кода, исполняемого с привилегиями сервиса при каждом старте/рестарте.
- **Артефакты регистрации: unit `/etc/systemd/system/raxd.service`, plist
  `/Library/LaunchDaemons/tech.oem.raxd.plist`, journald drop-in `/etc/systemd/journald.conf.d/raxd.conf`**
  — текстовые файлы, определяющие КОМАНДУ, ПОЛЬЗОВАТЕЛЯ, ПРИВИЛЕГИИ и ОКРУЖЕНИЕ сервиса. Если они доступны
  на запись непривилегированному пользователю — он переопределит `ExecStart`/`User`/`AmbientCapabilities`
  и получит исполнение от нужного ему UID при следующем старте. Целостность = root:root, не world-writable.
- **Каталог состояния и его содержимое (`StateDir` → `/var/lib/raxd`: `keys.db`, `tls/`, `uploads/`, аудит)**
  — персистентные секреты и данные демона. Должны принадлежать пользователю `raxd` с правами `0700` на
  каталог (дефолт `StateDirectoryMode` systemd = 0755 ШИРЕ baseline!) и `0600` на `keys.db`/приватный TLS-ключ.
- **Сервис-пользователь `raxd:raxd`** — выделенный непривилегированный аккаунт. Без shell (`nologin`), без
  обычного home, минимум прав. Если у него есть login-shell или широкие права на ФС — поверхность атаки
  растёт; если при uninstall он остаётся с доступом к чужим данным — остаточная привилегия.
- **Аудит-журнал сервиса (stderr → journald / `StandardErrorPath`)** — журнал каждого действия демона.
  Актив расследования инцидентов; его неограниченный рост заполняет диск (остаточный риск ОР-2/ОР-U4),
  поэтому требуется системное ограничение роста.
- **Окружение сервиса (`Environment=`/`EnvironmentVariables`: `XDG_*`, `HOME`, `PATH`)** — определяет, где
  демон ищет конфиг/состояние и (косвенно) бинари при exec. Подмена/непредсказуемость окружения = чтение
  чужого конфига, запись состояния не туда, неконтролируемый `PATH` при выполнении команд.
- **Состояние системы во время install/uninstall** — целостность перехода: полу-установленный сервис
  (часть артефактов создана, часть нет) = неработающий или непредсказуемо привилегированный сервис; «осиротевшие»
  артефакты после uninstall = остаточный автозапуск/привилегии.
- **Журнал установки (вывод CLI `raxd service install/uninstall/...`)** — сообщения об успехе/ошибке. Пути и
  имена в нём допустимы; содержимое секретов (тело ключа, приватный TLS-ключ) и сырые трассы — НЕТ (AC12).
- **Среда сборки/проверки (Docker, vendor/)** — сборка/тесты/запуск сервиса ТОЛЬКО в контейнере (baseline §6);
  офлайн из `vendor/` без новых внешних рантайм-зависимостей (AC15).

## Поверхность атаки

- **CLI-операции жизненного цикла `raxd service {install,uninstall,start,stop,status}`** (AC1) — точка, где
  привилегированные действия (запись unit/plist в системные каталоги, создание пользователя, вызов
  `systemctl`/`launchctl`) инициируются. Требует прав администратора для регистрации (ожидаемо); недостаток
  прав → `ErrPermission` с нейтральным сообщением (AC12).
- **Генератор описания сервиса `renderUnit/renderPlist` (`internal/service/templates.go`)** — превращает
  `TemplateData{ExecPath, User, Group, Label, Port, StateDir, ConfigDir, LogPath, NeedNetBindCap}` в текст
  unit/plist через `text/template`. Значения частично происходят из конфига/окружения (`ExecPath` =
  `os.Executable()`, `Port` = `config.Load(...).Port`, имена пользователя/лейбла); если они попадут в шаблон
  без валидации, спецсимволы/переводы строк могут ИНЪЕКТИРОВАТЬ директивы (новый `ExecStart=`, новый
  `User=`, доп. `AmbientCapabilities=`) → исполнение от другого UID/с лишней привилегией.
- **Вызов системного менеджера `runManager(ctx, name, args...)` (`internal/service/exec.go`) над `os/exec`**
  — запуск `systemctl`/`launchctl`. Если бинарь менеджера ищется по непредсказуемому `PATH` или вызывается
  через shell-строку с подставленными значениями — подмена менеджера / shell-инъекция.
- **Файлы регистрации (unit/plist/drop-in) на диске** — после записи это команда сервиса; права на них
  определяют, кто может их подменить.
- **Создаваемый сервис-пользователь и каталоги** (`useradd --system`, `StateDirectory`/`mkdir`+`chown`) —
  права/режим каталогов и атрибуты пользователя.
- **Кто атакует:**
  - **Непривилегированный локальный пользователь хоста** — ОСНОВНОЙ актор этой задачи. Пытается: подменить
    unit/plist/drop-in или сам бинарь `raxd`, если те world-writable; читать `keys.db`/TLS-ключ, если каталог
    состояния шире 0700; воспользоваться тем, что демон бежит от root (через любую уязвимость демона
    повысить привилегии). Цель: исполнение от root / чтение секретов / эскалация.
  - **Аутентифицированный MCP-клиент (легитимный/скомпрометированный агент)** — наследуемый актор серверного
    периметра; его команды/загрузки исполняются под UID демона. Если демон от root — его действия от root
    (наследуемая угроза, ЗАКРЫВАЕТСЯ не-root раскладкой). Сам по себе он не управляет сервисом (CLI lifecycle
    локален), но его влияние усиливается уровнем привилегий процесса.
  - **Оператор/администратор, ставящий сервис** — может задать конфиг со спецсимволами в путях/имени (вектор
    инъекции в шаблон), привилегированный порт (вектор лишней capability), запустить install в окружении с
    «грязными» `PATH`/`HOME`. Не злонамеренный, но входные значения недоверенны для генератора.
  - **Инфраструктурный фактор** — прерывание install на полпути (сбой/нехватка прав/недоступность менеджера)
    → полу-установленное состояние; контейнер/dev-окружение, стартующее от root (вектор euid==0).

## Угрозы (STRIDE-ориентир)

- **S (Spoofing) — подмена команды/бинаря сервиса.** Если unit/plist (или бинарь по `ExecPath`) доступны на
  запись непривилегированному пользователю, он подменит `ExecStart`/`ProgramArguments` или сам бинарь → при
  следующем старте/рестарте система выполнит его код с привилегиями сервиса. Смягчение: unit/plist/drop-in —
  владелец root:root, режим 0644 (не world/group-writable), в системных каталогах, куда пишет только root;
  `ExecPath` — абсолютный путь к доверенному бинарю — R-S1, R-S2.
- **T (Tampering) — инъекция директив в генерируемый unit/plist.** Значения `ExecPath`/`User`/`Group`/`Port`/
  путей/лейбла, попав в шаблон без валидации, могут нести `\n`/спецсимволы и инъектировать поддельную
  директиву (новый `ExecStart=`, лишний `AmbientCapabilities=`, переопределённый `User=`). Смягчение:
  валидация/санитизация всех подстановок ДО рендера (allowlist символов, запрет `\n`/`\r`/управляющих,
  числовой Port в диапазоне, абсолютный нормализованный `ExecPath`); деривация условных директив из
  типизированного `NeedNetBindCap`, а не из сырой строки — R-T1.
- **R (Repudiation) / прозрачность установки.** Операции lifecycle должны давать осмысленный код выхода и
  сообщение (AC12); это не аудит уровня демона (тот наследуется), но действия установки должны быть
  наблюдаемы и не «молча» оставлять привилегии. Смягчение: каждая операция → 0/ненулевой код + понятное
  сообщение; uninstall детерминированно снимает автозапуск/привилегии/артефакты — R-R1.
- **I (Information disclosure) — утечка секрета в журнал установки / широкие права на состояние.** (а)
  Сообщения install/uninstall не должны печатать тело ключа/приватный TLS-ключ/сырые трассы (AC12); (б)
  каталог состояния шире 0700 или `keys.db`/TLS-ключ шире 0600 → чтение секретов локальным пользователем.
  Смягчение: нейтральные сообщения без секретов (пути/имена ок); `StateDirectoryMode=0700` ЯВНО (дефолт
  systemd 0755 шире baseline); `keys.db`/TLS-ключ 0600 (наследуется `EnsureDirs`/keystore) — R-I1, R-I2.
- **D (Denial of service) — неограниченный рост журнала / полу-установленный сервис.** (а) Без системной
  ротации аудит-журнал демона заполнит диск (это остаточный риск ОР-2/ОР-U4); (б) прерванный install
  оставляет неработоспособный/частично привилегированный сервис. Смягчение: системное ограничение роста
  журнала (journald drop-in `SystemMaxUse=`/`SystemMaxFileSize=`, AC8); безопасный откат install (AC11) —
  R-D1, R-D2.
- **E (Elevation of privilege) — запуск от root / лишняя привилегия / setuid / неполный uninstall.** (а)
  Демон от root → exec/upload от root (наследуемые ОР-1/ОР-U1); (б) широкая capability (полный набор / при
  порте ≥1024) или setuid-root вместо точечной CAP_NET_BIND_SERVICE; (в) опущенный `NoNewPrivileges` при
  ambient (ADR-003) — окно ослабления hardening; (г) uninstall, оставляющий включённый автозапуск/capability/
  доступ. Смягчение: euid!=0 по умолчанию (порт 7822 непривилегированный); capability условна (Port<1024) и
  минимальна (только CAP_NET_BIND_SERVICE), не setuid root; `NoNewPrivileges=yes` при Port≥1024, опуск —
  только при ambient с компенсацией; uninstall снимает unit/plist/drop-in/автозапуск/capability — R-E1, R-E2,
  R-E3, R-E4.

## Риски (риск → вероятность/влияние → смягчение)

- [ ] **R-E1. Демон исполняется от root (euid==0) — exec и upload идут от root (ЗАКРЫВАЕТ command-exec
  ОР-1 / file-upload ОР-U1) — ГЛАВНЫЙ риск этой задачи.** raxd по назначению исполняет произвольные команды
  и пишет файлы; если процесс демона бежит с euid==0, каждое действие агента идёт от root → компрометация
  демона = root-доступ к хосту. Вероятность **средняя** (без выделенной раскладки демон легко стартует от
  root в контейнере/dev/при наивной установке) / влияние **критическое**. Смягчение: сервис по умолчанию
  исполняется под выделенным непривилегированным пользователем `raxd:raxd` — Linux `User=raxd`/`Group=raxd`
  в unit, macOS `UserName=raxd` в plist; дефолтный порт **7822** непривилегированный → root для запуска НЕ
  нужен; эффективный UID работающего процесса демона **!= 0** (AC6). Это и есть «основная защита», на которую
  ссылались command-exec ОР-1 и file-upload ОР-U1 → данный риск ЗАКРЫВАЕТ те остаточные риски для штатной
  не-root установки. Проверяемо: в Linux-контейнере после `service install`+start euid процесса демона != 0
  (`systemctl show -p MainPID` → `/proc/<pid>/status` Uid), сервис обслуживает дефолтный порт без root.
  (baseline §3 «демон НЕ от root»; spec AC6; ADR-002; closes command-exec ОР-1 / file-upload ОР-U1)

- [ ] **R-E2. Привилегия для порта <1024 шире необходимого (полный root / setuid-root / лишние capability).**
  Если оператор выбирает привилегированный порт (<1024), наивная реализация дала бы полный root или
  setuid-root, либо широкий набор capability. Вероятность **низкая** (дефолт 7822 не требует привилегии;
  релевантно лишь при ручной смене порта) / влияние **высокое** (полный root сводит на нет всю не-root
  раскладку). Смягчение: при Port<1024 генератор УСЛОВНО добавляет ТОЛЬКО `AmbientCapabilities=CAP_NET_BIND_SERVICE`
  (ADR-003) — точечная сетевая capability, выдаваемая менеджером systemd непривилегированному `User=raxd`
  через auto `keep-caps`; НИКОГДА не setuid-root и не полный root; для Port≥1024 (включая дефолт) директива
  НЕ генерируется (ноль привилегий). Проверяемо: тест — unit для Port<1024 содержит ровно
  `AmbientCapabilities=CAP_NET_BIND_SERVICE` и не содержит иных capability/setuid; unit для Port≥1024 НЕ
  содержит `AmbientCapabilities`; в контейнере сервис на порту <1024 слушает порт при euid!=0. (baseline §3
  «capabilities, а не setuid root»; spec AC7; ADR-003; research Q4)

- [ ] **R-E3. Опуск `NoNewPrivileges` при ambient-capability ослабляет hardening (ADR-003).** При Port<1024
  ADR-003 опускает `NoNewPrivileges=yes` (чтобы исключить недоказанный конфликт ambient×NoNewPrivileges) —
  процесс теряет защиту «никаких новых привилегий после execve» в этом окне. Вероятность **низкая** (только
  при редком ручном выборе привилегированного порта) / влияние **среднее** (расширяется поверхность
  пост-exec эскалации внутри уже непривилегированного процесса). Смягчение: (1) `NoNewPrivileges=yes`
  СТАВИТСЯ всегда при Port≥1024 (дефолт и подавляющее большинство установок — полный hardening); опуск —
  ТОЛЬКО при наличии `AmbientCapabilities` (узкое окно); (2) сохраняются прочие hardening-директивы
  (`ProtectSystem=strict`, `ProtectHome=yes`, `PrivateTmp=yes`) независимо от порта; (3) security ОБЯЗАН
  подтвердить совместимость/необходимость опуска при утверждении unit (research Открытый вопрос). Это
  **принятое отклонение** (П-1) с остаточным риском (ОР-1). Проверяемо: тест — unit при Port≥1024 содержит
  `NoNewPrivileges=yes`; unit при Port<1024 содержит `AmbientCapabilities` и НЕ содержит `NoNewPrivileges`,
  при этом `ProtectSystem`/`ProtectHome`/`PrivateTmp` присутствуют в обоих случаях. (baseline §3; spec AC7;
  ADR-003; research Открытые вопросы «ambient×NoNewPrivileges»)

- [ ] **R-E4. Неполный uninstall оставляет автозапуск / привилегии / артефакты («осиротевшие» элементы).**
  Если uninstall не снимает enable-on-boot, оставляет unit/plist/drop-in или capability, сервис продолжит
  стартовать при загрузке с прежними привилегиями, а подменяемый артефакт останется на диске. Вероятность
  **средняя** (частичное удаление — частый дефект) / влияние **высокое** (остаточный привилегированный
  автозапуск). Смягчение (AC10): `Uninstall` детерминированно выполняет stop+disable (снятие автозапуска) +
  удаление unit/plist + drop-in (а с ним — capability/hardening-директивы, т.к. они в unit) + удаление
  созданных при install каталогов регистрации; после успеха артефактов регистрации НЕТ (повторный `Status` →
  `Installed:false`). ОСОЗНАННО ОСТАЁТСЯ: системный пользователь `raxd:raxd` НЕ удаляется (ADR-002 — он без
  shell/home, не несёт автозапуска/привилегии сам по себе; удаление чужого/переиспользованного пользователя
  опаснее, чем оставить непривилегированный аккаунт) — это документируется (П-2). Проверяемо: тест — после
  uninstall в контейнере: `systemctl is-enabled raxd` → не enabled, unit/drop-in отсутствуют, статус
  «не установлен»; пользователь `raxd` сохранён БЕЗ login-shell и без автозапуска. (baseline §3; spec AC10;
  ADR-002; closes-context command-exec ОР-1 «не-root раскладка»)

- [ ] **R-T1. Инъекция директив в unit/plist через значения из конфига/окружения.** `ExecPath`, `User`,
  `Group`, `Label`, пути (`StateDir`/`ConfigDir`/`LogPath`), `Port`, попав в `text/template` без валидации,
  могут содержать перевод строки/спецсимволы и инъектировать поддельную директиву (`\nExecStart=/bin/sh ...`,
  `\nUser=root`, `\nAmbientCapabilities=CAP_SYS_ADMIN`). Так непривилегированный конфиг/окружение получает
  исполнение от другого UID / лишнюю привилегию. Вероятность **низкая** (значения обычно из доверенных
  `os.Executable()`/конфига, но конфиг недоверен) / влияние **критическое** (произвольная команда/UID/cap в
  сервисе). Смягчение: КАЖДОЕ подставляемое значение ВАЛИДИРУЕТСЯ ДО рендера: `User`/`Group`/`Label` — строгий
  allowlist (POSIX-имя пользователя/группы, reverse-DNS-лейбл; запрет пробелов/`\n`/`\r`/управляющих/`=`/
  кавычек); `ExecPath` — абсолютный, нормализованный (`filepath.IsAbs`+`Clean`), без переводов строк/
  управляющих; пути — абсолютные нормализованные без управляющих; `Port` — целое в `1..65535`; условные
  директивы (`AmbientCapabilities`/`NoNewPrivileges`) деривируются из ТИПИЗИРОВАННОГО `NeedNetBindCap (= Port<1024)`,
  а НЕ из сырой строки; невалидное значение → ошибка установки ДО записи артефакта, не «тихий» рендер.
  Проверяемо: тест — `TemplateData` с `\n`/спецсимволом/`=` в `User`/`Label`/`ExecPath`/пути → генератор
  возвращает ошибку, поддельная директива в выводе НЕ появляется; сгенерированный для валидных входов unit
  загружается менеджером без ошибок (AC2), plist — валиден по unit-тесту (AC13). (baseline §3 «валидация
  входа, без shell-инъекций»; spec AC2/AC12; plan §Хэндофф security «без инъекций»; research Q6)

- [ ] **R-S1. unit/plist/drop-in доступны на запись непривилегированному пользователю → подмена команды
  сервиса.** Если файл описания сервиса записываемый группой/миром, локальный пользователь перепишет
  `ExecStart`/`User`/`AmbientCapabilities` и получит исполнение своего кода от нужного UID при следующем
  старте/рестарте. Вероятность **низкая** (системные каталоги обычно root-only) / влияние **критическое**
  (исполнение от привилегированного UID). Смягчение: unit `/etc/systemd/system/raxd.service`, plist
  `/Library/LaunchDaemons/tech.oem.raxd.plist`, drop-in `/etc/systemd/journald.conf.d/raxd.conf` создаются с
  владельцем **root:root** и режимом **0644** (читаемы, НЕ записываемы группой/миром), в системных каталогах,
  куда пишет только root; запись выполняется операцией install с правами администратора. Проверяемо: тест —
  после install в контейнере владелец unit/drop-in = root:root, режим 0644 (нет `0002`/`0020` write-битов),
  непривилегированный пользователь НЕ может перезаписать файл. (baseline §3; spec AC2; plan §Хэндофф security
  «unit/plist/drop-in root:root 0644»; research Q6)

- [ ] **R-S2. Каталог состояния шире 0700 / `keys.db` / TLS-ключ шире 0600 → чтение секретов и подмена
  состояния.** Дефолт `StateDirectoryMode` systemd = **0755** (ШИРЕ baseline 0700) — без явного задания
  любой локальный пользователь прочитает `/var/lib/raxd` и подменит содержимое; `keys.db`/приватный TLS-ключ
  шире 0600 = утечка секретов аутентификации/транспорта. Вероятность **средняя** (дефолт systemd небезопасен
  по умолчанию — легко упустить) / влияние **высокое** (утечка ключей = полный доступ к инструменту).
  Смягчение: `StateDirectory=raxd` + ЯВНО `StateDirectoryMode=0700` в unit (ADR-002); на macOS каталог
  состояния создаётся install явно `mkdir 0700`+`chown raxd`; `keys.db` и приватный TLS-ключ остаются `0600`
  (наследуется существующим `EnsureDirs` 0700 / keystore — `internal/config/paths.go` прочитан); владелец
  состояния = `raxd:raxd`. Проверяемо: тест — после install каталог `/var/lib/raxd` имеет режим 0700 и
  владельца `raxd`; (наследуемо) `keys.db`/TLS-ключ — 0600; grep unit — присутствует `StateDirectoryMode=0700`.
  (baseline §3 «права 0600/0700»; spec AC6; ADR-002; research Q3 «дефолт 0755 шире baseline»)

- [ ] **R-I1. Утечка секрета в журнал/вывод операций установки.** Сообщения `service install/uninstall/...`
  (особенно ошибки) не должны печатать тело API-ключа, приватный TLS-ключ или сырые трассы менеджера,
  раскрывающие внутренности. Вероятность **низкая** (операции установки не оперируют телом ключа) / влияние
  **высокое** (секрет в выводе/логе = компрометация). Смягчение: вывод операций — нейтральные, действенные
  сообщения (что произошло + подсказка), пути/имена допустимы, секреты и сырой stderr менеджера НЕ
  пробрасываются (`runManager` захватывает stderr → нейтральная типизированная ошибка, plan §Contracts);
  install не читает/не печатает `keys.db`/TLS-ключ. Проверяемо: тест — каждый типовой сценарий ошибки
  (нет прав / уже установлен / не установлен / менеджер недоступен) даёт ненулевой код + сообщение
  «ошибка + подсказка», без секретов и без сырой трассы (grep вывода — нет `rax_live_`/PEM-маркеров/`panic:`).
  (baseline §4 «никаких секретов в логах/выводе CLI»; spec AC12; plan §Contracts `runManager`)

- [ ] **R-D1. Неограниченный рост аудит-журнала сервиса заполняет диск (ЗАКРЫВАЕТ command-exec ОР-2 /
  file-upload ОР-U4).** Демон пишет аудит каждого действия в stderr; без системного ограничения журнал
  растёт неограниченно → исчерпание диска → отказ демона и хоста. command-exec ОР-2 и file-upload ОР-U4
  явно делегировали ротацию системе/дистрибуции — service-install обязан это закрыть. Вероятность **средняя**
  (долгоживущий сервис под нагрузкой) / влияние **высокое** (исчерпание диска = отказ). Смягчение (ADR-004,
  AC8): аудит остаётся в stderr → journald (`Type=exec` + journald дефолт), рост ограничивается drop-in
  `/etc/systemd/journald.conf.d/raxd.conf` с `SystemMaxUse=`/`SystemMaxFileSize=`; journald-ротация/vacuum
  синхронны и автоматичны; механизм документируется (tech-writer). Это ЗАКРЫВАЕТ ротационную часть ОР-2/ОР-U4
  для systemd-установки. ГРАНИЦА: размерные лимиты journald — глобальные (per-host, не per-unit) → принятое
  отклонение П-3 / остаточный риск ОР-2. Проверяемо (AC8): в контейнере занизить `SystemMaxUse=`/
  `SystemMaxFileSize=` в drop-in, наполнить журнал синтетикой выше порога, `journalctl --disk-usage` показывает
  ограниченный рост (старые записи вытесняются), drop-in присутствует и описан в доке. (baseline §4
  «с ротацией»; spec AC8; ADR-004; closes command-exec ОР-2 / file-upload ОР-U4)

- [ ] **R-D2. Полу-установленное состояние при сбое install (неработоспособный/частично привилегированный
  сервис).** Если install прерывается на полпути (ошибка записи unit / сбой `daemon-reload`/`enable` /
  нехватка прав / недоступность менеджера), часть артефактов создана, часть нет → сервис в неопределённом
  состоянии (может стартовать с неполной конфигурацией или с привилегией без рабочего бинаря). Вероятность
  **средняя** / влияние **среднее** (битая регистрация; в худшем — частичная привилегия). Смягчение (AC11):
  `Install` при сбое на ЛЮБОМ шаге ОТКАТЫВАЕТ созданные им артефакты (unit/drop-in/созданные каталоги; НЕ
  пользователь — он переиспользуем, ADR-002/plan §Contracts) и возвращает понятную ошибку, оставляя систему
  в исходном (до установки) состоянии; повторная корректная установка после сбоя проходит штатно.
  Идемпотентность (AC9): повторный install при уже установленном → `ErrAlreadyInstalled` без дубликата/
  повреждения. Проверяемо: тест — смоделированный сбой на шаге install → ошибка + отсутствие остаточных
  артефактов регистрации; последующий install проходит штатно; двойной install → ровно одна регистрация.
  (baseline §3; spec AC9/AC11; plan §Contracts `Install`/`Uninstall`)

- [ ] **R-T2. Подмена/непредсказуемость менеджера или окружения сервиса (PATH/HOME/cwd).** (а) Если
  `systemctl`/`launchctl` вызываются по неполному пути с непредсказуемым `PATH`, либо через shell-строку с
  подставленными значениями — подмена бинаря менеджера / shell-инъекция; (б) если окружение сервиса
  (`PATH`/`HOME`/cwd) не задано явно — демон унаследует «грязное» окружение установщика, влияющее на резолв
  путей и (косвенно) на поиск бинарей при exec. Вероятность **низкая** / влияние **высокое** (исполнение
  подменённого менеджера / непредсказуемый резолв). Смягчение: `runManager` вызывает менеджер через
  `exec.Command(name, args...)` БЕЗ shell-интерполяции (никогда `sh -c <строка>`); имя менеджера — известное
  (`systemctl`/`launchctl`), значения передаются как ОТДЕЛЬНЫЕ аргументы (не конкатенация в строку);
  `exec.ErrNotFound` → `ErrManagerUnavailable` (AC12). Окружение сервиса задаётся ЯВНО и предсказуемо через
  `Environment=`/`EnvironmentVariables` (`XDG_CONFIG_HOME`/`XDG_STATE_HOME`/`HOME`, ADR-002); рабочая
  директория/окружение демона ограничены и предсказуемы (baseline §3); hardening `ProtectSystem=strict`/
  `ProtectHome=yes`/`PrivateTmp=yes` сужают видимую ФС. Проверяемо: grep `internal/service/exec.go` — вызов
  через `exec.Command(name, args...)`, нет `sh -c`/конкатенации аргументов в строку; grep шаблонов —
  `Environment=`/`EnvironmentVariables` задают `XDG_*`/`HOME` явно; unit содержит `ProtectSystem`/`ProtectHome`/
  `PrivateTmp`. (baseline §3 «exec без shell-интерполяции; окружение предсказуемо»; plan §Contracts
  `runManager`/§Шаблоны; research Q6)

- [ ] **R-V1. Новая внешняя рантайм-зависимость / сборка не из vendor / запуск на хосте.** Введение
  `kardianos/service` или иной внешней зависимости расширило бы supply-chain поверхность и нарушило офлайн-
  вендоринг; запуск/тест сервиса на хосте (а не в контейнере) исполнял бы инструмент произвольных команд вне
  изоляции (нарушение baseline §6). Вероятность **низкая** (ADR-001 выбрал stdlib) / влияние **среднее**.
  Смягчение: реализация ТОЛЬКО на stdlib (`text/template`/`os/exec`/`embed`/`runtime`/`os`/`context`) + уже
  вендоренный cobra (ADR-001); НОВЫХ внешних рантайм-зависимостей НЕТ (AC15); сборка/тесты офлайн из `vendor/`
  (`-mod=vendor`, без `go mod download`); запуск демона и сервисная интеграция — ТОЛЬКО в Docker (baseline §6,
  AC16). Проверяемо: grep `go.mod` — нет новой внешней зависимости от service-install; CI/локальный прогон в
  контейнере проходит из `vendor/`; интеграция запускается в systemd-контейнере, не на хосте. (baseline §6;
  spec AC15/AC16; ADR-001; plan §Trade-offs «новых зависимостей нет»)

> **Наследуемые риски (закрыты в `tls-transport`/`mcp-server`/`command-exec`/`file-upload`, service-install
> их НЕ переоткрывает).** Полное описание — соответствующие `threat-model.md`. Демон, обёрнутый сервисом,
> несёт тот же периметр:
> - R-1/R-M1 (обход auth) — Bearer-auth ДО любого tool; service-install это не меняет.
> - R-M-Origin (DNS-rebinding) — Origin-валидация + bind `127.0.0.1`; не меняется.
> - R7/R8 / command-exec/file-upload (rate-limit, лимит тела) — per-key/per-IP→429, `MaxBodyBytes`→413; не меняется.
> - command-exec R-12 / file-upload R-U12 (утечка секретов транспорта/содержимого) — fingerprint вместо ключа,
>   содержимое не логируется; не меняется.
> - **command-exec ОР-1 / file-upload ОР-U1 (исполнение/запись от root)** — ЗАКРЫВАЮТСЯ этой задачей (R-E1).
> - **command-exec ОР-2 / file-upload ОР-U4 (ротация аудита)** — ротационная часть ЗАКРЫВАЕТСЯ этой задачей
>   (R-D1) для systemd; per-host-граница journald — остаточный риск ОР-2 ниже.

## Принятые отклонения от baseline (red line 4)

> Согласно red line 4 и §Эскалация baseline, каждое отступление от буквы baseline зафиксировано здесь с
> риском, обоснованием и смягчением. Решения финальны (security — владелец baseline) и обязательны к
> исполнению нижними ролями (system-dev/developer/devops/tech-writer).

- [ ] **П-1. При привилегированном порте (<1024) `NoNewPrivileges` ОПУСКАЕТСЯ (ADR-003) — узкое окно
  ослабления hardening.**
  - **Что отклоняется:** research Q6/baseline-дух предполагают `NoNewPrivileges=yes` как «дешёвый» hardening
    всегда. При Port<1024 (ADR-003) директива не ставится — вместе с `AmbientCapabilities=CAP_NET_BIND_SERVICE`.
  - **Почему (обоснование security):** дословной фразы о совместимости `AmbientCapabilities` с
    `NoNewPrivileges=yes` в systemd.exec(5)/capabilities(7) research НЕ зафиксировал (Открытый вопрос); чтобы
    исключить недоказанный конфликт, при ambient `NoNewPrivileges` опускается. Это касается ТОЛЬКО редкого
    ручного выбора привилегированного порта; дефолт 7822 (≥1024) сохраняет `NoNewPrivileges=yes`.
  - **Смягчение/условия (ОБЯЗАТЕЛЬНЫ):** (1) `NoNewPrivileges=yes` СТАВИТСЯ всегда при Port≥1024 (дефолт и
    почти все установки); (2) опуск — ТОЛЬКО при наличии `AmbientCapabilities` (Port<1024); (3) прочие
    hardening-директивы (`ProtectSystem=strict`/`ProtectHome=yes`/`PrivateTmp=yes`) сохраняются независимо от
    порта; (4) демон всё равно непривилегированный (`User=raxd`, R-E1) и capability точечная (R-E2); (5)
    security ОБЯЗАН подтвердить необходимость/совместимость опуска при ревью unit. Проверяемо: SR-86, SR-87.
  - **Остаточный риск:** при Port<1024 процесс не имеет защиты `NoNewPrivileges` — ОР-1.

- [ ] **П-2. uninstall НЕ удаляет системного пользователя `raxd:raxd` (ADR-002) — осознанно оставляемый
  артефакт.**
  - **Что отклоняется:** «полное» удаление (AC10) буквально могло бы означать снятие всего, включая
    пользователя. Пользователь `raxd` сохраняется после uninstall.
  - **Почему (обоснование security):** удаление системного пользователя опаснее, чем его сохранение: он мог
    быть создан/переиспользован вне raxd; его удаление может осиротить файлы (UID-reuse — будущий другой
    пользователь унаследует владение остатками состояния). Сам по себе `raxd` без shell (`nologin`)/home и
    без unit — НЕ несёт автозапуска и НЕ даёт привилегий (capability в unit, который удалён).
  - **Смягчение:** (1) uninstall снимает ВСЁ, что несёт автозапуск/привилегию: stop+disable, unit/plist,
    drop-in (а с ним capability/hardening); (2) пользователь остаётся БЕЗ login-shell и БЕЗ автозапуска; (3)
    факт сохранения пользователя ДОКУМЕНТИРУЕТСЯ (tech-writer), чтобы оператор знал. Проверяемо: SR-89.
  - **Остаточный риск:** пользователь `raxd` остаётся в системе после uninstall (без привилегий/автозапуска) —
    приемлемо; при требовании «нулевого следа» оператор удаляет вручную — ОР-3.

- [ ] **П-3. Лимиты роста journald — глобальные (per-host), не per-raxd (ADR-004) — наследует command-exec
  ОР-2.**
  - **Что отклоняется:** baseline §4 «с ротацией» подразумевает ограничение журнала raxd; journald
    `SystemMaxUse=`/`SystemMaxFileSize=` ограничивают ВЕСЬ журнал хоста, а не только raxd.
  - **Почему (обоснование security):** размерные лимиты journald — глобальные (per-host, не per-unit;
    research Q5); per-raxd лимит требовал бы файлового вывода + logrotate (доп. артефакт, более инвазивно —
    отклонено ADR-004). Аудит уже пишется в stderr → journald без правок кода.
  - **Смягчение:** (1) рост журнала ОГРАНИЧЕН (синхронная ротация/vacuum journald) — главная цель AC8
    достигнута; (2) развёртывание — выделенный контейнер/хост под raxd (baseline §6), где «весь журнал хоста»
    практически и есть журнал raxd; (3) если оператору нужен изолированный per-raxd лимит — fallback
    `LogsDirectory=/var/log/raxd` + logrotate (ADR-004 §Альтернативы), документируется. Проверяемо: SR-90.
  - **Остаточный риск:** на хосте с другими сервисами лимит делится между ними — ОР-2.

## Остаточные риски / эскалации

- [ ] **ОР-1. При Port<1024 опущен `NoNewPrivileges` (R-E3/П-1).** Почему: исключение недоказанного конфликта
  ambient×NoNewPrivileges (research Открытый вопрос). Компенсирующий контроль: касается только редкого
  привилегированного порта; демон непривилегированный (`User=raxd`), capability точечная (CAP_NET_BIND_SERVICE),
  `ProtectSystem`/`ProtectHome`/`PrivateTmp` сохранены. **Эскалация:** перед релизом установки с
  привилегированным портом — security ОБЯЗАН верифицировать на целевой systemd, можно ли удержать
  `NoNewPrivileges=yes` вместе с ambient (если да — поставить и закрыть окно); согласовать с пользователем.
  (baseline §3; spec AC7; ADR-003)

- [ ] **ОР-2. Лимиты journald глобальны (per-host), не per-raxd (R-D1/П-3; наследует command-exec ОР-2).**
  Почему: документированное ограничение journald (размерные лимиты per-host). Компенсирующий контроль:
  рост журнала всё равно ограничен; выделенный контейнер/хост под raxd; fallback `LogsDirectory`+logrotate
  для per-raxd лимита. **Эскалация:** при размещении raxd на хосте с другими активно логирующими сервисами и
  жёстким лимитом диска — devops/distribution согласуют per-raxd политику (logrotate) или отдельный journal
  namespace; tech-writer документирует механизм и пороги. (baseline §4; spec AC8; ADR-004)

- [ ] **ОР-3. Системный пользователь `raxd` остаётся после uninstall (R-E4/П-2).** Почему: удаление
  системного пользователя опаснее сохранения (UID-reuse, осиротевшие файлы; ADR-002). Компенсирующий
  контроль: пользователь без shell/home и без unit не несёт автозапуска/привилегии; факт документируется.
  **Эскалация:** при требовании «нулевого следа» (compliance) — согласовать с пользователем ручное удаление
  пользователя ПОСЛЕ проверки, что ему не принадлежат нужные файлы вне raxd; tech-writer документирует
  процедуру. (baseline §3; spec AC10; ADR-002)

- [ ] **ОР-4. macOS-раскладка проверяется ВНЕ Docker (AC13) — интеграционная проверка не-root/прав/ротации
  на launchd в контейнере НЕВОЗМОЖНА.** Почему: Docker — Linux; launchd/`UserName=`/`StandardErrorPath`/
  newsyslog и создание macOS-пользователя (`dscl`) не воспроизводимы в контейнере (research Открытые
  вопросы). Это ОГРАНИЧЕНИЕ СРЕДЫ ПРОВЕРКИ, НЕ снятие требования: AC1–AC12 как контракт применимы к ОБЕИМ
  платформам. Компенсирующий контроль: для macOS обязательны (а) генерация ВАЛИДНОГО plist (включая
  `UserName=raxd`, права/расположение, `KeepAlive={SuccessfulExit=false}`) и (б) unit-тесты генератора plist
  и логики выбора платформы по `runtime.GOOS` (AC2/AC13); идентичная security-логика (валидация подстановок,
  не-root `UserName`, права 0644 на plist) покрыта тестами генератора. **Эскалация:** перед релизом
  macOS-сборки — system-dev/QA ОБЯЗАНЫ прогнать install→status→stop→uninstall + проверку euid!=0/прав/ротации
  на РЕАЛЬНОМ macOS вне Docker (ручной/CI-прогон); процедура создания/выбора macOS-пользователя и механика
  порта <1024 (root-bind/socket activation) и ротации (newsyslog) — открытые вопросы для system-dev.
  (baseline §6; spec AC13/AC2; research Открытые вопросы macOS)

- [ ] **ОР-5. systemd-в-Docker для интеграции требует privileged/cgroup-mount (research Q9) — среда проверки,
  не прод.** Почему: systemd как PID 1 в контейнере требует `--privileged` (или `CAP_SYS_ADMIN`) +
  cgroup-mount; точные mount-опции зависят от cgroup v2 на CI-хосте. Это СРЕДА ПРОВЕРКИ baseline §6, НЕ режим
  прод-развёртывания (контейнер/k8s как первичный режим — spec Out of Scope). Компенсирующий контроль:
  privileged-режим используется ТОЛЬКО в тест-контейнере для AC16; прод-раскладка — нативный systemd на
  не-контейнерном хосте под `raxd:raxd`. **Эскалация:** devops/qa уточняют cgroup v2 mount-опции на целевом
  CI; образ systemd предзагружается/кэшируется для офлайн-CI (стык с AC15). (baseline §6; spec AC16; research Q9)
