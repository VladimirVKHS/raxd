# Threat Model: command-exec — MCP-инструмент `execute_command` (выполнение произвольных команд на хосте)

Автор: security (raxd). Дата: 2026-05-22. Язык: русский.
Автор продукта: Vladimir Kovalev, OEM TECH.

Вход: `specs/command-exec/{spec.md (AC1–AC18, pm-guardian pass), plan.md, research.md}`,
ADR-001 (process-group kill), ADR-002 (формат аудита logfmt + системная ротация),
ADR-003 (входные лимиты + политика root), ADR-004 (аудит exec в handler);
`.claude/reference/SECURITY-BASELINE.ru.md` (§1–§6, ОБЯЗАТЕЛЬНЫЙ контракт),
`.claude/reference/MCP-INTEGRATION.ru.md`; готовый код `internal/server/*`
(цепочка `bodyLimit→recover→Host/Origin→auth→rate-limit→authSuccessAudit→mux`,
`AuditRecord`/`writeAudit`, `FingerprintFromContext`/`RemoteAddrFromContext`),
`internal/mcp/*` (`NewHandler`, `withAudit`, `pingHandler`/`serverInfoHandler`),
`internal/config/config.go`; УЖЕ ВЫПОЛНЕННЫЕ `specs/{tls-transport,mcp-server}/{threat-model,security-requirements}.md`.

> **Это самая опасная фича продукта.** `execute_command` по назначению исполняет произвольный
> бинарь на хосте по сетевому запросу ИИ-агента — функционально это удалённое исполнение кода
> уровня SSH. Любая дыра здесь = RCE на хосте. Поэтому модель угроз ниже исчерпывающе покрывает
> baseline §3 (выполнение команд) и §4 (аудит без секретов), а наследуемый периметр §1/§2 НЕ
> переоткрывает (см. «Контекст наследования»). Каждый риск имеет смягчение; невыполнимое/отложенное
> — в «Остаточные риски / эскалации». Принятые отступления от буквы baseline — в отдельном разделе
> «Принятые отклонения от baseline» (red line 4).

## Контекст наследования

`execute_command` регистрируется ВНУТРИ уже готового и зашлюзованного контракта:
- **Транспорт `tls-transport`** (риски R1–R15, требования SR-1…SR-26): TLS 1.3, Bearer-auth ДО
  обработки (`keystore.Verify`, constant-time), Host/Origin-валидация, rate-limit per-key/per-IP,
  аудит без секретов, bind `127.0.0.1`, лимит тела, таймауты, graceful shutdown. **Не переопределяется.**
- **MCP-слой `mcp-server`** (риски R-M1…R-M7, требования SR-27…SR-39): монтаж SDK-handler за той
  же цепочкой, JSON-RPC-устойчивость, `additionalProperties:false`, MCP-аудит вызова, точка
  расширения `mcp.AddTool`. **Не переопределяется.** `mcp-server/threat-model.md` ОР-М2 ПРЯМО
  передаёт сюда обязательство реализовать все контроли baseline §3 — данная модель его исполняет.

Эта модель фиксирует ТОЛЬКО новое, что вносит `execute_command`: фактический запуск дочернего
процесса на хосте и весь класс угроз исполнения команд (shell-инъекция, подмена бинаря, осиротевшие
процессы, инъекция через окружение, обход allowlist, DoS входом/выводом/временем, эскалация
привилегий, утечка секретов через аргументы/вывод/ошибки), а также расширение аудит-записи полями
команды.

## Активы

- **Хост и его файловая система / процессы / сеть** — конечная цель атаки. `execute_command`
  запускает реальный бинарь с реальными правами демона. Компрометация = полный контроль над хостом
  в пределах прав процесса демона. Это НОВЫЙ актив под прямой угрозой (в `mcp-server` инструменты
  без побочных эффектов; здесь — с побочными эффектами на хост).
- **Привилегии процесса демона (UID/GID, особенно euid==0)** — дочерний процесс наследует их как
  есть. Если демон работает от root, любая исполняемая команда исполняется от root → эскалация до
  полного root-RCE. Актив: фактический уровень привилегий исполнения.
- **API-ключи (тело `rax_live_…`)** — главный секрет транспорта. exec-слой к телу ключа доступа НЕ
  имеет (только `Fingerprint` из ctx). Утечка ключа через результат/ошибку/exec-аудит = полный
  доступ к инструменту RCE.
- **Секреты ВНУТРИ аргументов команды** — НОВЫЙ актив-источник утечки. Клиент может прислать
  `command/args`, содержащие пароль/токен (`mysql -p<password>`, `curl -H "Authorization: …"`,
  `git clone https://user:token@host`). baseline §4 требует логировать «команда+аргументы», но также
  «никаких секретов в логах» — прямой конфликт. См. R-13 и раздел «Принятые отклонения».
- **Аудит-лог exec** — журнал каждого вызова `execute_command` (timestamp, fingerprint, команда+args,
  exit code, длительность, remote, результат). Ключевой ресурс расследования RCE-инцидентов.
  Отсутствие записи о вызове / запись с секретом / нечитаемая запись = repudiation или утечка.
- **Память процесса демона** — гигантский вывод команды (OOM, AC11), мегабайтные/многочисленные
  аргументы (argv-DoS, Q10/AC11), осиротевшие процессы (исчерпание PID/CPU/памяти после таймаута,
  AC6) могут исчерпать ресурсы демона.
- **Динамический загрузчик / окружение дочернего процесса** — `LD_PRELOAD`/`LD_LIBRARY_PATH`/
  `DYLD_INSERT_LIBRARIES`/`IFS` в наследуемом окружении = вектор перехвата исполнения (CWE-426/427).
- **PATH и cwd дочернего процесса** — относительный путь бинаря из cwd (`exec.ErrDot`), небезопасный
  cwd (выход вверх, чтение/запись чувствительных каталогов) — векторы подмены/path traversal.
- **`vendor/`-дерево** — целостность завендоренного кода (наследуется; новых внешних зависимостей
  command-exec НЕ вводит — всё на stdlib `os/exec`/`syscall`/`io`/`os`/`time` + уже вендоренные SDK
  и `charmbracelet/log`, plan §Trade-offs).

## Поверхность атаки

- **MCP-инструмент `execute_command`** (`tools/call`, метод `execute_command`) — единственная
  поверхность исполнения (AC1). Регистрируется `sdkmcp.AddTool(s, execTool(), execHandler(...))` в
  `internal/mcp/server.go`, рядом с `ping`/`server_info`, ЗА той же наследуемой цепочкой
  `bodyLimit→recover→Host/Origin→auth→rate-limit→authSuccessAudit→mux`. Отдельного не-`/mcp`
  эндпоинта и CLI-подкоманды `raxd exec` НЕТ (AC1, spec Out of Scope).
- **Вход инструмента** `ExecInput{command, args, timeout_ms, cwd}` — недоверенный JSON-RPC-ввод от
  аутентифицированного клиента: имя/путь бинаря, массив аргументов, таймаут, рабочая директория.
  Поля `env` НЕТ (AC3, отклонено в v1 — spec Out of Scope). Схема строгая
  (`additionalProperties:false`, AC3).
- **Раннер `internal/cmdexec`** — собирает и запускает `exec.Cmd` на хосте: allowlist-проверка,
  валидация cwd, whitelist env, process-group + kill, capped-writers вывода.
- **Дочерний процесс и его дерево потомков** — реальный исполняемый код на хосте; потомки могут
  пережить родителя при некорректном завершении (AC6).
- **exec-аудит-запись** — расширенный `AuditRecord` (поля command/args/exit/duration/timed_out),
  пишется самим `execHandler` (ADR-004), сериализуется `writeAudit` (logfmt, ADR-002).
- **Кто атакует:**
  - **Аутентифицированный MCP-клиент (легитимный ключ, но скомпрометированный/злонамеренный агент)**
    — ОСНОВНОЙ актор. Прошёл auth/Origin/rate-limit; шлёт `execute_command` с shell-метасимволами,
    относительным путём бинаря, командой-обходом allowlist, гигантским выводом/argv, долгой командой,
    форк-бомбой, секретами в аргументах. Цель: RCE, эскалация, DoS, обход аудита, утечка.
  - **Внешний клиент без ключа** — отсекается наследуемой auth ДО инструмента (R-M1/SR-27); до
    запуска команды не доходит (AC12).
  - **Локальный/инфраструктурный фактор** — демон, ошибочно запущенный от root (AC9), небезопасная
    раскладка сервиса (отдельная задача service-install).
  - **Браузерный контекст / DNS-rebinding** — отсекается наследуемой Origin-валидацией (R-M-Origin).

## Угрозы (STRIDE-ориентир)

- **S (Spoofing) — обход аутентификации перед исполнением.** Если бы `execute_command` был доступен
  мимо цепочки (отдельный эндпоинт/CLI/собственный auth-канал SDK), команда исполнялась бы анонимно.
  Смягчение: единственная поверхность — MCP-инструмент за наследуемой цепочкой; auth ДО инструмента;
  отдельного эндпоинта/CLI нет (AC1/AC12) — R-1.
- **T (Tampering) — подмена исполняемого бинаря / инъекция через shell или окружение.** (а) Запуск
  через `sh -c <строка>` дал бы shell-инъекцию (CWE-78); (б) относительный путь из cwd / манипуляция
  PATH → запуск подменённого бинаря; (в) `LD_PRELOAD`/`DYLD_*`/`IFS` в окружении → перехват
  исполнения (CWE-426/427). Смягчение: `exec.CommandContext(ctx, bin, args...)` без shell (AC2);
  отвержение `exec.ErrDot` (AC8); явный env-whitelist без опасных переменных (AC10) — R-2, R-4, R-7.
- **R (Repudiation) — вызов не зафиксирован / нечитаем / без полей.** Без аудит-записи RCE
  невозможно расследовать; нечитаемый формат / пропуск deny-ветки = слепая зона. Смягчение:
  `execHandler` сам пишет РОВНО одну машиночитаемую запись во всех ветках success/deny/fail
  (ADR-004), logfmt-формат (ADR-002), поля по AC13 — R-10, R-11.
- **I (Information disclosure) — утечка секрета через результат/ошибку/аудит.** Тело ключа,
  приватный TLS-ключ ИЛИ секрет внутри аргументов команды просачиваются в `stdout/stderr`-результат,
  текст ошибки или exec-аудит. Смягчение: fingerprint вместо ключа (AC15); нейтральные сообщения об
  ошибках (AC8); явная политика логирования аргументов с предупреждением оператора (R-13,
  «Принятые отклонения») — R-12, R-13.
- **D (Denial of service) — исчерпание ресурсов входом/выводом/временем/процессами.** Гигантский
  вывод (OOM), мегабайтные/многочисленные аргументы (argv-DoS), долгая команда (занятый слот),
  осиротевшие процессы после таймаута, форк-бомба. Смягчение: capped-writer вывода (AC11), входные
  лимиты `max_args`/`max_arg_len` ДО запуска (ADR-003), жёсткий максимум таймаута + kill (AC5),
  process-group kill всего дерева (ADR-001, AC6); форк-бомба — частичное смягчение + остаточный риск
  (ОР-3) — R-3, R-5, R-6, R-8, R-9.
- **E (Elevation of privilege) — исполнение от root / setuid.** Демон от root → команды от root;
  использование setuid/sudo. Смягчение: НЕ устанавливать `SysProcAttr.Credential` (наследование
  uid/gid как есть, без setuid, AC9); детекция euid==0 + WARN при каждом вызове (AC9); основная
  защита — не-root раскладка (baseline §3) + контейнер (baseline §6). Политика root = WARN —
  принятое отклонение с остаточным риском (R-14, ОР-1) — R-14.

## Риски (риск → вероятность/влияние → смягчение)

- [ ] **R-1. Исполнение мимо аутентификации (анонимный RCE).** Если бы `execute_command` имел
  отдельную поверхность (не-`/mcp` эндпоинт, CLI `raxd exec`, собственный auth-канал SDK) или был
  смонтирован раньше auth — команда исполнялась бы без валидного ключа. Вероятность **низкая**
  (контракт AC1/plan фиксирует единственную поверхность за цепочкой) / влияние **критическое**
  (анонимный RCE). Смягчение: единственная поверхность — MCP-инструмент `execute_command`,
  зарегистрированный `sdkmcp.AddTool` ВНУТРИ той же цепочки
  `bodyLimit→recover→Host/Origin→auth→rate-limit→authSuccessAudit→mux`; `auth` (`keystore.Verify`)
  отрабатывает ДО инструмента; отдельного эндпоинта/CLI/процесса/порта НЕТ; SDK-handler НЕ вводит
  второго auth-канала (наследуется R-M1). Проверяемо: неаутентифицированный `tools/call
  execute_command` → 401, никакая команда НЕ запущена; нет второй поверхности исполнения.
  (baseline §1/§3; spec AC1/AC12/AC16; threat-model `mcp-server` R-M1/ОР-М2)

- [ ] **R-2. Shell-инъекция через метасимволы (CWE-78).** Запуск команды через `sh -c <строка>`
  превратил бы любой метасимвол (`;`, `|`, `$()`, `&&`, `>`, `` ` ``) в инъекцию команд оболочки.
  Вероятность **средняя** (классическая ошибка «для удобства пайпов») / влияние **критическое**
  (произвольный RCE сверх запрошенного бинаря). Смягчение: запуск ТОЛЬКО формой `exec.CommandContext(
  ctx, bin, args...)` (бинарь + список аргументов), shell НЕ привлекается ни при каких входных
  данных; метасимволы попадают в процесс как литеральные аргументы. Проверяемо (тест-вектор):
  вход `command="echo"`, `args=["a; touch /tmp/pwned"]` → создаётся литеральный аргумент
  `a; touch /tmp/pwned`, файл `/tmp/pwned` НЕ создаётся (побочного эффекта, который дал бы только
  shell, нет); grep по `internal/cmdexec` — нет `sh -c`/`bash -c`/`exec.Command("sh", …)`.
  (baseline §3 «без shell-интерполяции»; spec AC2; research Q1)

- [ ] **R-3. DoS гигантским выводом (OOM демона).** «Болтливая» команда (`yes`, `cat /dev/urandom`)
  печатает неограниченный stdout/stderr → буфер растёт без предела → OOM процесса демона.
  Вероятность **средняя** / влияние **высокое** (падение демона = отказ всего сервиса). Смягчение:
  `cmd.Stdout`/`cmd.Stderr` = capped-writer с потолком `max_output_bytes` (дефолт 1 MiB на каждый,
  ADR-003); при превышении вывод обрезается до лимита, флаг `stdout_truncated`/`stderr_truncated` =
  `true`, остаток дренируется без ошибки Write (чтобы не подвесить процесс). Память на вывод
  ограничена сверху `2 × max_output_bytes`. Проверяемо: команда, печатающая больше лимита, →
  обрезанный вывод + `*_truncated:true`, рост памяти ограничен. (baseline §3 «окружение/вывод
  ограничены»; spec AC11; research Q3)

- [ ] **R-4. Подмена бинаря относительным путём из cwd / через PATH (CWE-426/427).** Имя бинаря с
  относительным путём (`./malicious`, `bin/x`) или резолв по подконтрольному PATH запускает не тот
  бинарь, что ожидался. Go 1.19+ возвращает `exec.ErrDot` для неявного/явного относительного пути из
  cwd. Вероятность **средняя** / влияние **высокое** (RCE подменённым бинарём). Смягчение: раннер
  отвергает имя бинаря, дающее `exec.ErrDot` (относительный путь из cwd) → `isError:true`, команда
  НЕ запускается; PATH в дочернем окружении задаётся явным whitelist (значение из окружения демона),
  НЕ принимается от клиента (нет поля `env`, AC3); резолв чистого имени идёт по контролируемому PATH
  через `LookPath`. Проверяемо: вход с относительным именем (`./x`) → `isError` (`ErrDot`), не
  запущено; PATH дочернего процесса = значение из whitelist, не из клиентского ввода. (baseline §3;
  spec AC2/AC8/AC10; research Q1, Q4)

- [ ] **R-5. argv-DoS: многочисленные/мегабайтные аргументы (исчерпание памяти до execve).**
  `exec.Command` не имеет встроенного лимита на число/размер аргументов; клиент шлёт тысячи
  аргументов или мегабайтные строки → рост памяти демона ДО того, как ядро отвергнет запуск
  (`E2BIG`/ARG_MAX — поздняя грубая защита, только на execve). Вероятность **средняя** / влияние
  **высокое** (OOM/деградация демона). Смягчение: прикладные лимиты входа, проверяемые ДО запуска:
  `max_args` (дефолт 256), `max_arg_len` (дефолт 128 KiB на аргумент) — превышение → `isError:true`,
  команда НЕ запускается; (наследуется лимит тела запроса `MaxBodyBytes` 1 MiB как внешняя граница).
  Проверяемо: `len(args) > max_args` или `len(arg) > max_arg_len` → `isError`, не запущено.
  (baseline §3; spec AC11; research Q10; ADR-003)

- [ ] **R-6. DoS долгой командой / зависшим процессом.** Команда без таймаута заняла бы слот
  исполнения навсегда; клиент мог бы запросить произвольно большой `timeout_ms`. Вероятность
  **средняя** / влияние **среднее**. Смягчение: каждая команда ограничена таймаутом через `context`
  (дефолт `default_timeout_ms` 30s); запрошенный `timeout_ms` сверх жёсткого `max_timeout_ms`
  (дефолт 5 мин) → `isError:true` ДО запуска; по истечении — kill (см. R-8). Проверяемо: команда
  дольше таймаута прерывается (`timed_out:true`); `timeout_ms` > max → `isError`, не запущено.
  (baseline §3 «таймаут на каждую команду»; spec AC5; ADR-003)

- [ ] **R-7. Инъекция через окружение (`LD_PRELOAD`/`LD_LIBRARY_PATH`/`DYLD_*`/`IFS`).** При
  `Cmd.Env=nil` дочерний процесс слепо наследует ВСЁ окружение демона; опасные переменные
  динамического загрузчика/shell перехватывают исполнение (CWE-426/427). Дополнительно: приём
  произвольного `env` от клиента дал бы прямую инъекцию. Вероятность **средняя** / влияние
  **высокое**. Смягчение: `Cmd.Env` = ЯВНЫЙ whitelist `env_whitelist` (дефолт `["PATH","HOME","LANG",
  "TERM"]`, значения берутся из окружения демона), БЕЗ слепого наследования; `LD_PRELOAD`/
  `LD_LIBRARY_PATH`/`DYLD_INSERT_LIBRARIES`/`IFS` в whitelist отсутствуют и в дочерний процесс НЕ
  попадают; поле `env` во входе инструмента НЕ принимается (AC3, spec Out of Scope). Проверяемо:
  дочерний процесс видит ТОЛЬКО whitelist-переменные; `LD_PRELOAD`/`DYLD_INSERT_LIBRARIES`/`IFS`
  отсутствуют в его окружении даже если заданы у демона. (baseline §3 «окружение ограничено»; spec
  AC10; research Q4; ADR-003)

- [ ] **R-8. Осиротевшие/неубиваемые процессы после таймаута/отмены (AC6).** `exec.CommandContext`
  по умолчанию убивает ТОЛЬКО головной процесс; потомки осиротевают (PPID=1) и продолжают потреблять
  ресурсы; при `WaitDelay=0` долгоживущий потомок с открытым пайпом подвешивает `Wait`. Вероятность
  **высокая** (любая команда, порождающая детей) / влияние **высокое** (накопление зомби/нагрузки,
  скрытое исполнение после «завершения»). Смягчение: `cmd.SysProcAttr={Setpgid:true}` (новая группа
  процессов) + custom `cmd.Cancel`→`killGroup`=`syscall.Kill(-pgid, SIGKILL)` (kill всей группы) +
  ненулевой `cmd.WaitDelay` (страховка от зависших пайпов); при отмене контекста (обрыв клиента) —
  то же завершение группы. Платформенный код под `//go:build unix` (Linux+darwin). Проверяемо: после
  таймаута/отмены долгой команды живых дочерних процессов группы НЕ остаётся; `timed_out:true`.
  (baseline §3; spec AC5/AC6; research Q2; ADR-001)

- [ ] **R-9. Форк-бомба / исчерпание системных ресурсов командой.** Команда `:(){ :|:& };:` или
  массовый форк исчерпывает PID/CPU/RAM хоста до срабатывания таймаута. Вероятность **низкая**
  (требует злонамеренного аутентифицированного клиента) / влияние **высокое**. Смягчение
  (частичное): process-group kill (R-8) убивает всё дерево при таймауте/отмене; whitelist env и
  отсутствие shell сужают, но НЕ исключают форк-бомбу из самого бинаря. Полная защита (cgroups/
  rlimits/seccomp/namespaces) — НЕ в scope v1 (spec Out of Scope). Остаточный риск зафиксирован
  (ОР-3): изоляция возлагается на контейнерное развёртывание (baseline §6) + не-root + allowlist.
  Проверяемо (частично): kill группы убивает дерево при таймауте; полное ресурсное ограничение —
  в ОР-3. (baseline §3/§6; spec AC6 + Out of Scope; ADR-001)

- [ ] **R-10. Вызов `execute_command` не зафиксирован в аудите (repudiation).** Если запись не
  пишется в какой-то ветке (особенно deny по allowlist / превышение входных лимитов внутри раннера,
  где generic `withAudit` дал бы лишь «fail» без exec-полей или двойную запись), RCE-инцидент
  невозможно расследовать. Вероятность **средняя** / влияние **высокое**. Смягчение: ADR-004 —
  `execute_command` НЕ оборачивается generic `withAudit`; `execHandler` сам пишет РОВНО одну
  exec-аудит-запись через `server.AuditFn` во ВСЕХ ветках: success/таймаут → `Result:"success"` (+
  `TimedOut`), deny → `Result:"deny"`, fail → `Result:"fail"`; плюс отдельный root-WARN при euid==0.
  Поля: timestamp(UTC), fingerprint (из ctx, не ключ), `tool=execute_command`, команда+args, exit
  code, duration, remote, результат. Проверяемо: успешный вызов → запись с fingerprint+команда+args+
  exit+duration+remote+result; отклонённый вызов → запись с fingerprint+команда+args+remote+
  result(deny); запись РОВНО одна за вызов. (baseline §4 «аудит каждого действия»; spec AC13;
  ADR-004; threat-model `mcp-server` R-M3)

- [ ] **R-11. Нечитаемый/несовместимый формат аудит-записи.** Если exec-запись пишется в формате,
  несовместимом с существующим аудитом (или ломает формат не-exec записей), машинный разбор
  невозможен и нарушается единый канал. Вероятность **низкая** / влияние **среднее**. Смягчение:
  ADR-002 — единый logfmt (строгий key=value, машиночитаемый); поля команды/exit/duration добавлены
  в `AuditRecord` как ОПЦИОНАЛЬНЫЕ, логируются ТОЛЬКО при `Tool=="execute_command"` (по аналогии с
  `tool=` при `Tool!=""`); формат не-exec записей (AUTH/FAIL/DENY/RATE/MCP-ping) НЕ меняется.
  Ротация — системная (journald/logrotate, ОР-2). Проверяемо: exec-запись парсится как структурная
  logfmt-запись существующего формата; не-exec записи (наследуемые тесты tls-transport/mcp-server)
  не ломаются. (baseline §4 «структурно, с ротацией»; spec AC14; ADR-002; «Принятые отклонения»)

- [ ] **R-12. Утечка тела ключа / приватного TLS-ключа через результат/ошибку/аудит.** Тело
  API-ключа или приватный TLS-ключ попадают в `stdout/stderr`-результат, текст JSON-RPC-ошибки или
  exec-аудит. Вероятность **низкая** (exec-слой к телу ключа доступа НЕ имеет — только `Fingerprint`
  из ctx) / влияние **критическое**. Смягчение: exec-аудит оперирует ТОЛЬКО `server.FingerprintFromContext`
  (тело ключа MCP/exec-слою недоступно по контракту); сообщения об ошибках нейтральны (AC8); раннер
  не читает keystore/TLS-файлы. Проверяемо: предъявленный полный ключ как ПОДСТРОКА ОТСУТСТВУЕТ в
  захваченном exec-аудите И в теле MCP-ответа; приватный TLS-ключ не встречается. (baseline §4
  «никаких секретов в логах», §1; spec AC15; threat-model `mcp-server` R-M2; наследует tls-transport
  SR-21)

- [ ] **R-13. Утечка секрета, ПЕРЕДАННОГО клиентом в аргументах команды.** Клиент шлёт команду с
  секретом в аргументе (`mysql -psecret`, `curl -H "Authorization: Bearer T"`, `git clone
  https://u:tok@h`); baseline §4 требует логировать «команда+аргументы», но также «никаких секретов
  в логах» — ПРЯМОЙ КОНФЛИКТ. Если args логируются как есть, секрет оператора/третьей стороны попадёт
  в аудит. Вероятность **высокая** (типичные CLI-утилиты принимают секреты в argv) / влияние
  **среднее** (секрет в аудит-логе, доступ к которому ограничен оператором/системой). РЕШЕНИЕ
  security (см. «Принятые отклонения», П-3): args логируются КАК ЕСТЬ (без маскирования) —
  обоснованно: (а) надёжное определение «что секрет» в произвольном argv невозможно (нет общего
  паттерна; маскирование дало бы ложное чувство безопасности и риск пропуска); (б) полнота аудита
  критична для расследования RCE — урезание args ослабило бы главный контроль baseline §4; (в)
  секрет в argv — это секрет КЛИЕНТА/КОМАНДЫ, а не raxd; тело API-ключа raxd в аудит НЕ попадает
  (R-12). Смягчение/компенсирующий контроль: (1) аудит-лог — ограниченного доступа (оператор/journald,
  baseline §4/§6); (2) ОБЯЗАТЕЛЬНОЕ предупреждение в документации (tech-writer): «аргументы команд
  логируются дословно; не передавайте секреты в argv — используйте файлы/переменные окружения
  целевой утилиты на стороне команды»; (3) тело API-ключа raxd под защитой R-12 (не путать с
  секретами в argv). Проверяемо: args в аудите соответствуют присланным; в доке есть явное
  предупреждение о логировании argv. ОСТАТОЧНЫЙ риск зафиксирован (ОР-4). (baseline §4 «команда+
  аргументы» ↔ «никаких секретов»; spec AC13/AC15; «Принятые отклонения» П-3)

- [ ] **R-14. Эскалация: исполнение команд от root (демон с euid==0).** Если демон работает от root,
  каждая команда исполняется от root → подмена/чтение любых файлов, полный контроль хоста.
  Вероятность **средняя** (контейнеры/dev часто стартуют от root; неверная раскладка сервиса) /
  влияние **критическое**. РЕШЕНИЕ security (см. «Принятые отклонения», П-2): политика = ТОЛЬКО WARN
  (не отказ) по умолчанию, с ТРЕБОВАНИЕМ опционального конфиг-флага `exec.deny_root` для жёсткого
  отказа. Обоснование WARN-дефолта: полный отказ сломал бы легитимные старт-от-root/dev-сценарии;
  обязательный контроль AC9 — детекция + WARN. Смягчение: (1) раннер НЕ устанавливает
  `SysProcAttr.Credential`/setuid/sudo — наследует uid/gid демона как есть, не повышает привилегии
  (AC9-тест №1); (2) `os.Geteuid()==0` → WARN-аудит при КАЖДОМ вызове (AC9-тест №2); (3) опциональный
  `exec.deny_root` (дефолт `false`) — при `true` и euid==0 → `isError:true`, команда не запускается
  (даёт оператору жёсткий рычаг); (4) ОСНОВНАЯ защита — не-root раскладка сервиса (baseline §3,
  задача service-install) + контейнер (baseline §6). Проверяемо: дочерний процесс под тем же uid,
  что демон (Credential не задан); euid==0 → WARN-запись каждый вызов; при `deny_root:true` и euid==0
  → `isError`, не запущено. ОСТАТОЧНЫЙ риск при WARN-дефолте зафиксирован (ОР-1). (baseline §3 «демон
  НЕ от root»; spec AC9; ADR-003; «Принятые отклонения» П-2)

- [ ] **R-15. Path traversal / небезопасный cwd.** Пустой `cwd` → команда исполняется в «случайном»
  cwd процесса демона; произвольный `cwd` (`/`, `/etc`, несуществующий путь) → непредсказуемое/
  опасное окружение исполнения, доступ к чувствительным каталогам. Вероятность **средняя** / влияние
  **среднее**. Смягчение: при отсутствии `cwd` используется безопасный дефолт `default_cwd` (дефолт
  `/tmp` — предсказуемый, не `/`, не cwd демона); при заданном `cwd` он валидируется до запуска
  (`os.Stat`+`IsDir`: существует И является каталогом) — невалидный → `isError:true`, команда НЕ
  запускается; `Cmd.Dir` устанавливается явно (не пустой). Проверяемо: команда видит ожидаемый cwd
  (дефолт при пустом вводе); невалидный `cwd` (несуществующий/файл) → `isError`, не запущено.
  (baseline §3 «рабочая директория ограничена»; spec AC10; research Q5)

> **Наследуемые риски (закрыты в `tls-transport`/`mcp-server`, command-exec их НЕ переоткрывает).**
> Полное описание — `specs/tls-transport/threat-model.md` и `specs/mcp-server/threat-model.md`.
> `execute_command` сидит за тем же периметром:
> - R-M1/R1 (обход auth) — auth ДО инструмента; база для R-1 здесь.
> - R-M-Origin/R2 (DNS-rebinding) — Origin present&invalid→403; bind `127.0.0.1`.
> - R-M2/R6 (утечка секретов транспорта) — fingerprint вместо ключа; база для R-12.
> - R7/R8 (брутфорс/rate-limit) — per-key/per-IP→429 ДО инструмента (AC16); защищает и exec.
> - R-M4 (паника на битом JSON-RPC) — корректные JSON-RPC-ошибки; `recoverMiddleware`; база для AC17.
> - R-M6 (supply chain SDK) — вендоринг офлайн, `go.sum`; command-exec НОВЫХ зависимостей не вводит.
> - R-M5/ОР-М2 (преждевременный exec) — ИСПОЛНЯЕТСЯ здесь: `execute_command` теперь зарегистрирован
>   и снабжён всеми контролями baseline §3.

## Принятые отклонения от baseline (red line 4)

> Согласно red line 4 и §Эскалация baseline, каждое отступление от буквы baseline зафиксировано
> здесь с риском, обоснованием и смягчением. Эти решения — финальные (security принимает их как
> владелец baseline) и обязательны к исполнению нижними ролями.

- [ ] **П-1. Формат аудита: logfmt (key=value) вместо буквального baseline §4 «Структурно (JSON)».**
  - **Что отклоняется:** baseline §4 буквально требует «Структурно (JSON), с ротацией». Принимается
    `LogfmtFormatter` (строгий, машиночитаемый, структурный key=value), НЕ JSON (ADR-002).
  - **Почему (обоснование security):** транспорт (`tls-transport`) и MCP-слой (`mcp-server`) УЖЕ
    пишут весь аудит в key=value через `charmbracelet/log`; переход на JSON фрагментировал бы
    формат аудита (часть записей key=value, часть JSON), сломал бы AC14 «не ломать формат
    не-`execute_command` записей» и затронул бы уже зашлюзованные `tls-transport`/`mcp-server` (вне
    scope command-exec по коду). logfmt — стандартный СТРУКТУРНЫЙ и МАШИНОЧИТАЕМЫЙ формат: ДУХ
    требования baseline §4 «структурно, машиночитаемо» сохраняется, меняется лишь сериализация.
  - **Смягчение/условия принятия (ОБЯЗАТЕЛЬНЫ):** (1) формат остаётся СТРОГО ПАРСИМЫМ logfmt
    (`LogfmtFormatter`, НЕ human-readable TextFormatter со styling — TextFormatter недопустим, т.к.
    не гарантирует машинный разбор); (2) единый канал и схема с остальным аудитом; (3) ротация —
    СИСТЕМНАЯ (journald для systemd / logrotate при файловом выводе), требование ротации baseline §4
    НЕ снимается, а делегируется дистрибуции/доке (см. ОР-2). Проверяемо: SR-26, SR-23.
  - **Остаточный риск:** при чисто файловом выводе без journald нужен внешний `logrotate`-конфиг —
    зафиксировано как требование к дистрибуции/доке (ОР-2).

- [ ] **П-2. Политика root (AC9): по умолчанию WARN, не отказ; обязателен опциональный жёсткий
  отказ `exec.deny_root`.**
  - **Что отклоняется:** baseline §3 предписывает «демон НЕ от root». В реальности демон МОЖЕТ
    оказаться запущен от root (контейнер/dev/неверная раскладка). architect (ADR-003) рекомендовал
    только WARN. security ПРИНИМАЕТ WARN как дефолт, НО УСИЛИВАЕТ требование: ОБЯЗАТЕЛЕН конфиг-флаг
    `exec.deny_root` (дефолт `false`), дающий оператору жёсткий отказ исполнять при euid==0.
  - **Почему (обоснование security):** полный безусловный отказ сломал бы легитимные сценарии
    (контейнер часто стартует от root до сброса привилегий; dev-окружение); AC9 фиксирует
    обязательным лишь детекцию+WARN. Но оставить ТОЛЬКО WARN недостаточно для прод-инсталляций с
    высокими требованиями — поэтому security требует наличие опционального hard-fail.
  - **Смягчение:** (1) детекция `os.Geteuid()==0` + WARN-аудит при КАЖДОМ вызове (обязательно, AC9);
    (2) опциональный `exec.deny_root` (дефолт `false`) → при `true` и euid==0 команда НЕ запускается
    (`isError`); (3) НЕ повышать привилегии (никогда setuid/sudo/`Credential`, AC9); (4) ОСНОВНАЯ
    защита вне раннера — не-root раскладка сервиса (baseline §3, service-install) + контейнер
    (baseline §6). Проверяемо: SR-12, SR-13, SR-14.
  - **Остаточный риск:** при дефолте WARN команды от root-демона ИСПОЛНЯЮТСЯ (R-14) — зафиксировано
    в ОР-1; смягчается раскладкой + контейнером + доступным `deny_root`.

- [ ] **П-3. Логирование аргументов команды «как есть» (без маскирования), несмотря на возможный
  секрет в argv.**
  - **Что отклоняется:** baseline §4 требует одновременно «логировать команда+аргументы» И «никаких
    секретов в логах». Аргумент команды МОЖЕТ содержать секрет (`mysql -psecret`). security ПРИНИМАЕТ
    логирование args ДОСЛОВНО, осознавая конфликт.
  - **Почему (обоснование security):** (1) надёжно определить «секрет ли данный токен argv»
    невозможно — нет общего паттерна для произвольных утилит; маскирование по эвристике дало бы
    ЛОЖНОЕ чувство безопасности (пропуски) и риск порчи легитимных аргументов; (2) полнота args
    КРИТИЧНА для расследования RCE — урезание/маскирование ослабило бы главный контроль baseline §4
    (аудит как доказательная база); (3) секрет в argv — это секрет КЛИЕНТА/целевой команды, а не
    raxd; тело API-ключа raxd в аудит НЕ попадает (R-12, отдельный инвариант).
  - **Смягчение/компенсирующие контроли:** (1) аудит-лог ограниченного доступа (оператор/journald,
    baseline §4/§6); (2) ОБЯЗАТЕЛЬНОЕ предупреждение в документации (tech-writer, SR-22):
    «аргументы команд логируются дословно; НЕ передавайте секреты в argv — используйте механизмы
    целевой утилиты (файл/переменная окружения команды)»; (3) инвариант R-12: тело API-ключа raxd /
    приватный TLS-ключ в аудит/результат/ошибку НЕ попадают (отделено от секретов клиента в argv).
    Проверяемо: SR-21, SR-22.
  - **Остаточный риск:** секрет, переданный клиентом в argv, окажется в аудит-логе (R-13) — принят
    осознанно, зафиксирован в ОР-4.

## Остаточные риски / эскалации

- [ ] **ОР-1. Команды от root-демона при дефолтной политике WARN (R-14/П-2).** Почему: WARN-дефолт
  выбран ради совместимости (см. П-2); при euid==0 и `deny_root:false` команды исполняются от root.
  Компенсирующий контроль: обязательная детекция+WARN каждый вызов; доступный `exec.deny_root` для
  жёсткого отказа; не-root раскладка сервиса (baseline §3, service-install) + контейнер (baseline §6)
  как основная защита. **Эскалация:** перед релизом ЛЮБОЙ продакшн-инсталляции, где демон может
  стартовать от root и не сбрасывает привилегии, — согласовать с пользователем включение
  `exec.deny_root:true` ИЛИ подтвердить не-root раскладку. (baseline §3)

- [ ] **ОР-2. Ротация аудит-лога в коде НЕ реализуется — делегирована системе (П-1/ADR-002).**
  Почему: `charmbracelet/log` не умеет ротацию нативно; реализация в коде потребовала бы новой
  внешней зависимости (`lumberjack`), что нарушает офлайн-вендоринг (baseline §6, STACK). Требование
  ротации baseline §4 НЕ снимается. Компенсирующий контроль: вывод аудита в stderr; ротация —
  journald (systemd) или logrotate (файловый вывод), как предписывает STACK. **Эскалация:** для
  файлового вывода без journald — devops/distribution ОБЯЗАНЫ поставить `logrotate`-конфиг; tech-writer
  документирует ротацию. Если дистрибуция не обеспечивает ротацию — эскалировать пользователю до
  релиза (риск переполнения диска аудит-логом). (baseline §4 «с ротацией»; baseline §6)

- [ ] **ОР-3. Полная ресурсная изоляция исполнения (cgroups/rlimits/seccomp/namespaces) НЕ
  реализуется в v1 — форк-бомба/исчерпание ресурсов смягчены частично (R-9).** Почему: spec Out of
  Scope — sandboxing отдельных команд не в v1; изоляция возлагается на развёртывание демона в
  контейнере (baseline §6) и не-root пользователя. Компенсирующий контроль: process-group kill
  убивает всё дерево при таймауте/отмене (ADR-001); whitelist env + отсутствие shell сужают вектор;
  контейнер ограничивает ресурсы хоста; allowlist (если включён) сужает набор исполняемых бинарей.
  **Эскалация:** при развёртывании демона ВНЕ контейнера или без ресурсных лимитов рантайма —
  эскалировать пользователю до релиза необходимость cgroups/rlimits (отдельная задача sandboxing).
  (baseline §3/§6; spec Out of Scope)

- [ ] **ОР-4. Секрет, переданный клиентом в аргументах команды, попадает в аудит-лог (R-13/П-3).**
  Почему: маскирование произвольного argv ненадёжно и вредит полноте аудита (см. П-3); args
  логируются дословно. Компенсирующий контроль: аудит-лог ограниченного доступа; обязательное
  предупреждение в доке (НЕ передавать секреты в argv); тело API-ключа raxd под защитой R-12.
  **Эскалация:** если конкретное развёртывание требует обработки команд с секретами в argv (напр.
  обязательный `mysql -p`), — согласовать с пользователем доп. меру (ограничение доступа к
  аудит-логу, отдельный канал, или отложенная задача «redaction-политики argv»). (baseline §4)

- [ ] **ОР-5. Allowlist по умолчанию ВЫКЛЮЧЕН → разрешена любая команда (по контракту AC7).**
  Почему: spec AC7 / baseline §3 фиксируют allowlist как ОПЦИОНАЛЬНЫЙ, по умолчанию выключенный
  (`exec.allowlist=[]`); включённый — строгое точное сопоставление по присланному `command` ДО
  `LookPath`. При выключенном allowlist аутентифицированный клиент может запустить ЛЮБОЙ бинарь
  (это и есть назначение инструмента — RCE уровня SSH). Компенсирующий контроль: вся остальная
  защита §3 (без shell, таймаут, лимиты, env-whitelist, не-root, аудит) действует независимо от
  allowlist; включённый allowlist — дополнительный жёсткий рычаг сужения. Замечание по достаточности
  «точного равенства ДО LookPath»: при выключенном allowlist обхода нет (всё разрешено); при
  включённом — `ls` и `/bin/ls` суть РАЗНЫЕ записи (клиент может прислать другой алиас/абсолютный
  путь к тому же бинарю); это осознанная цена буквального соответствия AC7 «точное совпадение, не
  regex, не префикс» и предсказуемости для оператора. TOCTOU между проверкой allowlist и `LookPath`/
  exec не даёт обхода СПИСКА (проверка по присланной строке, а не по резолву), но НЕ гарантирует, что
  по этому имени на диске не подменён бинарь (защита — целостность хоста/PATH, вне раннера). **Эскалация:**
  для инсталляций с требованием минимальных привилегий — рекомендовать оператору ВКЛЮЧИТЬ allowlist и
  согласовать его состав; если требуется матч по абсолютному резолву (устойчивость к алиасам) —
  отдельная задача (research Q7 вариант B). (baseline §3; spec AC7; research Q7)

- [ ] **ОР-6. Поле `env` во входе инструмента НЕ принимается в v1 (отложено).** Почему: spec Out of
  Scope — приём произвольного окружения от клиента = риск инъекции переменных (`LD_PRELOAD` и др.,
  R-7) и непредсказуемости. Компенсирующий контроль: окружение задаётся только серверным
  env-whitelist (AC10). **Эскалация:** при доказанной потребности передавать переменные — отдельная
  задача со СТРОГО контролируемым allowlist-набором имён переменных (не значений-инъекций). (spec
  Out of Scope; baseline §3)
