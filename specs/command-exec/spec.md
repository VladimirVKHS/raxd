# Spec: command-exec — безопасное выполнение команд на хосте по запросу ИИ-агента

Автор продукта: Vladimir Kovalev, OEM TECH

## Goal
Дать ИИ-агенту (MCP-клиенту) возможность выполнить НЕинтерактивную команду на хосте `raxd` и
получить её результат (stdout/stderr/код возврата) — поверх УЖЕ ГОТОВОГО аутентифицированного
HTTP/TLS-транспорта и MCP-сервера (задачи `tls-transport`/`mcp-server`). Это самая опасная часть
продукта (как удалённый вызов через SSH), поэтому фича целиком подчинена `SECURITY-BASELINE.ru.md §3`:
запуск без shell-интерполяции, обязательный таймаут, опциональный allowlist, ограниченные рабочая
директория и окружение, лимиты на размер вывода и аудит каждого вызова без секретов. Поверхность —
новый MCP-инструмент `execute_command`, регистрируемый в существующей точке расширения; транспорт,
аутентификация, rate-limit и аудит НЕ переписываются, а потребляются.

## User Stories
- Как ИИ-агент (MCP-клиент), я хочу вызвать `execute_command` с именем бинаря и аргументами и
  получить stdout/stderr/код возврата, чтобы выполнять задачи на удалённом хосте.
- Как специалист по безопасности, я хочу, чтобы любое выполнение команды проходило аутентификацию,
  ограничивалось таймаутом, лимитами вывода и (опционально) allowlist, и фиксировалось в аудите без
  секретов, чтобы расследовать инциденты и предотвращать эскалацию привилегий.
- Как администратор `raxd`, я хочу управлять поведением исполнения (таймаут по умолчанию, лимиты
  вывода, allowlist, рабочая директория) через конфиг с безопасными дефолтами, чтобы не открывать
  лишних возможностей по умолчанию.

## Acceptance Criteria
- [ ] AC1 (поверхность): выполнение команд доступно ТОЛЬКО как MCP-инструмент `execute_command`,
      зарегистрированный в существующей точке расширения MCP-сервера за той же наследуемой цепочкой
      auth → Origin/Host → rate-limit → аудит. Отдельного не-/mcp сетевого эндпоинта и CLI-подкоманды
      `raxd exec` в этой задаче НЕТ. Тест: после `serve` инструмент присутствует в `tools/list`
      рядом с `ping`/`server_info`, а его регистрация не меняет транспорт и маршрут `/mcp`.
- [ ] AC2 (baseline §3, без shell): команда запускается формой «бинарь + список аргументов» БЕЗ
      shell-интерполяции; ни при каких входных данных не выполняется `sh -c <строка>`. Тест: вход с
      shell-метасимволами (`;`, `|`, `$()`, `&&`, `>`, обратные кавычки) трактуется как литеральные
      аргументы бинаря, а не как команды оболочки (нет побочного эффекта, который дал бы только shell).
- [ ] AC3 (формат входа): инструмент принимает `command` (строка, обязателен, имя/путь бинаря),
      `args` (массив строк, опц.), `timeout_ms` (целое, опц.), `cwd` (строка, опц.). Поле `env`
      НЕ принимается в v1 (см. Out of Scope). Любые неизвестные поля отвергаются (строгая схема,
      `additionalProperties:false`). Тест: запрос с лишним полем → ошибка валидации ввода
      (`isError:true`), команда не запускается.
- [ ] AC4 (формат выхода): успешный вызов возвращает структурированный результат с полями `stdout`,
      `stderr`, `exit_code` (целое), `duration_ms` (целое), `timed_out` (булево), `stdout_truncated`
      и `stderr_truncated` (булево). Тест: запуск команды с известным выводом и кодом возврата даёт
      ожидаемые значения этих полей; форма результата соответствует объявленной выходной схеме.
- [ ] AC5 (baseline §3, таймаут): каждая команда ограничена таймаутом через контекст. Дефолт берётся
      из конфига (ориентир 30s, конфигурируемо). Если запрошенный `timeout_ms` превышает жёсткий
      максимум из конфига — вызов отклоняется: возвращается `isError:true` с причиной «превышение
      лимита таймаута», команда НЕ запускается. Тест №1: команда, превышающая (допустимый) таймаут,
      прерывается, процесс завершается (kill), результат содержит `timed_out:true` и непустой/частичный
      вывод до прерывания. Тест №2: запрос с `timeout_ms` больше жёсткого максимума → `isError:true`,
      команда не запускалась.
- [ ] AC6 (edge, обрыв/отмена): при отмене контекста запроса (обрыв соединения клиента) запущенный
      процесс гарантированно завершается (не остаётся «осиротевших» процессов), вызов возвращает
      признак прерывания. Тест: отмена контекста во время выполнения долгой команды приводит к её
      завершению (проверяется отсутствием живого дочернего процесса после возврата).
- [ ] AC7 (baseline §3, allowlist): поддерживается опциональный allowlist команд со СТРОГИМ
      сопоставлением (точное совпадение, НЕ regex и НЕ префикс). По умолчанию allowlist ВЫКЛЮЧЕН.
      Зафиксированное поведение: allowlist выключен → разрешена любая команда; allowlist включён →
      разрешены ТОЛЬКО команды из списка, всё остальное отклоняется. Тест: при включённом allowlist
      команда из списка выполняется, команда вне списка отклоняется (`isError:true`, причина «не
      разрешена») и НЕ запускается; при выключенном — обе выполняются.
- [ ] AC8 (edge, несуществующий бинарь): запрос на запуск несуществующего/недоступного бинаря
      возвращает ошибку выполнения (`isError:true`) с нейтральным сообщением, без паники и без 5xx;
      сервер остаётся работоспособным (следующий валидный вызов отрабатывает штатно). Тест: вызов
      заведомо отсутствующего бинаря → `isError:true`, сервер жив.
- [ ] AC9 (baseline §3, привилегии): инструмент НЕ повышает привилегии — не использует setuid/sudo,
      не пытается стать root, наследует UID/GID процесса демона как есть. Дополнительно: если
      эффективный UID демона == 0 (демон работает от root), при КАЖДОМ вызове `execute_command`
      пишется аудит-запись уровня WARN об исполнении команд от root (детекция обязательна). Конкретная
      политика (только предупреждение vs полный отказ исполнять команды от root) — явный пункт для
      architect/security; в этом контракте обязательны факт детекции и WARN. Тест №1: дочерний процесс
      выполняется под тем же пользователем, что и демон (эффективный UID не меняется). Тест №2: при
      эффективном UID демона == 0 вызов `execute_command` порождает WARN-запись в аудите.
- [ ] AC10 (baseline §3, рабочая директория и окружение): рабочая директория предсказуема и
      ограничена — при отсутствии `cwd` используется заданный конфигом безопасный дефолт (не
      произвольный текущий каталог процесса по случайности); при заданном `cwd` он валидируется
      (существует, является каталогом) и недопустимое значение отклоняется до запуска. Окружение
      дочернего процесса минимально и предсказуемо: ПРИНЦИП — явный whitelist разрешённых переменных,
      БЕЗ слепого наследования всего окружения демона. Конкретный состав разрешённых переменных
      определяет architect/security; AC фиксирует лишь принцип. Тест: команда видит ожидаемую рабочую
      директорию и ограниченное окружение (присутствуют только разрешённые переменные); невалидный
      `cwd` → `isError:true` без запуска.
- [ ] AC11 (baseline §3, лимиты вывода): размер захватываемых stdout и stderr ограничен
      конфигурируемым максимумом (защита от OOM «болтливой» команды). При превышении вывод
      обрезается до лимита, соответствующий флаг `*_truncated` выставляется в `true`, процесс не
      приводит к неограниченному росту памяти. Тест: команда, печатающая объём больше лимита, даёт
      обрезанный вывод и `*_truncated:true`, потребление памяти ограничено.
- [ ] AC12 (baseline §1/§2, аутентификация): вызов `execute_command` проходит ту же Bearer-аутентификацию
      транспорта ДО исполнения; запрос без `Authorization: Bearer`, с неизвестным/отозванным ключом не
      доходит до инструмента (401), повреждение keystore → 403. Новых каналов аутентификации не вводится.
      Тест: неаутентифицированный вызов `execute_command` не запускает никакую команду.
- [ ] AC13 (baseline §4, аудит каждого вызова): каждый вызов `execute_command` пишет аудит-запись
      через существующий аудит-механизм. Запись содержит как минимум поля: timestamp (UTC),
      fingerprint ключа (НЕ сам ключ), имя инструмента, выполненную команду + аргументы, exit code,
      длительность (duration), удалённый адрес (remote), результат (success/fail/deny). Тест:
      аудит-запись успешного вызова содержит timestamp (UTC), fingerprint, команду+аргументы, exit
      code, длительность (duration), удалённый адрес (remote) и результат; запись отклонённого вызова
      содержит fingerprint, команду+аргументы, удалённый адрес и результат (deny).
- [ ] AC14 (baseline §4, формат и ротация аудита): аудит-запись `execute_command` имеет
      машиночитаемый структурированный формат, совместимый с существующей аудит-инфраструктурой
      (единый канал и схема записей с остальным транспортом). Требование ротации логов наследуется от
      baseline §4 и НЕ снимается этой задачей. Тест: аудит-запись машиночитаема (парсится как
      структурированная запись существующего формата) и не ломает формат не-`execute_command` записей.
- [ ] AC15 (baseline §4/§1, без секретов): в аудит-логе, в результате `execute_command` и в сообщениях
      об ошибках НЕТ тела API-ключа, его хэша, соли, raw `Authorization` и приватного TLS-ключа. Тест:
      предъявленный ключ не встречается подстрокой в захваченном логе и в теле MCP-ответа.
- [ ] AC16 (rate-limit, наследуется): вызовы `execute_command` подчиняются существующему per-key и
      per-IP rate-limit транспорта (429 при превышении ДО исполнения инструмента). Тест: превышение
      лимита отклоняется до запуска команды.
- [ ] AC17 (edge, протокол): некорректный JSON-RPC и вызов с неверными параметрами возвращают
      корректную JSON-RPC-ошибку либо `isError:true` (ошибка валидации ввода — НЕ protocol error по
      контракту MCP-сервера), без паники и без 501; сервер остаётся работоспособным.
- [ ] AC18 (baseline §6, тесты в Docker): все тесты выше зелёные в Docker; сборка/тесты офлайн из
      `vendor/` (`-mod=vendor`), без `go mod download`. Запуск демона и прогон команд — только в
      контейнере (baseline §6), не на хосте.

## Зависимости от готового кода
- `mcp-server`: MCP-сервер на официальном Go SDK с точкой расширения регистрации инструментов
  (`internal/mcp/server.go`, `NewHandler` + `mcp.AddTool`) и обёрткой аудита (`internal/mcp/audit.go`,
  `withAudit`). `execute_command` регистрируется здесь, наравне с `ping`/`server_info`.
- `tls-transport`: цепочка middleware (bodyLimit → recover → Host/Origin → auth → rate-limit →
  authSuccessAudit → mux), монтаж `/mcp`, контекст с fingerprint/remote address
  (`server.FingerprintFromContext`, `server.RemoteAddrFromContext`). Не переписывается.
- `key-management`: `keystore.Verify`/`keystore.Fingerprint` — потребляются транспортом; этот слой
  тело ключа не получает.
- Аудит-инфраструктура: `internal/server/audit.go` (`AuditRecord`, `AuditFn`, `writeAudit`). Запись о
  команде использует существующий аудит-канал; перечень полей записи задаёт AC13. Конкретный способ
  представления полей команды/exit code/длительности (форма данных в аудите) определяет
  architect/security — без изменения формата уже существующих не-`execute_command` записей.
- `config`: `internal/config/config.go` (viper, безопасные дефолты). Новые параметры (дефолтный и
  максимальный таймаут, лимиты вывода, allowlist, рабочая директория, набор окружения) добавляются
  как конфиг-поля с безопасными дефолтами.
- `internal/version` — версия демона (для контекста, не для исполнения).

## Out of Scope
- Интерактивные команды, PTY/TTY-сессии, stdin-стриминг и любой ввод в процесс после старта —
  отложено (отдельная задача, требует stateful-канала).
- Передача файлов / `upload_file` — задача `file-upload`.
- Отдельный не-/mcp сетевой эндпоинт исполнения и CLI-подкоманда `raxd exec` для локального теста —
  не вводим, чтобы не плодить дублирующих поверхностей с собственной auth/аудитом; при доказанной
  потребности — отдельная задача.
- Приём произвольного окружения от клиента (`env` во входе инструмента) — не в v1 (риск инъекции
  переменных и непредсказуемости); возможен позже строго контролируемым allowlist-набором.
- Стриминг вывода в реальном времени (server→client SSE) — v1 stateless, вывод отдаётся целиком
  после завершения (с учётом лимитов); стриминг — отдельная задача.
- Изменение модели системного сервиса, выбор системного пользователя, capabilities, setuid —
  свойство развёртывания (`service-install`/`distribution`); здесь только требование «не повышать
  привилегии» и детекция root-демона (AC9).
- Sandboxing/контейнеризация исполнения отдельных команд, cgroups, seccomp, namespaces — не в v1;
  изоляция обеспечивается развёртыванием демона в контейнере (baseline §6) и не-root пользователем.
- Управление ключами, TLS, rate-limit, протокол MCP, инструменты `ping`/`server_info` — готовы в
  смежных задачах, здесь не дорабатываются.

## Open Questions
- None. Развилки решены обоснованными дефолтами и зафиксированы в AC: поверхность — только
  MCP-инструмент `execute_command` (AC1); вход `command/args/timeout_ms/cwd`, без `env` (AC3); выход
  `stdout/stderr/exit_code/duration_ms/timed_out/*_truncated` (AC4); таймаут → kill + `timed_out`, а
  запрос свыше жёсткого максимума → `isError` без запуска (AC5); обрыв → завершение процесса (AC6);
  allowlist выкл. по умолчанию, строгое сравнение, поведение вкл./выкл. зафиксировано (AC7);
  несуществующий бинарь → `isError` (AC8). Открытые ПОЛИТИЧЕСКИЕ решения, оставленные явно для
  architect/security (не блокируют контракт, факт-требования уже в AC): конкретная политика при
  root-демоне (только WARN vs отказ; AC9), точный состав whitelist переменных окружения (AC10),
  числовое значение жёсткого верхнего предела таймаута (AC5).

## Примечание о размере задачи
Задача крупная (18 AC, целиком завязана на `SECURITY-BASELINE.ru.md §3`) и НЕДЕЛИМА по смыслу:
безопасное выполнение команды — единый контракт, где запуск без shell, таймаут, allowlist, лимиты
вывода, ограничение привилегий/окружения и аудит без секретов неразрывны (любое из них в отрыве от
остальных создаёт небезопасную или неполную фичу). Разбивка на под-задачи нецелесообразна —
получились бы заведомо незавершённые куски опасного функционала. AC многочисленны не из-за раздувания
scope, а из-за плотности обязательных требований безопасности на единственный инструмент.
