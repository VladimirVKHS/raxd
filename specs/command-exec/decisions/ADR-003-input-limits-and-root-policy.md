# ADR-003: Входные лимиты, числовые пороги и политика исполнения от root

## Контекст
spec.md явно оставляет architect/security три числовых/политических развилки (Open Questions, AC5/AC9/
AC10/Q10): (1) жёсткий верхний предел таймаута; (2) политику при root-демоне (только WARN vs полный
отказ исполнять команды); (3) состав env-whitelist и прикладные лимиты входа (`max_args`/`max_arg_len`)
против аргументной DoS (research Q10: `exec.Command` не имеет встроенного лимита argv; `E2BIG` от ядра
— поздняя и грубая защита уже на этапе exec). research подтвердил факт-механизмы, но числа и политику
оставил следующему слою. Этот ADR фиксирует конкретные значения и политику — как контракт для
developer/security; они дополняют research, не противоречат ему.

## Решение
1. **Жёсткий максимум таймаута (AC5):** `max_timeout_ms = 300000` (5 минут). Запрос `timeout_ms` сверх
   → `isError:true`, команда не запускается. Дефолтный таймаут — `default_timeout_ms = 30000` (30s,
   ориентир baseline §3). 5 минут покрывают типичные неинтерактивные задачи без превращения инструмента
   в канал долгоживущих фоновых процессов.
2. **Политика root (AC9): только WARN, НЕ отказ.** При `os.Geteuid()==0` на КАЖДЫЙ вызов
   `execute_command` пишется WARN-аудит (детекция обязательна по AC9). Полный отказ НЕ вводим: он сломал
   бы легитимные сценарии (контейнерный демон часто стартует от root до сброса привилегий, dev-окружение).
   Основная защита от исполнения-от-root — правильная раскладка сервиса (выделенный не-root пользователь,
   baseline §3, задача service-install) и контейнерная изоляция (baseline §6), а не отказ в раннере.
3. **Прикладные лимиты входа (Q10/AC9-DoS), проверяются ДО запуска → isError:**
   `max_args = 256`, `max_arg_len = 131072` (128 KiB на один аргумент), `max_output_bytes = 1048576`
   (1 MiB на каждый из stdout/stderr, AC11). Эти потолки отсекают аргументную/выводную DoS на стороне
   демона раньше, чем сработает ядро (`E2BIG`) или вырастет память.
4. **env-whitelist (AC10):** дефолт `["PATH","HOME","LANG","TERM"]`, конфигурируемо. `PATH` обязателен
   (LookPath). `HOME/LANG/TERM` — минимум для корректной работы типовых утилит. Запрещённые к
   наследованию: `LD_PRELOAD`/`LD_LIBRARY_PATH`/`DYLD_INSERT_LIBRARIES`/`IFS` (research Q4, CWE-426/427).

## Альтернативы
- **Без жёсткого максимума таймаута / очень большой (часы).** Отвергнут: позволяет занять слот
  исполнения надолго, приближается к фоновому процессу — против духа неинтерактивного stateless v1.
- **Полный отказ исполнять при euid==0.** Отвергнут как дефолт: ломает легитимные старт-от-root и
  dev-сценарии; AC9 требует обязательным лишь детекцию+WARN. Может быть добавлен опционально позже
  (конфиг-флаг `exec.deny_root`), если security в threat-model сочтёт нужным.
- **Без прикладных лимитов входа (полагаться на `E2BIG`).** Отвергнут: ядро отвергает только на
  execve, аргументы уже в памяти демона; не прикладная защита от OOM/DoS (research Q10).
- **Пустой env (только PATH).** Отвергнут как дефолт: многие утилиты ломаются без HOME/LANG/TERM;
  баланс безопасности и пригодности — минимальный безопасный набор, конфигурируемый.

## Последствия
- Плюсы: предсказуемые безопасные дефолты, закрывающие DoS-векторы (argv/вывод/таймаут) без shell и
  без новых зависимостей; политика root не ломает легитимные развёртывания при обязательной детекции.
- Минусы/цена: WARN-политика root оставляет риск исполнения от root (смягчён раскладкой сервиса +
  контейнером); числовые пороги — компромисс, могут потребовать тюнинга под конкретные нагрузки
  (поэтому конфигурируемы). Все значения — дефолты конфига, переопределяемы в config.yaml.
- Влияние на стек: согласуется со STACK.ru.md (stdlib, без новых зависимостей) и baseline §3 (таймаут,
  ограниченное окружение, не-root).

## Зависимость от security
Окончательное подтверждение достаточности порогов и политики root — за **security** (threat-model.md:
риск + смягчение). architect фиксирует дефолты; security вправе ужесточить (например, включить
`deny_root` или снизить пороги) — это не ломает контракт plan.md, лишь меняет значения/флаги конфига.

## Статус (proposed|accepted)
accepted — ратифицирован гейтами security-guardian + architect-guardian (command-exec), 2026-05-22.
Финальное принятие отклонений П-2 (root: WARN + опциональный `deny_root`) и П-3 (лимиты args/arg_len
дословно) зафиксировано security в `specs/command-exec/threat-model.md` (раздел «Принятые отклонения»).
