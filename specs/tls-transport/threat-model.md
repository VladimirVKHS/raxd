# Threat Model: TLS Transport — защищённый аутентифицированный сетевой транспорт raxd

Автор: security (raxd). Дата: 2026-05-21. Язык: русский.
Автор продукта: Vladimir Kovalev, OEM TECH.
Вход: `specs/tls-transport/{spec.md,plan.md,research.md}`, `.claude/reference/SECURITY-BASELINE.ru.md`
(§1 аутентификация, §2 транспорт, §4 аудит/устойчивость, §6 Docker), `.claude/reference/MCP-INTEGRATION.ru.md`,
контракт `internal/keystore` (`Verify`/`Fingerprint`/`ErrCorrupt`).

> Это первый сетевой слой raxd. raxd по назначению исполняет произвольные команды по сети (как SSH) —
> поэтому транспорт обязан аутентифицировать КАЖДОЕ подключение ДО любой обработки. В этой задаче
> реальный exec/MCP/upload ещё НЕ реализуются (диспетчер — заглушка 501), но именно здесь строится
> периметр, на который сядут будущие `command-exec`/`mcp-server`/`file-upload`. Каждый риск ниже имеет
> смягчение; невыполнимые пункты baseline зафиксированы в «Остаточные риски / эскалации».

## Активы

- **API-ключи (тело `rax_live_…`)** — главный секрет: даёт аутентифицированный канал к демону. В
  транспорте ключ проходит через заголовок `Authorization: Bearer` КАЖДОГО запроса. Утечка = полный
  доступ. Хранится только хэш+соль в keystore (вне scope), но в рантайме транспорта ключ проходит в
  открытом виде через память/заголовки.
- **Приватный TLS-ключ** (`TLSDir/key.pem`, ECDSA P-256) — позволяет выдать себя за сервер и расшифровать
  перехваченный трафик. Ценно, права файла обязаны быть `0600`.
- **TLS-сертификат** (`TLSDir/cert.pem`) — self-signed; не секрет, но его целостность важна (битый серт →
  отказ запуска).
- **Аудит-лог** — журнал аутентификаций/отказов/rate-limit; ценен для расследования и не должен содержать
  секретов. Подделка/обход = repudiation.
- **Сервер-процесс и хост** — конечная цель: будущий exec даст исполнение команд на хосте; пока заглушка,
  но DoS на процесс уже актуален.
- **Буфер использования ключей (`Store.usageBuf`)** — сбрасывается `FlushUsage` при shutdown; потеря =
  пропуск метаданных аудита (low).

## Поверхность атаки

- **Сетевой вход: TCP → TLS 1.3 → HTTP/1.1+ (`net/http`) на `127.0.0.1:Port` (дефолт 7822)** — основной
  вектор. Внутри: TLS-handshake, HTTP-заголовки (`Authorization`, `Origin`, `Host`), тело запроса, путь
  (`/healthz` + catch-all).
- **CLI `raxd serve`** (foreground-запуск в Docker, AC11) — точка старта; флаги/конфиг (`BindAddr`, `Port`,
  лимиты, таймауты).
- **Файлы состояния**: `TLSDir/{cert,key}.pem` (генерация/переиспользование/повреждение), `keys.db`
  (читается `Verify`; повреждение в рантайме = «нет валидных ключей»), `config.yaml` (адрес/лимиты).
- **Кто атакует:**
  - **Внешний клиент без ключа** — пытается достучаться до health/диспетчера, обойти аутентификацию,
    устроить DoS, перебрать ключи.
  - **Клиент с валидным ключом** — может злоупотребить частотой (DoS), но это в рамках авторизации.
  - **MITM в сети** — перехват/подмена трафика (TLS закрывает конфиденциальность; доверие к self-signed —
    остаточный риск, см. R1).
  - **Локальный пользователь на хосте** — чтение TLS-ключа/`keys.db`, подсматривание ключа через `ps`/`/proc`
    (argv/env), чтение логов.
  - **Браузерный контекст / вредоносная веб-страница** — DNS-rebinding на `127.0.0.1` (R2).

## Угрозы (STRIDE-ориентир)

- **S (Spoofing) — подмена клиента / обход аутентификации.** Запрос без `Bearer`, с неизвестным/отозванным
  ключом, попытка дойти до диспетчера в обход auth-middleware. Смягчение: `keystore.Verify` в middleware
  ДО маршрутизации (R3); фиксированная цепочка `audit → recover → Host/Origin → auth → rate-limit → mux`.
- **S — подмена сервера (MITM).** Self-signed серт без PKI: клиент должен доверять серту при первом
  подключении (TOFU). Смягчение: TLS 1.3 + bind `127.0.0.1`; остаточный риск R1 (mTLS отложен).
- **T (Tampering) — подмена трафика/конфига/серта.** Изменение запроса в сети; подсунутый битый/чужой серт
  в `TLSDir`. Смягчение: TLS 1.3 целостность канала (R1); строгая обработка битого серта без перезаписи
  (R10); права файлов `0600`/`0644`.
- **R (Repudiation) — отсутствие/обход аудита.** Отказ или превышение лимита не зафиксированы → нельзя
  расследовать. Смягчение: аудит на КАЖДОЕ соединение (success/fail/rate-limited) с timestamp/Fingerprint/
  remoteAddr (R7).
- **I (Information disclosure) — утечка ключа/секретов.** Логирование raw `Authorization`/тела ключа,
  утечка приватного TLS-ключа, раскрытие причины отказа («нет такого id»), timing-leak. Смягчение: аудит
  только по `Fingerprint` (R6); запрет логировать секреты (R6); единый необъясняющий отказ (R4);
  constant-time `Verify` (R5); ключ только через заголовок, не argv/env (R12).
- **D (Denial of service) — исчерпание ресурсов.** Коннект-флуд, медленные клиенты (Slowloris), брутфорс
  ключей, неограниченный рост map лимитеров, большое тело запроса. Смягчение: rate-limit per-key/per-IP
  (R8); таймауты Read/Write/Idle/Header (R9); лимит тела (R9); TTL-GC лимитеров под мьютексом (R8/R11);
  graceful shutdown (R13).
- **E (Elevation of privilege) — повышение привилегий.** Запуск от root, привязка к публичному интерфейсу.
  В этой задаче exec ещё заглушка; периметр: bind `127.0.0.1` по умолчанию (R2/R14), запуск не от root —
  но системная регистрация сервиса вне scope (см. ОР-2).

## Риски (риск → вероятность/влияние → смягчение)

- [ ] **R1. MITM/перехват трафика; доверие к self-signed серту (TOFU).** Канал шифруется, но self-signed
  серт не проверяется через PKI: клиент доверяет серту «на первом подключении». Активный MITM при первом
  коннекте (до закрепления отпечатка клиентом) может подменить серт → вероятность **низкая** (bind
  `127.0.0.1`, локальный/доверенный сегмент) / влияние **высокое** (раскрытие ключа и трафика). Смягчение:
  `MinVersion: tls.VersionTLS13` (целостность+конфиденциальность канала, отсечение downgrade на 1.2/1.1);
  bind на `127.0.0.1` по умолчанию резко сужает сетевую достижимость; самозакрепление отпечатка серта на
  клиенте (TOFU). Остаточный риск (mTLS/PKI) — см. ОР-1, эскалирован. (baseline §2)

- [ ] **R2. DNS-rebinding на локальный демон.** Вредоносная веб-страница в браузере жертвы резолвит свой
  домен в `127.0.0.1` и шлёт запросы к локальному `raxd`. Вероятность **низкая** (raxd-агенты не браузерные;
  нужен валидный ключ) / влияние **высокое** (если бы не было auth). Смягчение (defense-in-depth, ADR-002):
  (1) bind `127.0.0.1`; (2) **аутентификация по ключу — основной гейт**: без валидного `Bearer` rebinding
  бесполезен; (3) `Host`-allowlist `localhost`/`127.0.0.1`/`::1` (Host вне списка → 403); (4) `Origin`
  present И вне allowlist → 403; `Origin` ОТСУТСТВУЕТ → НЕ отклонять (не-браузерные клиенты curl/SDK Origin
  не шлют — иначе сломаются легитимные клиенты). Полная браузер-ориентированная Origin-политика обязательна
  с включением `mcp-server` (MCP MUST). (baseline §2)

- [ ] **R3. Обход аутентификации (запрос доходит до обработчика без валидного ключа).** Если auth-проверка
  стоит ПОСЛЕ маршрутизации или health/диспетчер доступен в обход middleware. Вероятность **низкая** /
  влияние **критическое** (открытая точка входа). Смягчение: `keystore.Verify` в `authMiddleware` ДО любой
  обработки и ДО `http.ServeMux`; ВСЕ маршруты (включая `/healthz` и catch-all) — за единой цепочкой
  middleware; тест: запрос без ключа и с неизвестным/отозванным ключом НЕ достигает health-обработчика.
  (baseline §1)

- [ ] **R4. Раскрытие причины отказа аутентификации (user enumeration / подсказка атакующему).** Ответы
  вида «нет такого id», «ключ отозван», «неверный формат» помогают перебору. Вероятность **средняя** /
  влияние **среднее**. Смягчение: на нет/невалидный `Bearer` и `Verify→(_,false,nil)` — единый ответ 401
  без деталей о причине; различие «нет ключа / неизвестен / отозван» в ОТВЕТ клиенту НЕ просачивается
  (фиксируется только в аудит-логе на стороне сервера, R7). (baseline §1)

- [ ] **R5. Timing-атака на ключ.** Сравнение секретов небезопасным образом раскрывает по времени, какой
  ключ ближе. Вероятность **низкая** (constant-time уже внутри keystore) / влияние **высокое**. Смягчение:
  сравнение ТОЛЬКО через `keystore.Verify` (внутри `subtle.ConstantTimeCompare`, перебор всех записей без
  раннего выхода — подтверждено по коду `internal/keystore/keystore.go`); транспорт НЕ реализует
  собственного сравнения ключей/хэшей. (baseline §1)

- [ ] **R6. Утечка ключа в логах/выводе/ошибках (AC9).** Логирование raw-заголовка `Authorization`, тела
  ключа, его хэша/соли или приватного TLS-ключа в аудит/stdout/сообщениях ошибок. Вероятность **средняя**
  (легко допустить при отладке) / влияние **критическое**. Смягчение: аудит и любые сообщения оперируют
  ТОЛЬКО `keystore.Fingerprint` (12 hex sha256, необратим); raw `Authorization`/тело ключа НЕ логируются
  никогда; приватный TLS-ключ не пишется в лог. Проверяемо: предъявленный ключ как подстрока ОТСУТСТВУЕТ в
  захваченном выводе (тест grep). (baseline §4, §1)

- [ ] **R7. Отсутствие аудита отказов (repudiation, слепота к атаке).** Неудачные аутентификации и
  срабатывания rate-limit не записаны → всплеск перебора незаметен. Вероятность **средняя** / влияние
  **среднее**. Смягчение: структурированная (JSON) аудит-запись на КАЖДОЕ соединение: timestamp (UTC),
  `Fingerprint` (НЕ ключ), `RemoteAddr`, `Result` (success/fail/rate-limited), `Reason`; неуспешные
  аутентификации и rate-limit логируются обязательно. (baseline §4)

- [ ] **R8. Брутфорс/перебор ключей по сети.** Атакующий перебирает значения ключа на скорости сети.
  Вероятность **средняя** / влияние **высокое** (при ≥256 бит энтропии перебор непрактичен, но rate-limit
  обязателен по baseline и снижает «шум»/нагрузку). Смягчение: rate-limit **per-key И per-IP** (token
  bucket `golang.org/x/time/rate`), 429 при превышении + аудит-запись; constant-time `Verify` (R5);
  обязательная запись неудачных попыток (R7). (baseline §4, §1)

- [ ] **R9. DoS: коннект-флуд, медленные клиенты (Slowloris), большое тело.** Без таймаутов и лимита тела
  один клиент удерживает соединения/память. Вероятность **средняя** / влияние **среднее** (bind
  `127.0.0.1` сужает источник, но локальные/проброшенные клиенты остаются). Смягчение: таймауты
  `ReadTimeout`/`ReadHeaderTimeout`/`WriteTimeout`/`IdleTimeout` на `http.Server`; лимит размера тела
  запроса (`MaxBytesReader`/`MaxHeaderBytes`); обрыв на середине handshake обрабатывается без паники
  (recover). (baseline §4)

- [ ] **R10. Битый/чужой/пустой сертификат в `TLSDir` (порча состояния, отказ старта).** Пустой/повреждённый
  `cert.pem`/`key.pem` при старте. Вероятность **низкая** / влияние **среднее** (сервер не должен молча
  стартовать без TLS и не должен паниковать). Смягчение: `loadOrCreateCert` возвращает sentinel `ErrTLSCert`
  при пустом/битом файле, серт НЕ перезаписывается молча, процесс завершается с ненулевым кодом и понятной
  ошибкой; при отсутствии пары — генерация self-signed (AC2); при наличии валидной — переиспользование (AC3,
  без перегенерации). (baseline §2)

- [ ] **R11. Утечка памяти через рост map лимитеров (отложенный DoS).** per-key/per-IP лимитеры лениво
  создаются и без очистки растут безгранично (атакующий с меняющимся IP/перебором ключей раздувает map).
  Вероятность **средняя** / влияние **среднее**. Смягчение: хранилище лимитеров под `sync.Mutex`
  (потокобезопасность) + фоновая TTL-очистка простаивающих лимитеров (по времени последнего обращения),
  останавливается по контексту `Run`. (baseline §4)

- [ ] **R12. Передача ключа через argv/env (утечка через `ps`/`/proc`, SR-14 baseline).** Если ключ принять
  как флаг/позиционный аргумент или из переменной окружения. Вероятность **низкая** / влияние **высокое**
  (видно локальным пользователям и в логах процессов). Смягчение: ключ принимается ТОЛЬКО из HTTP-заголовка
  `Authorization: Bearer`; ни `serve`, ни транспорт НЕ читают тело ключа из argv/env. (baseline §1, §4)

- [ ] **R13. Потеря буферизованных метаданных аудита при остановке (устойчивость).** При резком завершении
  `usageBuf` (LastUsed) не сброшен → пропуск метаданных. Вероятность **средняя** / влияние **низкое**.
  Смягчение: graceful shutdown по SIGINT/SIGTERM/отмене контекста: `http.Server.Shutdown(ctxDeadline)`
  (перестать принимать новые, дождаться активных) → ЗАТЕМ `Store.FlushUsage()`; завершение в пределах
  дедлайна без зависания. Порядок строго `Shutdown → FlushUsage`. (baseline §4)

- [ ] **R14. Открытие демона на публичный интерфейс по ошибке конфигурации.** Если дефолт bind — не
  loopback или конфиг легко выставляет `0.0.0.0` без предупреждения. Вероятность **низкая** / влияние
  **высокое** (открытая сетевая точка входа к будущему exec). Смягчение: дефолт `BindAddr=127.0.0.1`;
  адрес/порт конфигурируемы осознанно (`config.Config.Port`, дефолт 7822); тест: при дефолтной конфигурации
  сокет недоступен с не-loopback адреса. (baseline §2)

- [ ] **R15. Маршрутизация на нереализованные операции даёт побочные эффекты.** Catch-all к будущему
  exec/MCP/upload в этой задаче не должен ничего исполнять. Вероятность **низкая** / влияние **среднее**
  (преждевременное исполнение). Смягчение: catch-all диспетчер возвращает «not implemented» (501) БЕЗ
  побочных эффектов; единственная обрабатываемая операция — health (`ping→pong`). (baseline §3 — наследуемая
  точка расширения)

## Остаточные риски / эскалации

- [ ] **ОР-1. mTLS / клиентские сертификаты (baseline §2 «mTLS — опционально») НЕ реализуется в v1.**
  Почему: решение дирижёра (spec Out of Scope / Q3, research Q3) — для модели «один продукт-клиент ↔ сервер»
  аутентификация по API-ключу поверх TLS 1.3 + bind `127.0.0.1` + rate-limit покрывает «аутентифицировать
  каждое подключение»; mTLS добавляет вес управления и дистрибуции клиентских сертификатов (это уровень
  service-install/distribution) без дополнительной защиты в текущей модели. Остаточный риск: при будущем
  выносе демона на не-loopback интерфейс отсутствие mTLS и доверие к self-signed серту (TOFU, R1) повышают
  риск MITM на первом подключении. Компенсирующий контроль: TLS 1.3, дефолт bind `127.0.0.1`, аутентификация
  по ключу как основной гейт, аудит; `tls.RequireAndVerifyClientCert` зафиксирован как будущая опция.
  **Эскалация:** перед релизом любой конфигурации с не-loopback bind — вернуть вопрос mTLS/доверия к серту
  пользователю. (baseline §2)

- [ ] **ОР-2. Запуск демона НЕ от root и системная регистрация сервиса (baseline §3 «демон не от root»;
  §4 «Restart=on-failure») НЕ закрываются этой задачей.** Почему: spec Out of Scope — регистрация
  systemd/launchd, права демона, capabilities, автоперезапуск — задача `service-install`/`distribution`;
  здесь только foreground-запуск в Docker (AC11). Остаточный риск: при неверной системной установке демон
  может оказаться запущен от root или без автоперезапуска. Компенсирующий контроль: в этой задаче запуск
  только в изолированном Docker (baseline §6); требование «не от root + capabilities для порта <1024 +
  Restart=on-failure» передаётся как ОБЯЗАТЕЛЬНОЕ в threat-model задачи `service-install`. **Эскалация:**
  зафиксировать в контракте service-install до релиза дистрибутива. (baseline §3, §4)

- [ ] **ОР-3. Полная браузер-ориентированная Origin-политика (MCP MUST: «validate Origin on all incoming
  connections») реализуется НЕ полностью в v1.** Почему: ADR-002 — в tls-transport «лёгкий» middleware
  (403 при present&invalid Host/Origin; пропуск при отсутствии Origin), т.к. raxd-агенты не браузерные и
  Origin обычно не шлют; жёсткое требование «Origin обязателен» сломало бы легитимных не-браузерных клиентов.
  Остаточный риск: при будущем браузерном MCP-клиенте отсутствие строгой Origin-проверки повышает риск
  DNS-rebinding. Компенсирующий контроль: bind `127.0.0.1` + аутентификация по ключу + Host-allowlist
  закрывают rebinding для текущей модели; middleware-каркас Origin заложен как точка расширения. **Эскалация:**
  полная Origin-политика обязательна при включении `mcp-server` — передано в его threat-model. (baseline §2)

- [ ] **ОР-4. Ротация и файловый бэкенд аудит-лога (baseline §4 «структурно (JSON), с ротацией») в этой
  задаче НЕ реализуются.** Почему: spec Out of Scope — ротация и выбор бэкенда (journald/syslog/файл) —
  уровень сервиса/дистрибуции; здесь обязательны только структура и состав полей записи. Остаточный риск:
  без ротации аудит-лог может неограниченно расти на проде. Компенсирующий контроль: структура записи (JSON,
  поля §4) задаётся здесь и совместима с внешней ротацией (logrotate/journald). **Эскалация:** ротация —
  обязательное требование к `service-install`/`distribution`. (baseline §4)
