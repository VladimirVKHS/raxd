# Spec: TLS Transport — защищённый аутентифицированный сетевой транспорт raxd

Автор: Vladimir Kovalev, OEM TECH

## Goal
Дать `raxd` сетевой фундамент: TCP-listener, обёрнутый в TLS 1.3, который аутентифицирует КАЖДОЕ
подключение по API-ключу ДО любой обработки запроса, ограничивает частоту (per-key/per-IP) и ведёт
аудит. После успешной аутентификации сервер обслуживает только минимальный health-обработчик
(ping→pong) как точку проверки живости; реальное выполнение команд, MCP и передача файлов в этой
задаче НЕ реализуются (диспетчер — заглушка). Задача остаётся цельной и включает оживление честной
заглушки `raxd serve` (foreground-запуск сервера в Docker): отдельный serve-task искусственен, т.к.
foreground-запуск не имеет смысла в отрыве от транспорта. Опирается на `SECURITY-BASELINE.ru.md
§1/§2/§4/§6`. Это зависимость для будущих `command-exec`, `mcp-server`, `file-upload`.

## User Stories
- Как ИИ-агент (сетевой клиент), я хочу установить TLS-соединение и предъявить API-ключ, чтобы
  получить аутентифицированный канал к `raxd` и проверить его живость (ping→pong).
- Как администратор сервера, я хочу запустить `raxd serve` и видеть, что демон слушает только
  локальный интерфейс с TLS и отвергает любое подключение без валидного ключа, чтобы не открыть
  незащищённую точку входа.
- Как специалист по безопасности, я хочу, чтобы каждый отказ аутентификации и каждое превышение
  лимита фиксировались в аудит-логе (без секретов), чтобы расследовать аномалии.

## Acceptance Criteria
- [ ] AC1 (baseline §2): TCP-listener обёрнут в `crypto/tls` с `MinVersion: tls.VersionTLS13`;
      попытка handshake клиентом с максимумом TLS 1.2 отвергается (тест в Docker).
- [ ] AC2 (baseline §2): при первом запуске генерируется self-signed x509-сертификат в
      `PathSet.TLSDir` (`~/.local/state/raxd/tls`); приватный ключ — права `0600`, сертификат —
      `0644`. Тест проверяет факт создания и точные права.
- [ ] AC3 (baseline §2): если сертификат и ключ уже существуют в `TLSDir` — они переиспользуются,
      повторная генерация НЕ выполняется (тест: два запуска дают один и тот же серт, файлы не
      перезаписаны).
- [ ] AC4 (baseline §1, SR-14): каждое подключение аутентифицируется по API-ключу через
      существующий контракт `internal/keystore` (`Verify`) ДО любой обработки полезной нагрузки;
      ключ передаётся НЕ через argv и НЕ через переменные окружения (а через заголовок/handshake
      соединения). Тест: запрос без ключа и запрос с неизвестным/отозванным ключом не доходят до
      health-обработчика.
- [ ] AC5 (baseline §1): неуспешная аутентификация (отсутствует/невалиден/отозван ключ) приводит к
      явному отказу (эквивалент 401/403) и закрытию соединения без обработки запроса; отозванный
      ранее валидный ключ перестаёт проходить немедленно (опирается на `Revoke` из key-management).
- [ ] AC6 (baseline §4): rate limiting per-key И per-IP на базе `golang.org/x/time/rate`
      (token bucket); при превышении — отказ (эквивалент 429) и аудит-запись; лимиты и burst имеют
      разумные дефолты и конфигурируемы. Тест: всплеск сверх лимита получает отказ 429-эквивалент.
- [ ] AC7 (baseline §2): по умолчанию сервер биндится на `127.0.0.1`; интерфейс/адрес и порт
      конфигурируемы, порт берётся из `config.Config.Port` (дефолт 7822). Тест: при дефолтной
      конфигурации сокет недоступен с не-loopback адреса.
- [ ] AC8 (baseline §4): на каждое подключение пишется структурированная (JSON) аудит-запись:
      timestamp (UTC), fingerprint ключа через `keystore.Fingerprint` (НЕ сам ключ), удалённый
      адрес, результат аутентификации (success/fail) и причина отказа. Неуспешные аутентификации и
      срабатывания rate-limit логируются обязательно.
- [ ] AC9 (baseline §4, §1): в аудит-логе, выводе сервера и сообщениях об ошибках НЕТ ни тела ключа,
      ни его хэша, ни salt, ни приватного TLS-ключа (тест: предъявленный ключ не встречается
      подстрокой в захваченном логе).
- [ ] AC10: после успешной аутентификации сервер обслуживает минимальный health-обработчик —
      аутентифицированный `ping` возвращает `pong` (или эквивалент «жив»); это единственная
      обрабатываемая операция. Маршрутизация на реальные операции (exec/MCP/upload) — явная
      заглушка/точка расширения, возвращающая «not implemented» без побочных эффектов.
- [ ] AC11: `raxd serve` запускает сервер в foreground (заменяет honest stub из
      `internal/cli/serve.go`); запуск выполняется и проверяется ТОЛЬКО в Docker (baseline §6).
      Регистрация системного сервиса (systemd/launchd) в scope НЕ входит.
- [ ] AC12 (baseline §4): graceful shutdown — по сигналу (SIGINT/SIGTERM) или отмене контекста
      сервер перестаёт принимать новые соединения, корректно завершает активные и сбрасывает
      буферизованные данные (`Store.FlushUsage`), затем выходит без зависания. Тест проверяет
      завершение в пределах разумного дедлайна.
- [ ] AC13: edge-cases обработаны без паники и с понятной ошибкой/ненулевым кодом: пустой/повреждённый
      сертификат в `TLSDir`; занятый порт; отсутствие/повреждение `keys.db` (= «нет валидных ключей»
      → все подключения отвергаются); обрыв соединения на середине handshake.
- [ ] AC14 (baseline §6): все тесты (TLS-версия, права файлов, переиспользование серта,
      аутентификация, rate-limit, аудит без секретов, graceful shutdown, ping→pong, edge-cases)
      зелёные в Docker; сборка/тесты офлайн из `vendor/` (`-mod=vendor`), без `go mod download`.

## Out of Scope
- Реальное выполнение команд по сети, allowlist, таймауты команд — задача `command-exec` (baseline §3).
- MCP-сервер, его tools/resources и транспорт Streamable HTTP — задача `mcp-server`.
- Передача файлов — задача `file-upload`.
- Управление ключами (`key create|list|delete`, хранение, формат, генерация id) — готово в
  `key-management`; здесь только потребление контракта `keystore.Verify`/`Fingerprint`.
- mTLS / клиентские сертификаты (`tls.RequireAndVerifyClientCert`) — РЕШЕНО (дирижёр): отложено как
  будущая опция, фиксируется в baseline §2; аутентификации по API-ключу достаточно для v1.
- Регистрация и управление системным сервисом (systemd unit / launchd plist), права демона,
  capabilities — задача `service-install`/`distribution`.
- Установка, дистрибуция, подпись, install.sh, goreleaser — задача `distribution`.
- Финальный визуальный дизайн вывода `serve`/статус-блоков (lipgloss/tablewriter) — задача `cli-ux`.
- Выбор формата конфигурации новых полей (адрес/лимиты) и их сериализация — architect; здесь
  зафиксирован лишь источник порта (`config.Config.Port`) и каталог TLS (`PathSet.TLSDir`).
- Ротация аудит-лога и его файловый бэкенд (journald/syslog/файл) — уровень сервиса/дистрибуции;
  здесь обязательны только структура и состав полей записи.

      будущим MCP Streamable HTTP и Origin-валидацией). Выбор с источниками — следующий шаг pipeline;
      влияет на форму AC4 (как передаётся ключ) и на Q2.
- [ ] Q2 (Origin-валидация, baseline §2) — делегировано research-analyst/architect, зависит от Q1.
      Рекомендуемый дефолт PM — Origin-middleware сразу при выборе HTTP/TLS; при сыром TCP — заложить
      как требование, активируемое в `mcp-server`.
- Q3 (mTLS) — РЕШЕНО (дирижёр): отложено, mTLS вне scope этой задачи (см. Out of Scope), будущая
  опция в baseline §2.
