# ADR-003: Политика допустимых значений поля `mode` для upload_file

## Контекст
`upload_file` принимает опциональное поле `mode` (восьмеричный POSIX-режим, напр. `"0600"`) — права
создаваемого файла (AC2/AC9). spec Q2 явно делегирует architect/security точную политику допустимых
значений: диапазон/маску и запрет опасных битов (setuid/setgid/sticky, world-writable). research
(Q5/ADR-002) зафиксировал механизм выставления прав (chmod по fd, umask-независимо, дефолт 0600), но НЕ
числовую политику диапазона — это решение architect.

Запись файла идёт по сети по запросу ИИ-агента. Если позволить произвольный `mode`, клиент мог бы
создать файл с битом setuid/setgid (потенциальный вектор эскалации привилегий, особенно если демон
когда-либо стартует от root, AC11) или world-writable файл (любой локальный пользователь хоста сможет
его перезаписать). Это противоречит духу spec (Out of Scope: «setuid/sudo, эскалация привилегий —
вне scope»; «инструмент создаёт ТОЛЬКО обычный файл»).

## Решение
Поле `mode` (и дефолт `default_mode`) валидируется функцией `fileupload.ParseMode`:
1. Строка парсится как восьмеричное число (база 8). Непарсимое → `ErrBadMode` (AC14 → deny).
2. Допустимы только биты прав в маске **`0777`** (rwx для user/group/other). Любой бит вне `0777` —
   то есть **setuid (`04000`), setgid (`02000`), sticky (`01000`)** — запрещён → `ErrBadMode`.
3. Дополнительно запрещён **world-writable** бит (`0002`) → `ErrBadMode`.
4. Результат — `fs.FileMode` в диапазоне `0000..0777` без `0002`, применяемый через `(*os.File).Chmod`
   по fd ДО записи (ADR-002, umask-независимо).

Дефолт `default_mode = "0600"` (AC9) проходит ту же валидацию на старте (невалидный дефолт в конфиге →
ошибка загрузки, AC15). Владелец файла — UID/GID демона как есть, без chown/setuid (AC9).

## Альтернативы
- **Разрешить любой `mode` 0000..07777 (включая setuid/setgid/sticky).** Отвергнута: даёт вектор создания
  setuid-файла по сети (эскалация привилегий) — прямо против Out of Scope spec («эскалация привилегий вне
  scope»; «только обычный файл»).
- **Жёстко фиксировать 0600, игнорировать поле `mode`.** Отвергнута: AC2/AC9 явно требуют принимать поле
  `mode` и применять заданный режим (тест AC9 «заданный при наличии»). Полный запрет нарушил бы контракт.
- **Разрешить 0000..0777 без явного запрета world-writable.** Отвергнута как недостаточная: world-writable
  файл на хосте — локальный риск целостности (любой пользователь хоста перезапишет доставленный артефакт);
  дешёвый запрет `0002` усиливает безопасность без потери легитимных сценариев (rw для user/group хватает).

## Последствия
- Плюсы: по сети нельзя создать setuid/setgid/sticky/world-writable файл (защита от эскалации и локальной
  подмены); поле `mode` поддержано (AC2/AC9); валидация дешёвая, юнит-тестируемая (`mode.go`, без I/O);
  дефолт 0600 строг (STACK §Раскладка «права 0600» для чувствительных файлов).
- Минусы / цена: клиент не может выставить setuid/setgid/sticky даже если легитимно нужно (для таких
  сценариев — ручная настройка на хосте вне инструмента, согласовано с Out of Scope); чуть больше кода
  валидации (несколько строк маски).
- Влияние на стек: без новых зависимостей (stdlib `io/fs`, `strconv` для парса базы 8); согласуется с
  ADR-002 (chmod по fd) и AC9.

## Зависимость от security
Достаточность маски `0777` без `0002` и запрета setuid·setgid·sticky подтверждает **security** в
threat-model.md (риск эскалации/локальной подмены + смягчение). Этот ADR фиксирует политику диапазона;
механизм применения (chmod по fd, umask-независимо) — в ADR-002.
