# Spec: file-upload — безопасная передача файла НА хост по запросу ИИ-агента (`upload_file`)

Автор продукта: Vladimir Kovalev, OEM TECH

## Goal
Дать ИИ-агенту (MCP-клиенту) возможность записать файл на хост `raxd` — поверх УЖЕ ГОТОВОГО
аутентифицированного HTTP/TLS-транспорта и MCP-сервера. Это опасная операция (запись в ФС хоста по
сети), поэтому фича подчинена принципам `SECURITY-BASELINE.ru.md` (валидация входа §3, аудит каждой
записи §4): запись ТОЛЬКО внутрь конфигурируемого разрешённого корня (без path traversal/симлинков
наружу), с лимитом размера, контролируемыми правами файла, атомарной записью и аудитом каждой загрузки
без логирования содержимого. Поверхность — новый MCP-инструмент `upload_file`; транспорт,
аутентификация, rate-limit и аудит НЕ переписываются, а потребляются (как у `execute_command`).

## User Stories
- Как ИИ-агент (MCP-клиент), я хочу передать файл по относительному пути и его содержимое и получить
  подтверждение записи, чтобы доставлять артефакты (скрипты, конфиги, данные) на удалённый хост.
- Как специалист по безопасности, я хочу, чтобы любая запись файла была ограничена разрешённым корнем,
  лимитом размера, контролируемыми правами, и фиксировалась в аудите без содержимого, чтобы
  расследовать инциденты и предотвращать обход ФС и заполнение диска.
- Как администратор `raxd`, я хочу управлять поведением загрузки (корень, лимит размера, права,
  политика перезаписи) через конфиг с безопасными дефолтами, чтобы не открывать лишних возможностей.

## Acceptance Criteria
- [ ] AC1 (поверхность): загрузка файла доступна ТОЛЬКО через MCP-протокол как инструмент `upload_file`
      (виден в `tools/list`, вызывается через `tools/call`). Дополнительных сетевых эндпоинтов и
      CLI-подкоманды `raxd upload` в этой задаче НЕТ (по аналогии с `execute_command`: не плодить
      дублирующих поверхностей с собственной auth/аудитом). Инструмент наследует ту же цепочку
      аутентификации, валидации Origin/Host, rate-limit и аудита транспорта, что и `execute_command`
      (новых каналов и маршрутов не вводится). Тест: после `serve` инструмент присутствует в
      `tools/list` рядом с `ping`/`server_info`/`execute_command`; других сетевых поверхностей загрузки
      и CLI-подкоманды нет.
- [ ] AC2 (формат входа): инструмент принимает `path` (строка, обязателен — относительный путь внутри
      upload root), `content` (строка, обязателен — содержимое в кодировке base64), `overwrite`
      (булево, опц., дефолт `false`), `mode` (строка, опц. — восьмеричный POSIX-режим, напр. `"0600"`).
      Поле для абсолютных путей и поле для произвольного владельца НЕ принимаются. Любые неизвестные
      поля отвергаются (строгая схема, `additionalProperties:false`). Тест: запрос с лишним полем →
      ошибка валидации входа (`isError:true`), файл не создаётся.
- [ ] AC3 (формат выхода): успешный вызов возвращает структурированный результат с полями
      `path` (итоговый относительный путь, как принят сервером), `size` (целое — число записанных
      байт декодированного содержимого), `overwritten` (булево — был ли перезаписан существовавший
      файл), `mode` (строка — фактический режим созданного файла). Абсолютный путь файла на хосте в
      результат НЕ включается (не раскрывать раскладку ФС сверх необходимого). Тест: загрузка
      известного содержимого даёт ожидаемые `path`/`size`/`overwritten`/`mode`; форма соответствует
      объявленной выходной схеме.
- [ ] AC4 (path traversal — главный контроль безопасности): целевой путь резолвится ТОЛЬКО внутрь
      разрешённого upload root. Отвергаются (до записи, `isError:true`, deny): `path` с `..`-сегментами,
      выводящими за корень; абсолютный `path`; путь, итоговый абсолютный резолв которого (после
      нормализации и разыменования родительских компонентов) выходит за пределы upload root, в т.ч.
      через симлинк, указывающий наружу. Тест: каждый вектор — `"../etc/passwd"`, `"/etc/passwd"`,
      `"a/../../b"`, путь через символьную ссылку наружу — отклоняется (`isError:true`, deny), файл вне
      корня НЕ создаётся и существующий файл вне корня НЕ изменяется.
- [ ] AC5a (безопасный дефолт корня): при пустом/некорректном upload root в конфиге используется
      безопасный дефолт (НЕ `/`, НЕ корень файловой системы, НЕ `/root`, НЕ домашний каталог root) —
      конкретное значение фиксируется конфигом (см. AC15). Тест: при отсутствии/невалидной настройке
      корня файл записывается в безопасный дефолтный каталог (не в `/`, не в `/root`), а не отказывает
      молча с записью в небезопасное место.
- [ ] AC5b (создание подкаталогов внутри корня): недостающие промежуточные подкаталоги внутри upload
      root создаются автоматически при загрузке; создание каталогов ВНЕ корня невозможно (следствие
      AC4). Тест: валидная загрузка по пути с несуществующим промежуточным подкаталогом внутри корня
      создаёт этот подкаталог и файл по ожидаемому пути внутри корня.
- [ ] AC6 (содержимое — base64): `content` декодируется как base64; невалидный base64 → `isError:true`
      (fail/deny), файл не создаётся. Тест: валидный base64 даёт на диске байты, точно равные исходным
      (включая бинарные данные); невалидный base64 отклоняется без записи.
- [ ] AC7 (лимит размера файла): размер ДЕКОДИРОВАННОГО содержимого ограничен конфигурируемым
      максимумом `max_file_bytes`. Превышение → `isError:true` (deny), файл не создаётся (в т.ч. не
      создаётся частичный/временный файл, остающийся на диске). Тест: содержимое, декодированный размер
      которого превышает лимит, отклоняется; файл (и временный файл) на диске отсутствуют. (Соотношение
      этого лимита с лимитом тела транспорта — см. AC16.)
- [ ] AC8 (перезапись по умолчанию запрещена): при существующем целевом файле и `overwrite:false`
      (дефолт) запись отклоняется → `isError:true` (deny), существующий файл НЕ изменяется. При
      `overwrite:true` существующий файл заменяется атомарно (см. AC10). Тест №1: повторная загрузка по
      тому же пути без `overwrite` → `isError:true`, содержимое на диске прежнее. Тест №2: с
      `overwrite:true` → файл заменён новым содержимым, `overwritten:true` в результате.
- [ ] AC9 (права создаваемого файла): создаваемый файл получает предсказуемый POSIX-режим — дефолт из
      конфига (рекомендация `0600`), переопределяемый полем `mode` в пределах разрешённого диапазона
      (политику допустимых режимов задаёт architect/security; AC фиксирует факт контроля и применение
      umask-независимого режима). Файл создаётся под UID/GID процесса демона (наследуется как есть);
      инструмент НЕ повышает привилегии и НЕ меняет владельца (никакого chown/setuid/sudo). Тест:
      созданный файл имеет ожидаемый режим (дефолт при отсутствии `mode`; заданный при наличии) и
      владельца, совпадающего с пользователем демона.
- [ ] AC10 (атомарность записи): запись атомарна — целевой файл становится виден читателям только
      целиком, в финальном состоянии. При ошибке/обрыве ДО фиксации частичный целевой файл НЕ
      появляется, а при перезаписи (AC8) существующий файл остаётся в прежнем состоянии; временный файл
      (если использован) на диске НЕ остаётся. КАК реализовать атомарность — решает architect. Тест:
      при искусственном прерывании до фиксации целевой файл отсутствует (новый путь) или сохраняет
      прежнее содержимое (перезапись); посторонних временных файлов в каталоге назначения не остаётся.
- [ ] AC11 (детекция root): инструмент не повышает привилегии (AC9); дополнительно при эффективном UID
      демона == 0 при КАЖДОМ вызове `upload_file` пишется аудит-запись уровня WARN об операции записи
      файла от root (детекция обязательна). Опциональный жёсткий отказ от записи при root (по аналогии
      с `exec.deny_root`) — оставлен как политическое решение для architect/security; в этом контракте
      обязательны факт детекции и WARN. Тест: вызов `upload_file` при процессе демона, стартующем как
      root (эффективный UID == 0; в Docker-контейнере без ограничения `--user`), порождает WARN-запись
      в аудите.
- [ ] AC12 (аудит каждой загрузки, без содержимого): каждый вызов `upload_file` пишет ровно одну
      основную аудит-запись, содержащую достаточно для расследования инцидента: timestamp (UTC),
      fingerprint ключа (НЕ сам ключ), имя инструмента (`upload_file`), относительный путь назначения
      (внутри корня), размер записанных байт, результат (ok/deny/fail), удалённый адрес (remote).
      СОДЕРЖИМОЕ файла (`content`/декодированные байты) в аудит-лог НЕ пишется НИКОГДА. Способ
      представления полей пути и размера в записи (расширять ли структуру аудита или иначе) — оставлен
      architect/security (см. Open Questions Q5). Тест: success-запись содержит
      timestamp+fingerprint+tool+путь+размер+remote+результат; deny/fail-запись содержит
      fingerprint+путь+причину+remote+результат; декодированное содержимое подстрокой в логе отсутствует.
- [ ] AC13 (без секретов): в аудит-логе, в результате `upload_file` и в сообщениях об ошибках НЕТ тела
      API-ключа, его хэша, соли, raw `Authorization` и приватного TLS-ключа; вместо ключа —
      fingerprint; сообщения об ошибках нейтральны (не раскрывают абсолютные пути ФС/раскладку).
      Тест: предъявленный полный ключ как подстрока отсутствует в захваченном логе и в теле MCP-ответа.
- [ ] AC14 (устойчивость и edge-cases): инструмент НЕ паникует и не отдаёт 5xx на некорректный ввод;
      все перечисленные краевые случаи возвращают `isError:true` с нейтральным сообщением, сервер
      остаётся работоспособным (следующий валидный вызов отрабатывает). Покрытые края: traversal (AC4),
      существующий файл без overwrite (AC8), превышение `max_file_bytes` (AC7), невалидный base64 (AC6),
      целевой путь указывает на существующий КАТАЛОГ (а не файл) → deny, диск полон / ошибка записи →
      `isError:true` (fail, аудит fail), невалидный `mode` → deny. Тест: каждый край → `isError:true`,
      сервер жив, частичных/временных файлов на диске не остаётся.
- [ ] AC15 (конфигурация безопасных дефолтов): параметры загрузки добавляются как секция конфига (по
      аналогии с секцией `exec`) с безопасными дефолтами: upload root (безопасный дефолт по AC5a), лимит
      размера файла `max_file_bytes`, дефолтный режим файла, политика перезаписи по умолчанию
      (`false`). Дефолты применяются при отсутствии config.yaml; значения переопределяемы; невалидные
      значения отвергаются на загрузке. Тест: при отсутствии config.yaml применяются дефолты;
      переопределение работает; невалидный upload root / лимит отвергается на старте.
- [ ] AC16 (лимит размера vs лимит тела транспорта — ограничение зафиксировано): содержимое
      передаётся как base64 в теле MCP-запроса, поэтому тело запроса проходит через наследуемый лимит
      размера тела транспорта (конфиг `max_body_bytes`, дефолт 1 MiB) ДО инструмента. Base64 раздувает
      размер на ~33%, плюс накладные расходы протокола. Значит фактический потолок размера ОДНОГО
      загружаемого файла в текущей архитектуре ≈ `(max_body_bytes − overhead) × 3/4`. Контракт:
      (а) `max_file_bytes` НЕ должен превышать этот выводимый из `max_body_bytes` потолок, иначе файлы у
      верхней границы `max_file_bytes` будут отклоняться транспортом (413) ДО инструмента; (б) если
      требуется загрузка файлов больше этого потолка — это выходит за рамки v1 и требует отдельной
      задачи (chunked/стриминговая загрузка), что ЗАФИКСИРОВАНО в Out of Scope, а не решается молча.
      Тест №1: файл, чьё base64-тело укладывается в `max_body_bytes` и декодированный размер ≤
      `max_file_bytes`, загружается. Тест №2: запрос, чьё тело превышает `max_body_bytes`, отклоняется
      транспортом (413) ДО инструмента (файл не создаётся).
- [ ] AC17 (аутентификация наследуется): вызов `upload_file` проходит ту же Bearer-аутентификацию
      транспорта ДО записи; запрос без `Authorization: Bearer`, с неизвестным/отозванным ключом → 401
      (до инструмента не доходит), повреждение хранилища ключей → 403. Новых каналов аутентификации не
      вводится. Тест: неаутентифицированный вызов `upload_file` не создаёт никакого файла.
- [ ] AC18 (rate-limit наследуется): вызовы `upload_file` подчиняются существующему per-key и per-IP
      rate-limit транспорта (429 при превышении ДО исполнения инструмента). Тест: превышение лимита
      отклоняется до записи файла.
- [ ] AC19 (наследование планки command-exec): аудит загрузки ведётся по той же модели, что у
      `execute_command` (собственный аудит операции с полным набором полей результата, без потери
      записи о deny/fail), а не делегируется обобщённому аудиту, не различающему исход операции. Тест:
      на один вызов `upload_file` в аудите ровно одна основная запись с корректным результатом
      (ok/deny/fail); deny/fail-вызов даёт запись с соответствующим результатом, а не теряется.
- [ ] AC20 (тесты в Docker, офлайн из vendor): все тесты выше зелёные в Docker; сборка/тесты офлайн из
      `vendor/` (`-mod=vendor`), без `go mod download`; запуск демона — только в контейнере (baseline §6),
      не на хосте; новых внешних зависимостей file-upload не вводит (стандартная библиотека + уже
      вендоренные). Тест: CI/локальный прогон в контейнере проходит из `vendor/`.

## Зависимости от готового кода
- Инструмент `upload_file` наследует auth / Origin / rate-limit / аудит транспорта и регистрируется в
  той же точке расширения MCP-сервера, что и `execute_command` (транспорт и протокол НЕ
  переписываются, а потребляются). Образец планки и паттернов — задача `command-exec`
  (`specs/command-exec/`): собственный аудит операции в обработчике, типизированные вход/выход со
  строгой схемой, проверка лимитов входа ДО действия, детекция root + WARN, опциональный hard-fail.
- Внешний предел размера тела запроса наследуется от транспорта (`max_body_bytes`) — см. AC16.
- Атомарная запись на диск и секция конфига с безопасными дефолтами в проекте уже применяются в
  смежных модулях — это образец, не предписание (что и как реализовать, решает architect).

> Конкретные имена пакетов, сигнатуры функций, точки интеграции и образцы кода — в справочном файле
> `specs/file-upload/context.md` (для architect/research-analyst; НЕ часть контракта).

## Out of Scope
- Скачивание файлов с хоста (`download_file`) и любое чтение ФС хоста — отдельная задача.
- Передача каталогов, рекурсивная загрузка, архивы с авто-распаковкой — не в v1.
- Стриминговая / chunked / докачиваемая загрузка больших файлов (больше потолка, выводимого из
  `max_body_bytes`, см. AC16) — отдельная задача (требует stateful-канала или multipart); v1 — один
  файл целиком в теле запроса в пределах лимита тела.
- Создание символических/жёстких ссылок и спецфайлов (FIFO/устройства) через инструмент — запрещено;
  инструмент создаёт ТОЛЬКО обычный файл.
- Смена владельца/группы созданного файла (chown), setuid/sudo, эскалация привилегий — вне scope (AC9
  фиксирует «не повышать привилегии» и наследование UID/GID демона).
- Изменение модели системного сервиса, выбор системного пользователя, не-root раскладка —
  свойство развёртывания (`service-install`/`distribution`); здесь только «не повышать привилегии» и
  детекция root-демона (AC11).
- Антивирус/контент-фильтрация/проверка типа содержимого загружаемого файла — не в v1.
- Управление ключами, TLS, rate-limit, протокол MCP, инструменты `ping`/`server_info`/`execute_command`
  — готовы в смежных задачах, здесь не дорабатываются.

## Open Questions
- None блокирующих. Развилки закрыты обоснованными дефолтами и зафиксированы в AC: поверхность —
  только MCP-инструмент `upload_file` (AC1); вход `path/content(base64)/overwrite?/mode?`,
  `additionalProperties:false` (AC2); выход `path/size/overwritten/mode` без абсолютных путей (AC3);
  traversal/симлинк наружу → deny (AC4); запись только в upload root с безопасным дефолтом (AC5a/AC5b);
  base64 + лимит `max_file_bytes` (AC6/AC7); перезапись запрещена без `overwrite` (AC8); права файла —
  дефолт `0600`, наследование UID/GID демона (AC9); атомарность без частичного/временного файла (AC10);
  аудит без содержимого (AC12); ограничение размера vs `max_body_bytes` зафиксировано, большие файлы →
  Out of Scope (AC16). Открытые решения, делегированные architect/security (не блокируют контракт —
  факт-требования уже в AC):
  - [ ] Q1: конкретное числовое значение безопасного дефолта upload root и его расположение
    (рекомендация — выделенный каталог данных raxd, напр. `<data-dir>/uploads`, права `0700`) —
    AC5a/AC15.
  - [ ] Q2: точная политика допустимых значений поля `mode` (диапазон/маска; запрещать ли setuid/setgid/
    sticky и world-writable биты) — AC9.
  - [ ] Q3: политика при root-демоне (только WARN vs опциональный hard-fail для загрузки) — AC11.
  - [ ] Q4: числовое значение `max_file_bytes` и согласованное с ним значение `max_body_bytes` (с учётом
    base64-раздувания ~33% + overhead протокола, AC16) — AC7/AC15/AC16.
  - [ ] Q5: способ представления полей пути и размера в аудит-записи загрузки (расширять ли структуру
    аудита или иначе) — без изменения формата существующих записей — AC12.

## Примечание о размере задачи
Задача среднего размера (20 AC), плотно завязана на безопасность записи в ФС по сети, но НЕДЕЛИМА по
смыслу: безопасная загрузка файла — единый контракт, где валидация пути (анти-traversal), лимит
размера, политика перезаписи, права/владелец, атомарность и аудит без содержимого неразрывны (любое в
отрыве создаёт небезопасную фичу). Разбивка нецелесообразна. Множество AC обусловлено плотностью
обязательных требований безопасности, а не раздуванием scope.
