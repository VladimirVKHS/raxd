# Threat Model: Key Management — управление API-ключами raxd

> Scope этой модели — ЛОКАЛЬНАЯ логика управления ключами из пакета `internal/keystore`
> (`Create`/`List`/`Revoke`/`Verify`/`FlushUsage`/`Fingerprint`) и хранилище `keys.db` (плоский JSON
> + atomic write + advisory-flock, ADR-001/plan). Сеть, TLS, предъявление ключа по `Authorization:
> Bearer` и rate limiting — ВНЕ scope (задачи `tls-transport`, `mcp-server`, `command-exec`), но
> контракт `Verify` (constant-time, чисто читающий) фиксируется здесь как контракт для них.
> Базовый контракт безопасности — `SECURITY-BASELINE.ru.md §1` (целиком) + `§4` (аудит/секреты).

## Активы

- **A1. Тело API-ключа (plaintext `rax_live_<base64url>`)** — высшая ценность. Это секрет
  аутентификации к будущему сетевому доступу к `raxd` (как SSH-ключ). Существует в открытом виде
  только в момент `Create` (одноразовый вывод) и при `Verify` (предъявленное значение в памяти).
  Компрометация = полный доступ к серверу под видом легитимного агента.
- **A2. Хранилище `keys.db`** (путь `internal/config.PathSet.KeysDB`, права `0600`). Содержит на
  запись: `id`, `label`, `created`, `last-used`, `revoked`/`revoked-at`, `sha256(тело+per-key-salt)`,
  `per-key-salt`. НЕ содержит тело ключа. Ценность: метаданные ключей + хэши/соли (косвенный материал
  для офлайн-перебора при слабой энтропии).
- **A3. Аудит-записи create/delete** (timestamp, action, id, fingerprint) — журнал для расследования
  компрометации/отзыва. Ценность: целостность и полнота (несфабрикованность).
- **A4. Корректность отзыва (revoked-состояние)** — гарантия, что отозванный ключ ГАРАНТИРОВАННО не
  проходит `Verify`. Ценность: это и есть kill-switch при компрометации; его обход = непрекращающийся
  доступ скомпрометированного ключа.
- **A5. Целостность хранилища** — отсутствие тихой потери/перезаписи данных (повреждение файла,
  гонка отзыв↔flush, оборванная запись).
- **A6. Энтропия генерации** (источник `crypto/rand`) — фундамент непредсказуемости A1; при слабой
  энтропии вся схема хранения (SHA-256 без медленного KDF) становится уязвима к офлайн-перебору.

## Поверхность атаки

- **CLI-команды** `raxd key create [--name]` / `key list` / `key delete <id>` (`internal/cli/key.go`,
  ныне заглушки). Входы: флаг `--name` (label, ≤64), аргумент `<id>`. Выходы: stdout (полный ключ
  один раз при create; таблица без секретов при list), stderr (аудит, ошибки), exit-коды.
- **Файл `keys.db`** на диске: чтение/запись из ДВУХ процессов — CLI (`raxd key …`) и долгоживущего
  daemon (`raxd serve`, будущие задачи через `Verify`/`FlushUsage`). Конкуренция координируется
  advisory-flock.
- **Временные файлы** atomic-write (temp в каталоге `StateDir`, до `os.Rename`).
- **Логи/stderr демона** (journald/syslog через `charmbracelet/log` как slog-handler) — потенциальный
  канал утечки секрета, если туда попадёт тело/хэш/соль.
- **Аргументы процесса / окружение** — потенциальная утечка ключа, если он передаётся как arg или
  попадает в env (видно в `ps`, `/proc`).
- **Контракт `Verify(presented string)`** — точка предъявления секрета (вызывается будущими сетевыми
  задачами); таргет timing-атаки.

Кто атакует (в рамках локального scope):
- **L1. Локальный непривилегированный пользователь** на том же хосте (другой UID): пытается прочитать
  `keys.db`, temp-файлы, увидеть ключ в `ps`/`/proc`/логах.
- **L2. Атакующий с доступом к копии `keys.db`** (украл бэкап/диск/снапшот): пытается восстановить
  тело ключа из хэшей+солей офлайн-перебором.
- **L3. Будущий сетевой клиент** (через `Verify`, scope `tls-transport`/`mcp-server`): timing-атака
  на сравнение, попытка предъявить отозванный/угаданный ключ. Здесь — только контракт `Verify`.
- **L4. Сбой среды**: внезапное завершение процесса/питания во время записи; параллельный
  отзыв↔flush; повреждение файла на диске.

## Угрозы (STRIDE-ориентир)

- **S (Spoofing):** подбор/угадывание тела ключа (слабая энтропия → офлайн-перебор хэшей из A2);
  предъявление отозванного ключа, который из-за гонки/бага «воскрес» в `Verify`.
- **T (Tampering):** оборванная/частичная запись `keys.db` повреждает хранилище; гонка `Revoke`
  vs `FlushUsage` затирает revoked-флаг (last-used-апдейт «воскрешает» отозванный ключ); ручная/
  баговая перезапись повреждённого файла теряет все ключи.
- **R (Repudiation):** операции create/delete не зафиксированы в аудите → нельзя доказать, кто и
  когда выпустил/отозвал ключ; либо в аудит попало тело/хэш (само по себе утечка, см. I).
- **I (Information disclosure):** утечка A1 в `key list`, в логи/ошибки, в аргументы процесса, в
  temp-файлы; кража A2 (хэши+соли); хранение тела ключа в открытом виде в `keys.db`.
- **D (Denial of service):** повреждённый/нечитаемый `keys.db` → паника/краш вместо понятной ошибки;
  deadlock/бесконечное ожидание flock между CLI и daemon. (Сетевой DoS/rate-limit — вне scope.)
- **E (Elevation of privilege):** не специфично для локальной key-логики; права `0600`/`0700`
  ограничивают эскалацию через файлы (запуск не от root — задача `system-dev`/baseline §3, вне scope).

## Риски (риск → вероятность/влияние → смягчение)

- [ ] **R1. Слабая энтропия тела ключа** (использован `math/rand`, недостаточно байт, или «проглочен»
  сбой генерации) → ключ предсказуем, A2 ломается офлайн-перебором.
  → вероятность низкая / влияние критическое
  → смягчение: тело генерируется ТОЛЬКО `crypto/rand` (план: 32 байта = 256 бит, ≥ baseline-минимум
  16 байт); `math/rand` запрещён (grep по пакету `internal/keystore`); идиома Go 1.24 `rand.Read` —
  сбой источника = краш процесса, нельзя «глотать» ошибку и НЕЛЬЗЯ подменять на `math/rand` как
  fallback. То же для per-key-salt (≥16 байт `crypto/rand`) и id (8 байт `crypto/rand`). (baseline §1)

- [ ] **R2. Хранение тела ключа в открытом виде** в `keys.db` → кража A2 = немедленная компрометация
  всех ключей.
  → вероятность низкая / влияние критическое
  → смягчение: в `keys.db` хранится ТОЛЬКО `sha256(тело+per-key-salt)` (конкатенация key+salt per
  baseline §1) и сам per-key-salt; тело/`PlainKey` НИКОГДА не сериализуется. Поля `hash`/`salt` —
  неэкспортируемые с явными JSON-тегами; тип `PlainKey` без JSON-тегов и не входит в `Record`/
  `Database`. Тест: подстрока тела выпущенного ключа отсутствует в файле `keys.db`. (baseline §1)

- [ ] **R3. Кража `keys.db` (A2)** локальным пользователем/из бэкапа → что получает атакующий.
  → вероятность средняя / влияние среднее (ограничено)
  → смягчение: файл `0600`, каталог `0700` (каркас, не расширяется) — ограничивает доступ владельцем.
  Даже при краже атакующий получает только хэши+соли, НЕ тело; восстановление невозможно офлайн при
  ≥128 бит энтропии (см. R1) — это и есть обоснование, почему SHA-256 (быстрый хэш) достаточен, а
  bcrypt/argon2 избыточны (research Q3, baseline §1). Остаточный риск: метаданные (label/время) видны
  при краже — приемлемо, секретами не являются. (baseline §1)

- [ ] **R4. Timing-атака на `Verify`** (использовано `==`/`strings.EqualFold` по секрету/хэшу) →
  побайтовое восстановление валидного значения сетевым клиентом (L3).
  → вероятность средняя (при сетевом доступе) / влияние критическое
  → смягчение: сравнение хэшей ТОЛЬКО constant-time (`crypto/subtle.ConstantTimeCompare` или
  `hmac.Equal`) на КАЖДОМ кандидате; запрет `==`/`EqualFold`/`bytes.Equal` по секретам и хэшам (grep
  по `internal/keystore`). Сравниваются хэши фиксированной длины 32 байта → длина не «протекает».
  Тест: успех/провал `Verify` до и после `Revoke`. (baseline §1)

- [ ] **R5. Утечка ключа в `key list`/логи/ошибки** → раскрытие A1 (I).
  → вероятность средняя / влияние критическое
  → смягчение: полный ключ печатается РОВНО ОДИН РАЗ при `Create` (+предупреждение, что повторно не
  получить); `List` возвращает только id/label/created/last-used (без секрета/хэша/соли); сообщения
  ошибок и аудит оперируют id/label/fingerprint, но НЕ телом/хэшем/солью. Тесты: вывод `list` не
  содержит ключ/хэш/соль; текст ошибок не содержит секрет. (baseline §1, §4)

- [ ] **R6. Утечка ключа через аргументы процесса / окружение** → ключ виден в `ps`/`/proc`/env
  другому локальному пользователю (L1).
  → вероятность низкая / влияние высокое
  → смягчение: тело ключа НЕ передаётся как аргумент CLI и НЕ пишется в переменные окружения;
  `Create` возвращает `PlainKey` и печатает его в stdout (одноразово), предъявление при `Verify` —
  через память/будущий сетевой заголовок, не через arg/env. Инспекция: ни одна команда key-логики
  не принимает тело ключа как позиционный аргумент/флаг. (baseline §1, §4)

- [ ] **R7. Аудит create/delete отсутствует или содержит секрет** → невозможно расследовать выпуск/
  отзыв (R), либо аудит сам становится каналом утечки (I).
  → вероятность средняя / влияние высокое
  → смягчение: `Create` и `Revoke` порождают структурную аудит-запись `timestamp, action, id,
  fingerprint` (через `charmbracelet/log` как slog-handler на stderr демона/journald). `Fingerprint`
  = короткий префикс `sha256(тело)` в hex (8-12 симв., без соли) — идентифицирует ключ, не раскрывая
  его. В аудит НЕ пишется тело/хэш/соль. Тесты: запись присутствует для create и delete; grep по
  выводу аудита не находит тело/хэш/соль. (baseline §1, §4)

- [ ] **R8. Обход отзыва — гонка `Revoke` ↔ `FlushUsage` («воскрешение» revoked-ключа)** → отозванный
  ключ продолжает проходить `Verify` (S/T, обход A4 — kill-switch).
  → вероятность средняя / влияние критическое
  → смягчение: `Verify` — ЧИСТО ЧИТАЮЩИЙ (под shared flock, файл НЕ перезаписывается; `LastUsed`
  только буферизуется в памяти). `FlushUsage` под ЭКСКЛЮЗИВНЫМ flock ПЕРЕЧИТЫВАЕТ актуальный файл и
  мерджит буфер `LastUsed` ПОВЕРХ него, причём revoked-записям `LastUsed` НЕ трогает → отзыв не
  теряется. `Revoke` под эксклюзивным flock (read-modify-write). Тест: `Revoke` с последующим
  `FlushUsage` (включая буферизованный `LastUsed` по этому id) оставляет ключ revoked и `Verify`
  по-прежнему неуспешен. (baseline §1)

- [ ] **R9. Отзыв не мгновенный / revoked-ключ остаётся кандидатом** → скомпрометированный ключ
  продолжает работать после `delete` (обход A4).
  → вероятность низкая / влияние критическое
  → смягчение: `Revoke` — мягкий отзыв (`Revoked=true`+`RevokedAt`, запись сохраняется для аудита),
  атомарно фиксируется; `Verify` перебирает ТОЛЬКО активные (revoked исключены из кандидатов) →
  верификация ранее предъявленного значения немедленно неуспешна. Тест: `Verify` успешен до
  `Revoke` и неуспешен сразу после. (baseline §1)

- [ ] **R10. Повреждение/частичная запись `keys.db`** (сбой питания/процесса во время записи) → потеря
  всех ключей или паника при чтении (T/D, обход A5).
  → вероятность низкая / влияние высокое
  → смягчение: atomic write — temp-файл В ТОМ ЖЕ каталоге → `Chmod 0600` → `Sync` → `Close` →
  `os.Rename` (POSIX-rename атомарен; читатель видит старый ИЛИ новый файл, не частичный) → `fsync`
  родительского каталога (durability rename при сбое питания). Чтение повреждённого/нечитаемого файла
  → `ErrCorrupt`, БЕЗ паники и БЕЗ перезаписи исходника. Отсутствующий файл = пустое хранилище (не
  ошибка) для `List`/`Verify`. Тесты: подсунутый повреждённый файл → `ErrCorrupt` + исходник не
  изменён; отсутствующий файл → пустой результат. (baseline §1)

- [ ] **R11. Утечка секрета через temp-файлы atomic-write** → temp с правами по умолчанию (umask) или
  оставшийся temp читается локальным пользователем (L1, I).
  → вероятность низкая / влияние среднее (хранится только хэш+соль, не тело — но всё равно material)
  → смягчение: temp-файл создаётся/`Chmod 0600` ДО записи содержимого (нет окна с более широкими
  правами); temp в том же каталоге `0700`; при ошибке записи temp удаляется (не «течёт»); тело ключа
  в temp не попадает (в файл пишется только `Database` без `PlainKey`). Тесты: права temp на момент
  записи `0600`; после ошибки записи temp-файлы не остаются. (baseline §1)

- [ ] **R12. Deadlock/вис на flock между CLI и daemon** → `key …` зависает, недоступность управления
  (D); либо незащищённый read-modify-write повреждает данные (T).
  → вероятность низкая / влияние среднее
  → смягчение: read-modify-write (`Create`/`Revoke`/`FlushUsage`) — под ЭКСКЛЮЗИВНЫМ flock; чтение
  (`List`/`Verify`) — под SHARED flock; flock держится ТОЛЬКО на время одной операции (не на всё
  время открытия — в отличие от bbolt, ADR-001), освобождается всегда (в т.ч. при ошибке). Тест:
  параллельные операции не повреждают файл и не виснут бесконечно. (baseline §1)

- [ ] **R13. Утечка тела ключа из памяти процесса (swap/дамп/GC)** → plaintext A1 живёт в куче дольше
  необходимого, может попасть в swap или core-dump (I).
  → вероятность низкая / влияние среднее
  → смягчение (best-effort): время жизни `PlainKey` минимизируется — генерируется, отдаётся для
  одноразового вывода, не сохраняется в долгоживущих структурах/полях `Store`/`Record`; не пишется в
  swap-чувствительные места (логи/файлы/temp). См. остаточный риск ОР-1: managed-память Go (GC,
  копирование строк, отсутствие гарантированного wipe) не позволяет ДЕТЕРМИНИРОВАННО затереть
  plaintext — это ограничение языка, фиксируется честно, не выдаётся за гарантию. (baseline §1, §4)

## Остаточные риски / эскалации

- [ ] **ОР-1. Гарантированное затирание plaintext-ключа в памяти невыполнимо средствами Go.**
  Почему: строки в Go иммутабельны, GC может копировать/перемещать значения, нет переносимого
  гарантированного `memzero` или `mlock` без CGO/unsafe (а стек требует `CGO_ENABLED=0`). Поэтому
  baseline-намерение «минимизировать жизнь plaintext» выполнимо лишь как **best-effort** (R13), без
  детерминированной гарантии стирания/защиты от swap.
  → компенсирующий контроль: минимизация времени жизни `PlainKey`; запрет писать тело в логи/файлы/
  temp/аргументы (R5, R6, R11); опора на права `0600`/`0700` и (для будущего runtime) на
  изоляцию демона. На дисковую защиту от swap полагаемся на ОС/окружение (контейнер, baseline §6),
  не на код key-management.
  → **эскалировать пользователю до релиза**: подтвердить, что best-effort-минимизация без
  гарантированного memzero — приемлемый уровень для v1 (детерминированный wipe потребовал бы
  CGO/`unsafe`, что конфликтует с `CGO_ENABLED=0`).

- [ ] **ОР-2. `Verify` — O(n) линейный перебор активных записей.** Каждое предъявление сравнивает
  кандидатов constant-time по очереди. Для малого набора ключей (spec D4: жёсткого лимита нет, но
  набор мал) приемлемо и НЕ ухудшает безопасность (constant-time сохранён на каждом сравнении).
  → компенсирующий контроль: оптимизация через lookup-индекс (бессолевой `sha256(тело)` или
  встроенный lookup-id в формате ключа) сознательно отложена — она меняет формат/схему и относится к
  architect (research Q6). Риск безопасности нулевой; это перф-замечание.
  → эскалация не требуется (зафиксировано как осознанный trade-off plan).

- [ ] **ОР-3. Сетевые контроли baseline §4 (rate limiting per-key/per-IP, аудит сетевых запросов с
  удалённым адресом/командой/exit/длительностью, обнаружение всплеска отказов аутентификации) НЕ
  входят в scope** этой задачи (spec «Out of Scope»; относятся к `command-exec`/`mcp-server`/
  `tls-transport`).
  → компенсирующий контроль: данная задача обеспечивает фундамент — corant-time `Verify` (защита от
  online-подбора по таймингу) и fingerprint в аудите; сетевые ограничители обязаны быть реализованы
  в сетевых задачах и проверены их security-итерацией.
  → эскалация не требуется по этой задаче, но зафиксировать как **передаваемое требование** в
  threat-model задач `tls-transport`/`mcp-server`/`command-exec` (baseline §4 там обязателен).
