# Guardian Report: architect-guardian — key-management

## Summary
План сильный и почти полный: выбран ровно один подход (плоский JSON + atomic write), модули с путями, контракты с сигнатурами и обработкой ошибок, тел функций нет, AC не изменены, D1-D5 и SECURITY-BASELINE §1 отражены, идиома Go 1.24 `rand.Read` зафиксирована. Содержательное противоречие: `Verify` делает daemon писателем на каждой успешной верификации, что подрывает обоснование выбора формата и усиливает риск конкурентной записи.

## Issues
- [ ] Issue 1 (MUST, корректность конкурентного доступа): контракт `Verify` обновляет `LastUsed` + атомарно перезаписывает весь файл на КАЖДОМ успешном предъявлении ключа.
  - Где: plan.md Contracts (Verify) против Trade-offs («daemon в основном читает, запись редка»).
  - Почему: задание требует учесть конкурентный доступ daemon↔CLI. daemon станет писателем на каждый запрос → конфликт last-write-wins с CLI `Revoke`/`Create` (отзыв может быть затёрт обновлением LastUsed); перезапись всего файла на каждый запрос — ещё и перф-проблема. Обоснование выбора JSON («почти не пишет») перестаёт держаться; `flock` назван лишь «заделом».
  - Что делать: разрешить противоречие одним решением — (а) убрать запись LastUsed с горячего пути Verify (обновлять в памяти + периодический/при завершении flush, или только при CLI-операциях, или троттлинг), И/ИЛИ (б) сделать flock/единый писатель ОБЯЗАТЕЛЬНОЙ частью контракта v1 (не «задел») с описанием поведения при конфликте. Решение согласовать с обоснованием выбора формата.
- [ ] Issue 2 (minor, durability): в цепочке atomic write (temp→Chmod 0600→Sync→Close→Rename) не указан fsync родительского каталога после rename.
  - Что делать: дополнить контракт шагом fsync каталога после rename, либо явно отметить делегирование durability каталога ОС с обоснованием для v1.

## Looks good
- Чёткое следование ADR-001/research (bbolt отклонён по блокировке, sqlite по весу; цена названа).
- Безопасность аккуратна: sha256(тело+per-key-salt)+salt конкатенацией, constant-time на каждом сравнении, Fingerprint без соли, аудит id/fingerprint без тела — соответствие §1.
- Go 1.24 rand.Read и edge-cases (нет файла=пусто; повреждён→ErrCorrupt без перезаписи) зафиксированы.

## Повторная проверка (раунд 2)
Оба issue закрыты содержательно, новых проблем нет:
- Issue 1: `Verify` стал чисто читающим под shared flock (файл не перезаписывается, LastUsed в память); добавлен `FlushUsage()` под exclusive flock (read-merge-write поверх актуального состояния, revoked не трогает → параллельный Revoke не затирается); flock — обязательный контракт v1 (`internal/keystore/lock.go`, syscall.Flock; Create/Revoke/FlushUsage exclusive, List/Verify shared). Противоречие «daemon-писатель» снято.
- Issue 2: fsync родительского каталога после os.Rename добавлен.
- AC не изменены; тел функций нет; новых внешних зависимостей нет (syscall — stdlib, macOS+Linux); §1 сохранён (мгновенный отзыв не подорван — FlushUsage не воскрешает revoked).
