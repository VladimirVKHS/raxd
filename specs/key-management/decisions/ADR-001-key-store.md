# ADR-001: Формат локального хранилища API-ключей (`keys.db`)

## Контекст
Задача `key-management` наполняет логику `key create|list|delete` и функцию верификации (spec AC,
D1-D5). Spec фиксирует путь (`internal/config.PathSet.KeysDB`) и права `0600`, но ВЫБОР формата
хранилища относит к architect/research (см. spec «Out of Scope»). Ограничение стека: `CGO_ENABLED=0`
(статическая сборка, простая дистрибуция — STACK.ru.md), значит хранилище обязано быть pure-Go.
Топология raxd: CLI (`raxd key …`) и долгоживущий daemon (`raxd serve`) — РАЗНЫЕ процессы, которые
могут обращаться к одному файлу. Набор ключей мал (десятки-сотни; жёсткого лимита в v1 нет — D4).
Требуются: атомарность записи, устойчивость к повреждению, простая проверка прав 0600.

## Решение
Рекомендовать (proposed) **плоский файл JSON с атомарной записью**: сериализация через stdlib
`encoding/json`; запись через временный файл в ТОМ ЖЕ каталоге → `Chmod 0600` → `Sync()` → `Close` →
`os.Rename` (POSIX-rename атомарно заменяет целевой файл; читатель видит старый или новый файл, но не
частичный). Это даёт ноль внешних зависимостей при CGO_ENABLED=0 и тривиально удовлетворяет AC по
правам/повреждённому файлу. Финальный выбор остаётся за architect; ADR фиксирует обоснованную
рекомендацию research.
Источники: https://michael.stapelberg.ch/posts/2017-01-28-golang_atomically_writing/ ,
https://pkg.go.dev/github.com/google/renameio , https://pkg.go.dev/encoding/json

## Альтернативы
- **`go.etcd.io/bbolt` v1.4.3** (pure-Go, ACID, crash-safe). Отклонено: bbolt берёт ЭКСКЛЮЗИВНУЮ
  файловую блокировку — два процесса не открывают одну БД одновременно; daemon, держащий keys.db,
  заблокирует CLI-команды (вис/таймаут). Это прямо конфликтует с топологией «CLI + долгоживущий
  daemon на одном файле». Плюс лишняя зависимость и бинарный формат, который сложно чинить вручную.
  → https://pkg.go.dev/go.etcd.io/bbolt , https://github.com/etcd-io/bbolt/blob/main/README.md
- **`modernc.org/sqlite` v1.50.1** (CGo-free SQLite, SQL, индексы, штатная многопроцессность через
  WAL + busy_timeout). Не выбрано как основной вариант: самое тяжёлое дерево зависимостей (~23
  пакета + жёсткая привязка версии `modernc.org/libc`, issue cznic/sqlite#177) — избыточно для малого
  набора ключей. Оставить как ЗАПАСНОЙ путь, если позже понадобятся индексы/прямой lookup для
  верификации (research Q6) или большой объём.
  → https://pkg.go.dev/modernc.org/sqlite , https://sqlite.org/wal.html ,
  https://riverqueue.com/docs/sqlite

## Последствия
- Плюсы: ноль новых зависимостей (stdlib `encoding/json`+`os`), полностью pure-Go при CGO_ENABLED=0;
  атомарная замена устойчива к частичной записи; права 0600 и edge-case «повреждённый файл»
  проверяются тривиально (parse error → понятная ошибка без перезаписи исходника); файл
  человекочитаем, легко инспектировать в тестах (проверка отсутствия тела ключа — AC).
- Минусы / цена: нет встроенных транзакций/индексов; запись читает/пишет файл целиком; конкурентная
  запись из двух процессов требует ручной координации (для key-management запись делает CLI, daemon
  преимущественно читает — конфликт маловероятен; при необходимости добавить `flock`). Верификация
  по предъявленному значению — линейный перебор активных записей с per-key-salt + constant-time
  (research Q6); прямой lookup-индекс потребовал бы SQLite/изменения схемы.
- Влияние на стек (сверка со STACK.ru.md): соответствует CGO_ENABLED=0 и принципу «не вводить
  зависимости без обоснования»; не добавляет строк в `go.mod`; путь/права (`KeysDB`, 0600) и каталог
  состояния (0700) уже заданы каркасом `internal/config/paths.go` и не расширяются.
