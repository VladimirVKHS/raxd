# ADR-002: Вендоринг зависимостей для offline / hermetic-сборок в Docker

> Проектная политика (касается ВСЕХ задач raxd, не только key-management). Зафиксирована
> архитектором по решению дирижёра в автономном режиме.

## Контекст
SECURITY-BASELINE §6 требует, чтобы сборка/тесты/запуск `raxd` шли ТОЛЬКО в Docker и были
воспроизводимы: «CI прогоняет сборку и тесты в контейнере», «имитация свежего сервера». Факт
(подтверждён developer): `proxy.golang.org` НЕДОСТУПЕН из контейнеров (DNS timeout). Холодный
`docker build` с `RUN go mod download` падает без сети; прежние «зелёные» прогоны шли лишь на
ТЁПЛОМ кэше образа — это не воспроизводимо на чистой машине и нарушает §6. Стек тянет ~37
зависимостей (charmbracelet-стек lipgloss/log, cobra, viper, MCP SDK), любая из которых не
скачается офлайн.

## Решение
Перейти на **вендоринг** (вариант A): `go mod vendor` → каталог `vendor/` КОММИТИТСЯ в репозиторий
(НЕ в `.gitignore`); все сборки идут с `-mod=vendor` (Go выбирает его автоматически при наличии
`vendor/` и Go ≥ 1.14). Dockerfile делает `COPY` исходников вместе с `vendor/` и собирает БЕЗ
сетевого `go mod download`. Целостность гарантируется `go.sum` (его проверяет `go mod verify` и
сборка). Это единственный полностью in-repo hermetic-вариант, не зависящий от сети и внешней
инфраструктуры: контейнер строится из того, что уже в git.

## Альтернативы
- **(B) Baked/смонтированный Go module cache в образ.** Минус: кэш надо где-то наполнить (нужна
  сеть на этапе подготовки базового образа), он не лежит в git → «свежая машина» снова зависит от
  внешнего источника; не воспроизводимо по §6. Отвергнут.
- **(C) Внутренний GOPROXY-зеркало (Athens/`GOPROXY=...`).** Минус: новый сетевой сервис +
  инфраструктура + доступность; противоречит «hermetic из коробки» и усложняет CI. Отвергнут.
- **(D) Статус-кво `go mod download`.** Минус: падает без сети, «зелёные» прогоны только на тёплом
  кэше — невоспроизводимо, прямое нарушение §6. Отвергнут (это и есть исходная проблема).

## Последствия
- **Плюсы:** холодный `docker build` и CI работают офлайн; «свежий сервер» воспроизводим из git;
  ноль внешней инфраструктуры; `go.sum` + `go mod verify` дают контроль целостности; goreleaser-сборка
  тоже становится offline-совместимой (собирает из `vendor/`, без download).
- **Цена (плата за выбор):**
  - Размер репозитория растёт ~на 19 МБ (~37 зависимостей из-за charmbracelet-стека).
  - При ЛЮБОМ изменении зависимостей (`go get` / правка `go.mod`) ОБЯЗАН прогнать `go mod vendor`
    и закоммитить обновлённый `vendor/` + `go.sum`; забывчивость → рассинхрон и падение сборки.
  - PR-диффы шумнее (изменения в `vendor/` в ревью игнорируются как сгенерированные).
- **Влияние на стек (сверка со STACK.ru.md):** новых runtime-зависимостей НЕ вводим — вендорятся
  уже принятые библиотеки. `CGO_ENABLED=0` сохраняется. Затрагивает Dockerfile (правит developer),
  goreleaser и CI.
- **Хендофф задаче `distribution`:** goreleaser ОБЯЗАН собирать из `vendor/` (`-mod=vendor`,
  без сетевого download), CI-workflow — без шага `go mod download`; проверка `go mod verify` в CI
  как гейт целостности. install-flow (`curl | sh`) не затрагивается (качает уже собранные бинари).
