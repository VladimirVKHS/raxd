# Spec: Key Management — управление API-ключами raxd

## Goal
Наполнить реальной логикой заглушки `key create|list|delete` из каркаса `bootstrap-cli`: выпуск,
просмотр, отзыв API-ключей и их безопасное локальное хранение/верификация. Ключи — основа
аутентификации будущего сетевого доступа к `raxd`, поэтому генерация, формат, хранение и сравнение
обязаны соответствовать `SECURITY-BASELINE.ru.md §1`. Эта задача — только локальная логика
управления ключами и функция верификации; сеть, TLS, MCP и выполнение команд — вне scope.

## User Stories
- Как администратор сервера, я хочу выпустить именованный API-ключ одной командой и увидеть его
  полностью один раз, чтобы передать агенту и больше нигде секрет не светить.
- Как администратор, я хочу видеть список своих активных ключей с метаданными (без самого секрета),
  чтобы понимать, какие ключи активны и когда использовались.
- Как администратор, я хочу мгновенно отозвать скомпрометированный ключ по его id, чтобы дальнейшая
  проверка этого ключа гарантированно не проходила.

## Acceptance Criteria
- [ ] `raxd key create [--name <label>]` генерирует тело ключа через `crypto/rand` с энтропией
      ≥ 128 бит (≥ 16 байт); `math/rand` не используется (проверяется в коде и тестах).
- [ ] Выпущенный ключ имеет формат `rax_live_<base64url>`, где тело закодировано base64url БЕЗ
      padding (`base64.RawURLEncoding`); `key create` печатает ПОЛНЫЙ ключ ровно один раз и
      сопровождает его предупреждением, что повторно ключ получить нельзя.
- [ ] В хранилище НЕ попадает открытый ключ: хранится только `sha256(тело_ключа + per-key-salt)`
      и сам per-key-salt, как предписано SECURITY-BASELINE §1 (схема — конкатенация key+salt, не
      открытый ключ), плюс метаданные записи (id, label, created, last-used, статус). Точную проверку
      соответствия схемы baseline выполняет роль security; тест подтверждает отсутствие подстроки
      тела ключа в файле хранилища.
- [ ] При создании ключа генерируется отдельный короткий случайный id (через `crypto/rand`, напр.
      8 байт → hex/base32, вид типа `abc123de`); id хранится как ключ записи и используется в
      `list`/`delete`. Тело или хэш ключа НЕ используются как id.
- [ ] `raxd key list` выводит таблицу с колонками id, label, created, last-used; отозванные
      (`revoked`) ключи по умолчанию НЕ показываются. Если у ключа нет label, в колонке выводится
      «-». В выводе НЕТ ни самого ключа, ни хэша, ни salt. На пустом списке — понятное сообщение
      «ключей нет», exit 0.
- [ ] `raxd key delete <id>` выполняет мягкий отзыв: помечает запись статусом `revoked` (запись не
      удаляется, сохраняется для аудита). После отзыва верификация ранее предъявленного значения
      этого ключа немедленно неуспешна. Повторный `delete` уже отозванного id и `delete`
      несуществующего id дают понятную ошибку и ненулевой код.
- [ ] Операции `key create` и `key delete` порождают аудит-запись с timestamp, id и fingerprint
      ключа (НЕ телом ключа), как предписано SECURITY-BASELINE §1/§4.
- [ ] Существует функция верификации предъявленного ключа, использующая ТОЛЬКО constant-time
      сравнение (`crypto/subtle.ConstantTimeCompare` / `hmac.Equal`); сравнение по `==` /
      `strings.EqualFold` над секретами отсутствует (проверяется тестом/ревью). Функция доступна как
      контракт для будущих задач (`tls-transport`, `mcp-server`), без привязки к сети.
- [ ] Хранилище ключей создаётся по пути `KeysDB` из каркаса (`internal/config.PathSet.KeysDB`) с
      правами файла `0600`; каталог состояния уже `0700` (создаётся каркасом) и не расширяется.
- [ ] `label` опционален (флаг `--name` можно не указывать); дубликаты label разрешены —
      уникальность гарантируется только по id. Длина label ограничена ≤ 64 символов; превышение —
      понятная ошибка валидации и ненулевой код.
- [ ] Секрет не пишется в логи и не возвращается ни одной командой, кроме одноразового вывода
      `key create`; ошибки и аудит оперируют id/label/fingerprint, но не телом ключа.
- [ ] Обработаны крайние случаи: повреждённый/нечитаемый `keys.db` — понятная ошибка и ненулевой
      код без паники и без перезаписи данных; отсутствующий `keys.db` для `list`/`verify`
      трактуется как пустое хранилище.
- [ ] Юнит-тесты покрывают: энтропию и формат ключа (`rax_live_<base64url>` без padding),
      генерацию отдельного id, отсутствие открытого ключа в хранилище, права `0600`, успех/провал
      constant-time верификации до и после `delete`, наличие аудит-записей create/delete,
      сокрытие revoked в `list`, валидацию длины label, edge-cases (пустое хранилище,
      несуществующий id, повреждённый файл). Тесты зелёные в Docker (SECURITY-BASELINE §6).

## Out of Scope
- TLS, генерация сертификатов, сетевой сервер и предъявление ключа по сети — задача `tls-transport`.
- MCP-сервер и его инструменты/транспорт — задача `mcp-server`.
- Выполнение команд по сети, allowlist, таймауты — задача `command-exec`.
- Rate limiting и аудит-лог сетевых запросов (per-key/per-IP, 429, fingerprint в журнале) —
  относятся к сетевым задачам выше, не сюда.
- Установка, регистрация сервиса, дистрибуция (`install.sh`, goreleaser, подпись) — `distribution`.
- Финальный визуальный дизайн таблицы/вывода (lipgloss, tablewriter-стилизация) — задача `cli-ux`.
- Выбор формата хранилища (плоский файл vs встроенная БД) и схема сериализации — это architect/
  research; здесь зафиксирован только путь (`KeysDB`) и права (`0600`).

## Resolved Decisions
- D1 (кодировка тела ключа): тело — base64url БЕЗ padding (`base64.RawURLEncoding`); полный ключ
  вида `rax_live_<base64url>`.
- D2 (дубликаты label): label опционален, дубликаты разрешены (уникален id, не label); при
  отсутствии label в `key list` показывать «-».
- D3 (поведение delete): мягкий отзыв — запись помечается `revoked`, верификация такого ключа
  немедленно неуспешна; revoked-ключи по умолчанию НЕ показываются в `key list`, но запись
  сохраняется для аудита.
- D4 (лимиты): длина label ≤ 64 символов; жёсткого лимита на число ключей в v1 нет (возможное
  будущее ограничение — отметить, но не реализовывать сейчас).
- D5 (формат id): отдельный короткий случайный id, генерируемый при `create` (напр. 8 байт
  `crypto/rand` → hex/base32, вид типа `abc123de`); хранится как ключ записи, используется в
  `list`/`delete`. Тело/хэш ключа как id НЕ использовать.
