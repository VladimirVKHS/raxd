# Threat Model: service-purge — `raxd service uninstall --purge`

> Модель угроз для необратимой операции полного удаления раксд-присутствия на хосте
> (системный пользователь `raxd` + каталоги state/config, включая `keys.db` и аудит).
> Источники: `spec.md` (AC1–AC10), `plan.md` (метод `Purge`, `validatePurgePath`,
> `verifyTargetUser`), `SECURITY-BASELINE.ru.md` §3,§4. Каждый риск — со смягчением.

## Активы

- **`keys.db`** (хеши API-ключей + соли) — после `--purge` стирается необратимо; до удаления —
  главный секрет хоста. Ошибочное/частичное удаление = потеря доступа без следа.
- **Аудит-лог** действий раксд — стирается `--purge` вместе со state/config. Запись о самом факте
  `--purge` должна пережить удаление (иначе несвязуемое разрушение, repudiation).
- **Системный пользователь `raxd`** и его учётная запись (uid/gid, nologin-shell) — цель удаления;
  ошибочное удаление ЧУЖОГО пользователя = разрушение постороннего сервиса/данных.
- **Каталоги state/config** на хосте (платформенная раскладка) — содержимое удаляется `rm`-логикой;
  ошибочный путь = разрушение посторонних данных вплоть до `/` или `$HOME`.
- **Привилегии оператора** (root/sudo), под которыми идёт purge — мощный контекст исполнения;
  любая инъекция/подмена цели здесь имеет максимальное влияние.
- **Хост целиком** — как косвенный актив: рекурсивное удаление под root с неверным путём = катастрофа.

## Поверхность атаки

- **CLI** `raxd service uninstall --purge [--yes]` — единственный вход в операцию. Атакующий/ошибка:
  пропуск барьера `--yes`, опечатка, скриптовый запуск без проверки, передача окружения.
- **Окружение процесса** — `$HOME`, `cfg.StateDir`/`cfg.ConfigDir`, переменные, влияющие на резолв
  путей и имени пользователя; подменённое окружение может увести удаление на чужой путь.
- **Файловая система** — каталоги state/config могут быть подменены симлинком (TOCTOU между
  проверкой и `rm`), указывая за пределы раскладки.
- **Учётные записи ОС** — getent/passwd (Linux) и dscl (macOS): подменённая/совпавшая по имени, но
  чужая запись пользователя.
- **Внешние утилиты** — `userdel`/`dscl`/удаление каталогов вызываются из процесса; точка для
  command-injection, если аргументы интерполируются через shell.
- Кто атакует/ошибается: оператор (случайная необратимая команда), локальный пользователь с правами
  на подмену окружения/симлинков, скрипт автоматизации без барьеров. Сетевого вектора у фичи нет
  (операция локальная, под привилегиями) — это сужает поверхность, но повышает влияние.

## Угрозы (STRIDE-ориентир)

- **S**poofing (подмена цели): целевой пользователь по имени совпадает, но это ЧУЖАЯ учётка (login-shell,
  не раксд-раскладка) → удаляем постороннего → mitigated `verifyTargetUser` (имя + системность/nologin).
- **T**ampering (подмена пути/окружения): симлинк в каталоге state/config уводит `rm` за пределы
  раскладки; подменённый `$HOME`/`StateDir` → удаление `/`/`$HOME`/системного корня → mitigated
  `validatePurgePath` + `EvalSymlinks` внутри ожидаемой раскладки.
- **R**epudiation (несвязуемое разрушение): аудит стирается вместе со state → нет следа, что и кем
  удалено → mitigated аудит-запись о факте `--purge` ДО физического удаления каталогов.
- **I**nformation disclosure (утечка секретов): содержимое `keys.db`/ключей попадает в вывод отчёта
  или лог удаления → mitigated отчёт оперирует только фактами (что удалено/отсутствовало), без
  содержимого; нейтральные сообщения без сырых трасс.
- **D**enial of service / необратимая потеря: ошибочный `--purge` уничтожает `keys.db` и аудит без
  возврата → блокирует доступ к хосту → mitigated барьер `--yes` (без него НИЧЕГО не удаляется) +
  предупреждение о необратимости; частичное «грязное» состояние исключено упорядочиванием шагов.
- **E**levation of privilege / command-injection: `userdel`/`dscl` через `sh -c` с интерполяцией
  имени → инъекция; запуск без нужных прав маскируется идемпотентностью → mitigated exec БЕЗ shell
  (`runCommandRaw`, SR-91), фиксированные аргументы; `ErrPermission` без частичного состояния;
  отсутствие ≠ нет прав.

## Риски (риск → вероятность/влияние → смягчение)

- [ ] **R-1. Необратимая потеря `keys.db`/аудита по ошибке оператора** → вероятность средняя / влияние
  критическое → смягчение: обязательный `--yes` — без него `Purge` не вызывается и ничего не удаляется
  (AC9, SR-114); предупреждение о необратимости при `--purge` без `--yes` (SR-115). (baseline §4)
- [ ] **R-2. Разрушение записи аудита вместе с данными (repudiation)** → вероятность высокая (если
  не упорядочить) / влияние высокое → смягчение: аудит-запись о факте `--purge` формируется и
  эмитируется ДО физического удаления каталогов (AC8, SR-116). (baseline §4)
- [ ] **R-3. Удаление ЧУЖОГО пользователя ОС (имя совпало, учётка посторонняя)** → вероятность низкая
  / влияние высокое → смягчение: `verifyTargetUser` проверяет имя И системность (nologin-shell ∈
  {`/usr/sbin/nologin`,`/sbin/nologin`,`/usr/bin/false`}); несоответствие → `ErrUserMismatch`, ничего
  не удалено (AC6, SR-117). (baseline §3)
- [ ] **R-4. Удаление опасного/чужого каталога (`/`, `$HOME`, системный корень)** → вероятность низкая
  / влияние критическое → смягчение: `validatePurgePath` отвергает пустой путь, `/`, `$HOME` и его
  родителя, системные корни (`/etc`,`/var`,`/usr`,`/usr/local`); иначе `ErrSuspiciousPath`, ничего не
  удалено (AC7, SR-118). (baseline §3)
- [ ] **R-5. Симлинк-атака / TOCTOU при удалении каталогов** → вероятность низкая / влияние критическое
  → смягчение: путь резолвится `filepath.EvalSymlinks` и обязан остаться ВНУТРИ ожидаемой раскладки
  (`cfg.StateDir`/`cfg.ConfigDir`) непосредственно перед удалением; выход за раскладку →
  `ErrSuspiciousPath` (AC7, SR-119). (baseline §3)
- [ ] **R-6. Command-injection через имя пользователя/путь в `userdel`/`dscl`** → вероятность низкая /
  влияние критическое → смягчение: вызов только через `exec.Command(bin, args...)`/`runCommandRaw`
  БЕЗ shell, фиксированные аргументы, без интерполяции пользовательского ввода в строку команды
  (SR-120). (baseline §3)
- [ ] **R-7. Эскалация привилегий / запуск с лишними правами** → вероятность низкая / влияние высокое
  → смягчение: purge требует уже имеющихся прав (root/sudo) и НЕ повышает их сам (нет setuid/sudo
  внутри); при нехватке прав — `ErrPermission`, exit != 0, hint «запустите с sudo», НИЧЕГО не удалено
  (AC5, SR-121). (baseline §3)
- [ ] **R-8. Частичное «грязное» состояние (демон жив / шаг упал на полпути)** → вероятность средняя /
  влияние высокое → смягчение: строгий порядок privilege-check → Stop → Uninstall → validatePurgePath
  → verifyTargetUser → удаление; если Stop не удался или проверка не прошла — удаление пользователя/
  каталогов НЕ выполняется (AC4, SR-122). (baseline §3,§4)
- [ ] **R-9. Идемпотентность маскирует отказ прав (отсутствие ≠ нет прав)** → вероятность средняя /
  влияние высокое → смягчение: отсутствие пользователя/каталога → успех (exit 0); отказ в правах →
  `ErrPermission` (exit != 0); коды `userdel`/`dscl` различаются и НЕ сводятся к «уже удалено»
  (AC3+AC5, SR-123). (baseline §3,§4)
- [ ] **R-10. Утечка секретов/содержимого `keys.db` в отчёт или лог удаления** → вероятность низкая /
  влияние высокое → смягчение: `PurgeReport` и аудит оперируют только метаданными (action, platform,
  имена путей, флаги удалено/отсутствовало); содержимое ключей/файлов в вывод и лог не читается и не
  печатается; нейтральные сообщения без сырых трасс (AC8, SR-124, унаследованный SR-95). (baseline §4)
- [ ] **R-11. Регрессия `uninstall` без `--purge` (случайно стал удалять данные)** → вероятность низкая
  / влияние критическое → смягчение: отдельный метод `Purge`, `Uninstall` остаётся byte-for-byte; без
  `--purge` путь удаления пользователя/каталогов недостижим (AC2, SR-125). (baseline §3)
- [ ] **R-12. Тесты выполняют реальные `userdel`/`dscl`/`rm` на хосте/в CI** → вероятность средняя /
  влияние высокое → смягчение: поведение покрыто автотестами через `fakeManager` без реальных
  системных команд; прогон в Docker (baseline §6); обе платформы и все ветки отказа (AC10, SR-126).
  (baseline §6)

## Остаточные риски / эскалации

- [ ] **ОР-1. Отсутствие резервной копии `keys.db` перед удалением.** `--purge` только удаляет
  (бэкап/экспорт вне scope — spec Out of Scope). Почему: осознанное решение PM. Смягчение/компенсация:
  двойной барьер (`--purge` + `--yes`) + явное предупреждение о необратимости (SR-114, SR-115).
  Остаточный риск принят PM; эскалировать оператору в документации (tech-writer) как «backup до purge —
  ответственность оператора». Невыполнимого пункта baseline здесь нет.
- [ ] **ОР-2. `validatePurgePath`/`verifyTargetUser` снижают, но не исключают TOCTOU.** Между проверкой
  и удалением остаётся узкое окно (симлинк подменён после `EvalSymlinks`). Почему: stdlib-only,
  без атомарного «открыть-и-удалить-по-fd» в кросс-платформенном виде. Смягчение: проверка пути
  максимально близко к удалению; операция требует root, локальный атакующий с root уже всевластен —
  окно вне реалистичной модели угроз. Остаточный риск принят; документируется. Эскалация не требуется.
- [ ] **ОР-3. SR-95/SR-96 — наследуемые, не новые.** План ссылался на SR-95 (нейтральные сообщения)
  и SR-96 (stdlib-only/Docker) как на существующие требования `service-install`. Решение: НЕ вводим
  одноимённые дубли; в `security-requirements.md` зафиксировано наследование SR-95→SR-124 (содержание),
  SR-96→SR-127 (stdlib-only здесь), новые SR этой фичи — SR-114…SR-127. Ссылки плана закрыты, повисших
  номеров нет. Эскалация: согласовано переназначение в этом артефакте (см. раздел «Закрытие SR-95/SR-96»
  в security-requirements.md).
