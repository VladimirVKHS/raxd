# Security Requirements: service-purge — `raxd service uninstall --purge`

> Каждое требование проверяемо (ответ «выполнено/нет» одним вопросом) и ссылается на пункт
> `SECURITY-BASELINE.ru.md` и AC из `spec.md`. Эти требования ОБЯЗАНЫ выполнить developer/
> system-dev/devops; соответствие проверяют reviewer и security-guardian.
>
> **Нумерация.** Максимальный использованный номер в репозитории — SR-113 (distribution). Новые
> требования этой фичи нумеруются СКВОЗНО с **SR-114** по **SR-127** (14 требований, без пропусков).
> Наследуемые требования (SR-91, SR-95, SR-96 из `service-install`) НЕ дублируются — на них дана
> явная ссылка ниже.

## Закрытие SR-95 / SR-96 (по замечанию architect-guardian)

План `service-purge` ссылается на SR-95 и SR-96. Эти номера **уже существуют** в
`specs/service-install/security-requirements.md` и здесь **НЕ переопределяются** (иначе — дубль
номера):

- **SR-95 (унаследовано)** — «сообщения операций нейтральны и БЕЗ секретов/сырых трасс». В этой
  фиче конкретизируется требованием **SR-124** (нейтральные сообщения `ErrUserMismatch`/
  `ErrSuspiciousPath`/`ErrPermission`, без содержимого `keys.db`). Ссылка плана на SR-95 ⇒ читать
  как «соблюсти унаследованный SR-95, проверяемая конкретика — SR-124».
- **SR-96 (унаследовано)** — «сборка/проверка в Docker офлайн из `vendor/`; stdlib-only, без новых
  зависимостей». В этой фиче подтверждается требованием **SR-127** (только stdlib: `os`, `os/exec`,
  `path/filepath`, `context`; новых зависимостей нет; прогон в Docker). Ссылка плана на SR-96 ⇒
  читать как «соблюсти унаследованный SR-96, проверяемая конкретика — SR-127».

Итог: повисших ссылок нет; SR-95/SR-96 закрыты как наследуемые, новая нумерация — SR-114…SR-127.

## Барьер необратимости (baseline §4; spec AC9, AC8)

- [ ] **SR-114.** Фактическое удаление (пользователь + каталоги) выполняется ТОЛЬКО при явном `--yes`.
  Запуск `--purge` без `--yes` НЕ вызывает `Purge` (или `Purge` при `Confirmed=false` возвращает
  `ErrPurgeNotConfirmed`) и НИЧЕГО не удаляет. Проверяемо: тест `--purge` без `--yes` → exit != 0,
  ни userdel/dscl, ни удаления каталогов не вызвано (fakeManager не получил destructive-вызовов).
  (baseline §4; AC9)
- [ ] **SR-115.** При `--purge` без `--yes` печатается предупреждение о необратимости, явно
  упоминающее стирание `keys.db` и аудита. Проверяемо: вывод содержит предупреждение о необратимости
  до возврата с exit != 0; без `--yes` предупреждение всегда показывается. (baseline §4; AC9)
- [ ] **SR-116.** Аудит-запись о факте `--purge` (action, platform, что удалено/что отсутствовало)
  формируется и эмитируется ДО физического удаления каталогов state/config. Проверяемо: в тесте
  порядок вызовов — аудит-эмит предшествует удалению каталогов; запись не теряется при удалении
  state-каталога. (baseline §4; AC8)

## Граница «не удалять чужого» (baseline §3; spec AC6, AC7)

- [ ] **SR-117.** `verifyTargetUser` подтверждает, что целевой пользователь — раксд-системный аккаунт:
  имя совпадает И shell ∈ {`/usr/sbin/nologin`,`/sbin/nologin`,`/usr/bin/false`} (нет login-shell).
  Несоответствие → `ErrUserMismatch`, НИЧЕГО не удалено. Проверяемо: тест с пользователем-«самозванцем»
  (имя совпало, shell `/bin/bash`) → exit != 0, userdel/dscl не вызван, каталоги не тронуты.
  (baseline §3; AC6)
- [ ] **SR-118.** `validatePurgePath` отвергает пустой путь, `/`, `$HOME` и его родителя, системные
  корни (`/etc`,`/var`,`/usr`,`/usr/local`). При любом таком пути → `ErrSuspiciousPath`, НИЧЕГО не
  удалено. Проверяемо: табличный тест по перечню опасных путей → каждый даёт `ErrSuspiciousPath`,
  exit != 0, удаления не было. (baseline §3; AC7)
- [ ] **SR-119.** Перед удалением каталог резолвится `filepath.EvalSymlinks` и обязан остаться ВНУТРИ
  ожидаемой раскладки (`cfg.StateDir`/`cfg.ConfigDir` из `DefaultConfigForGOOS`); выход за раскладку
  через симлинк → `ErrSuspiciousPath`, НИЧЕГО не удалено. Проверяемо: тест с симлинком, ведущим наружу
  раскладки → `ErrSuspiciousPath`; проверка пути выполнена непосредственно перед удалением. (baseline §3; AC7)

## Выполнение команд без shell (baseline §3; plan SR-91)

- [ ] **SR-120.** Удаление пользователя (`userdel`/`dscl`) и каталогов вызывается ТОЛЬКО через
  `exec.Command(bin, args...)`/`runCommandRaw` БЕЗ shell-интерполяции; имя пользователя и пути
  передаются отдельными фиксированными аргументами, без сборки командной строки и без `sh -c`.
  Проверяемо: grep по коду purge — отсутствие `sh -c`/`bash -c`/конкатенации ввода в команду; имя/путь
  идут как отдельные элементы `args`. (baseline §3; наследует SR-91)

## Привилегии и устойчивость (baseline §3,§4; spec AC4, AC5, AC3)

- [ ] **SR-121.** Purge НЕ повышает привилегии сам (нет внутреннего sudo/setuid); требует уже имеющихся
  прав. При нехватке прав → `ErrPermission`, exit != 0, нейтральный hint «запустите с sudo», НИЧЕГО не
  удалено (нет частично удалённых артефактов). Проверяемо: тест с симуляцией отказа прав → `ErrPermission`,
  destructive-вызовы не совершены. (baseline §3; AC5)
- [ ] **SR-122.** Строгий порядок: privilege-check → (если запущен) Stop → Uninstall → `validatePurgePath`
  → `verifyTargetUser` → удаление пользователя → удаление каталогов. Если Stop не удался или любая
  проверка не прошла — удаление пользователя/каталогов НЕ выполняется (без частичного состояния).
  Проверяемо: тест «демон не остановился» → exit != 0, удаления не было; тест порядка вызовов.
  (baseline §3,§4; AC4)
- [ ] **SR-123.** Идемпотентность не маскирует отказ прав: отсутствие пользователя/каталога → успех
  (exit 0); отказ в правах → `ErrPermission` (exit != 0). Коды `userdel`/`dscl` «нет такого
  пользователя» и «отказано» различаются и НЕ сводятся оба к «уже удалено». Проверяемо: два теста —
  (а) пользователь отсутствует → exit 0; (б) отказ прав → `ErrPermission`, exit != 0. (baseline §3,§4; AC3, AC5)

## Аудит без утечки секретов (baseline §4; spec AC8)

- [ ] **SR-124.** `PurgeReport`, вывод CLI и аудит-запись содержат ТОЛЬКО метаданные (action, platform,
  имена путей, флаги удалено/отсутствовало); содержимое `keys.db`/файлов/ключей не читается и не
  печатается. Сообщения ошибок (`ErrUserMismatch`/`ErrSuspiciousPath`/`ErrPermission`) нейтральны, без
  сырых трасс. Проверяемо: grep по аудит-выводу/отчёту тестов — секретов/содержимого ключей нет; текст
  ошибок не содержит дампов. (baseline §4; AC8; конкретизирует унаследованный SR-95)

## Защита `uninstall` без `--purge` (baseline §3; spec AC2)

- [ ] **SR-125.** Путь удаления пользователя/каталогов недостижим без `--purge`: `Uninstall` остаётся
  byte-for-byte (та же логика, вывод, ошибки), новый необратимый путь изолирован в методе `Purge`.
  Проверяемо: тест `uninstall` без `--purge` → пользователь и каталоги на месте, destructive-вызовы не
  совершены, вывод не изменён. (baseline §3; AC2)

## Тестируемость без вреда хосту (baseline §6; spec AC10)

- [ ] **SR-126.** Поведение покрыто автотестами через `fakeManager` БЕЗ реальных `userdel`/`dscl`/`rm`
  на хосте; покрыты обе платформы (Linux/macOS), оба режима (с `--purge`/без), требование `--yes`,
  идемпотентность, отказ при запущенном демоне, отказ без прав, `ErrUserMismatch`, `ErrSuspiciousPath`.
  Проверяемо: `go test` зелёный без побочных системных эффектов; перечисленные ветки имеют тест.
  (baseline §6; AC10)
- [ ] **SR-127.** Реализация использует только stdlib (`os`, `os/exec`, `path/filepath`, `context`);
  новых внешних зависимостей нет; сборка и тесты прогоняются в Docker. Проверяемо: `go.mod` без новых
  модулей; CI/тесты запускаются в контейнере. (baseline §6; подтверждает унаследованный SR-96)
