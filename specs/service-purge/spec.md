# Spec: service-purge — `raxd service uninstall --purge` (полное удаление с нулевым следом)

## Goal
Дать оператору явный, безопасный способ полностью удалить раксд-присутствие на хосте одной командой
вместо ручного `userdel`/`dscl` + удаления каталогов. Флаг `--purge` у `raxd service uninstall`
дополнительно к обычному снятию сервиса удаляет системного пользователя `raxd` и каталоги
данных/конфига на Linux и macOS. Решает задокументированный риск из `docs/production-readiness.md` §6
(П-2: сегодня пользователь и `keys.db` остаются by-design, ручная зачистка — error-prone). Операция
необратима (стирает `keys.db` и аудит), поэтому должна быть явной и предсказуемой.

## User Stories
- Как оператор, выводящий сервер из эксплуатации, я хочу одной командой `raxd service uninstall --purge`
  убрать сервис, пользователя `raxd` и его данные, чтобы не оставлять `keys.db`/аудит и не делать
  ручные шаги, отличающиеся между Linux и macOS.
- Как оператор, который хочет временно снять сервис, я хочу, чтобы `uninstall` БЕЗ флага вёл себя
  как раньше (данные и пользователь сохраняются), чтобы случайно не уничтожить ключи.

## Acceptance Criteria
- [ ] AC1: `raxd service uninstall --purge` выполняет всё, что делает обычный `uninstall` (снятие unit/plist,
      autostart, CAP_NET_BIND_SERVICE, journald drop-in), И ДОПОЛНИТЕЛЬНО удаляет системного пользователя
      `raxd` и каталоги состояния/конфигурации согласно платформенной раскладке сервиса (обе платформы —
      Linux и macOS).
- [ ] AC2: `raxd service uninstall` БЕЗ `--purge` сохраняет прежнее поведение byte-for-byte: пользователь и
      каталоги данных остаются, вывод и подсказки не меняются (осознанный дефолт П-2).
- [ ] AC3: Идемпотентность: если пользователь `raxd` и/или каталоги уже отсутствуют (или сервис уже снят),
      `--purge` завершается успехом (exit 0) с понятным сообщением о том, что было удалено, а что уже
      отсутствовало; повторный запуск `--purge` подряд также даёт exit 0.
- [ ] AC4: Если демон ещё запущен, `--purge` сначала корректно останавливает сервис (graceful, как `stop`)
      перед удалением пользователя/каталогов; если остановить не удалось — удаление пользователя/данных
      НЕ выполняется, команда завершается ошибкой с понятным сообщением (без частичного «грязного» состояния).
- [ ] AC5: Недостаток прав (нужен root/sudo для `userdel`/`dscl`/удаления системных каталогов): команда
      завершается с `ErrPermission`-семантикой — exit-код, отличный от 0, нейтральное сообщение и hint
      «запустите с sudo»; чужие изменения при этом не вносятся (никаких частично удалённых артефактов).
- [ ] AC6: Граница «не удалять чужого»: если целевой пользователь ОС не соответствует ожидаемому
      системному пользователю раксд-раскладки (имя; системный аккаунт без оболочки входа) — команда
      отказывает (exit != 0, понятная ошибка) и НИЧЕГО не удаляет (ни пользователя, ни каталоги).
- [ ] AC7: Защита от подозрительных путей: если резолвленный путь к state/config-каталогу пуст, равен `/`,
      `$HOME`, корневому системному каталогу или содержит симлинк, ведущий за пределы ожидаемой раскладки —
      команда отказывает (exit != 0, понятная ошибка) и НИЧЕГО не удаляет.
- [ ] AC8: Безопасность/аудит (SECURITY-BASELINE §3,§4): факт `--purge` (удаление пользователя и каталогов,
      включая `keys.db`) фиксируется в аудит-выводе действия (action, platform, что удалено/что отсутствовало).
      Аудит-запись о факте `--purge` создаётся ДО физического удаления каталогов, чтобы запись не была
      уничтожена вместе с удаляемыми данными/логами. Секреты/содержимое ключей в вывод и логи не попадают.
- [ ] AC9: Барьер необратимости (неинтерактивный, без TTY-зависимости): `--purge` требует дополнительно
      явный флаг `--yes` для фактического удаления. Запуск `--purge` БЕЗ `--yes` ничего не удаляет, печатает
      предупреждение о необратимости (стирание `keys.db`/аудита) и завершается с exit != 0. С `--purge --yes`
      удаление выполняется без интерактивных запросов (инструментом управляют агенты/скрипты, а не человек у TTY).
- [ ] AC10: Тестируемость: поведение покрыто автотестами без вызова реальных системных команд на хосте;
      проверяются обе платформы, оба режима (с `--purge`/без), требование `--yes` (`--purge` без `--yes` →
      ничего не удалено, exit != 0), идемпотентность, отказ при запущенном демоне, отказ без прав, отказ при
      несоответствии пользователя, отказ при подозрительном пути. Тесты выполняются в Docker (baseline §6).

## Out of Scope
- Конкретный механизм удаления и проверки (как именно вызывать `userdel`/`dscl`, как проверять
  соответствие целевого пользователя ожидаемому, порядок шагов, rollback) — это architect/system-dev
  в `plan.md`/`service-design.md`.
- Конкретные платформенные пути каталогов состояния/конфигурации — определяются в `plan.md`/`service-design.md`.
- Финальный дизайн текста вывода и формулировка предупреждения о необратимости — cli-ux (`ux-spec.md`).
- Резервное копирование/экспорт `keys.db` перед удалением — не делаем; `--purge` только удаляет.
- Изменение поведения `install`/`start`/`stop`/`status` — не трогаем.
- Windows — вне scope продукта (только macOS + Linux).
- Удаление установленного бинаря `raxd` и артефактов установщика (`curl | sh`) — это не задача сервиса.

## Open Questions
- None — барьер необратимости решён (`--purge` + обязательный `--yes`, без интерактива; AC9). Конкретные
  механизмы проверки пользователя и удаления намеренно отданы architect/system-dev (см. Out of Scope) —
  это не блокеры уровня PM.
