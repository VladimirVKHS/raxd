# ADR-001: Инструмент воспроизводимой релизной сборки — ручной скрипт поверх Makefile vs goreleaser

## Контекст
spec distribution Q1 / AC8 / AC15: нужно воспроизводимо собирать матрицу из 4 артефактов
(darwin/linux × amd64/arm64) + `SHA256SUMS` ОФЛАЙН из `vendor/` внутри Docker (`proxy.golang.org`
недоступен, baseline §6, STACK §Кросс-компиляция). STACK называет инструментом `goreleaser` v2, но
фактически 4 бинаря уже собирает ручной `Makefile` (`build-all`), а goreleaser в проект не внесён
(нет `.goreleaser.yaml`, нет офлайн-способа его получить в контейнере). Развилку надо разрешить.

## Решение
**Принят вариант B (ручной релизный путь поверх существующего `Makefile`)** как основной и
единственный путь v1. Новые Make-таргеты `release`/`checksums` поверх готового `build-all`:
архивация каждого из 4 бинарей в `tar.gz` (внутри бинарь `raxd` + LICENSE + README) → генерация
`SHA256SUMS` нативным `sha256sum`-форматом (`<hash>␣␣<file>`). Полностью офлайн из `vendor/`, ноль
новых зависимостей, точный контроль формата `SHA256SUMS` (AC3/AC16). **Гибрид C (`.goreleaser.yaml`
как «спящий» артефакт) отклонён**: его ценность (матрица/архивы) уже покрыта `build-all`, а два пути
сборки на бумаге создают риск рассинхрона имён/версий без выигрыша для v1 без remote. **Разрешение
STACK mismatch:** STACK §Кросс-компиляция/§Установка называют goreleaser основным инструментом — он
помечается как **опциональный/неиспользуемый** (по аналогии с `kardianos/service` в service-install
ADR-001); фактический путь сборки — ручные Make-таргеты.

## Альтернативы
- **A: goreleaser v2 как основной путь сейчас** — индустриальный стандарт, `release --snapshot
  --clean` даёт матрицу+архивы+checksums локально без публикации
  (https://goreleaser.com/customization/snapshots/). Хуже: ВСЕ методы установки goreleaser требуют
  сети, `go install ...@latest` требует Go 1.26 (дословно «Requires Go 1.26» на странице установки;
  при этом Go 1.26 на май 2026 ещё НЕ выпущен — актуальная стабильная 1.25, https://go.dev/dl/),
  офлайн-режим OSS не предусмотрен (air-gapped касается только Pro-лицензии). Чтобы остаться hermetic —
  пришлось бы вендорить сам goreleaser или класть его бинарь в образ/репо (доп. сложность поверх
  работающего Makefile). → https://goreleaser.com/getting-started/install/oss/ , https://goreleaser.com/pro/
- **B: ручной скрипт поверх Makefile** (ПРИНЯТО) — офлайн уже работает, ноль новых зависимостей,
  точный контроль `SHA256SUMS`-формата (`sha256sum -c`-совместимый, AC3/AC16). Минус: нет
  «бесплатных» changelog/brew. → прочитан `Makefile`
- **C: гибрид** (ОТКЛОНЁН) — ручная сборка сейчас + `.goreleaser.yaml` на будущее. Цена: два пути на
  бумаге → риск рассинхрона имён/версий (AC16) без выигрыша для v1; при появлении remote
  `.goreleaser.yaml` пишется с нуля (приемлемо, ценность покрыта `build-all`).

## Последствия
- Плюсы: hermetic-сборка без сети сегодня; нет проблемы офлайн-получения goreleaser; полный
  контроль имён артефактов и формата `SHA256SUMS`; переиспользование отлаженных `build-all`/`verify-cross`.
- Минусы: больше ручного кода для архивации/checksum; нет changelog/brew «из коробки» (Q6 всё равно
  откладывает пакеты).
- Стек: STACK §Кросс-компиляция (там goreleaser назван основным) ОБНОВИТЬ → goreleaser помечается
  опциональным/неиспользуемым, фактический путь — ручные Make-таргеты `release`/`checksums`.
