# Spec: MCP Server — MCP-сервер поверх готового HTTP/TLS-транспорта raxd

Автор продукта: Vladimir Kovalev, OEM TECH

## Goal
Дать ИИ-агенту (MCP-клиенту) возможность подключиться к `raxd` по протоколу MCP и пользоваться
инструментами через УЖЕ ГОТОВЫЙ аутентифицированный HTTP/TLS-транспорт (`internal/server`, задача
`tls-transport`). Текущая заглушка-диспетчер (501 «not implemented») заменяется реальным
MCP-эндпоинтом, который доступен за той же аутентификацией, Origin/Host-проверкой, rate-limit и
аудитом, что и остальной транспорт (контракт `tls-transport`). В scope — протокол + транспорт MCP +
аутентификация + аудит + МИНИМАЛЬНЫЙ набор инструментов, доказывающий рабочий MCP-цикл end-to-end и
НЕ зависящий от невыполненных фич. Транспорт НЕ переписывается. Задача цельная и не разбивается:
протокол + транспорт + аутентификация + аудит неделимы для рабочего MCP-цикла (по аналогии с
`tls-transport`). Опирается на `MCP-INTEGRATION.ru.md` и `SECURITY-BASELINE.ru.md §1/§2/§4/§6`.

## User Stories
- Как ИИ-агент (MCP-клиент), я хочу выполнить `initialize`, получить список инструментов и вызвать
  один из них, чтобы убедиться, что MCP-канал к `raxd` рабочий end-to-end.
- Как администратор `raxd`, я хочу, чтобы MCP-эндпоинт поднимался тем же `raxd serve` (тот же порт,
  тот же TLS, тот же ключ), чтобы не открывать второй сетевой канал и не плодить точки входа.
- Как специалист по безопасности, я хочу, чтобы каждый MCP-вызов проходил аутентификацию по
  API-ключу и фиксировался в аудите (fingerprint, метод/инструмент, результат — без секретов),
  чтобы расследовать аномалии.
- Как ИИ-агент, я хочу честный минимальный набор инструментов, который работает уже сейчас и не
  зависит от невыполненных задач `command-exec`/`file-upload`, чтобы проверить канал без блокеров.

## Acceptance Criteria
- [ ] AC1 (MCP-INTEGRATION «Транспорт»): MCP обслуживается по Streamable HTTP — единый эндпоинт,
      POST для запросов клиента (JSON-RPC) и GET для server→client потока (SSE при необходимости).
      Точный маршрут и версия — из Open Questions; тест: POST на маршрут возвращает корректный
      JSON-RPC-ответ, а не 501.
- [ ] AC2 (baseline §1/§2): аутентификация по Bearer-ключу происходит ДО любой обработки
      MCP-запроса; новых каналов аутентификации не вводится. Тест: запрос без `Authorization: Bearer`
      и с неизвестным/отозванным ключом не доходит до MCP-обработки.
- [ ] AC3 (протокол): `initialize` возвращает корректный ответ с согласованием возможностей
      (capabilities), именем сервера `raxd` и версией; клиент после `initialize` видит сервер готовым.
- [ ] AC4 (протокол): `tools/list` возвращает ровно заявленный в этой задаче набор инструментов
      (`ping`, `server_info`), с корректными именами, описаниями и схемами входа.
- [ ] AC5 (протокол): `tools/call` для `ping` возвращает успешный результат «pong» (или эквивалент
      «жив»); вызов выполняется без побочных эффектов на хосте.
- [ ] AC6 (протокол): `tools/call` для `server_info` возвращает структурированный результат с
      версией `raxd` и базовыми сведениями о демоне без секретов (без тел ключей, без приватного
      TLS-ключа, без путей к секретам).
- [ ] AC7 (протокол, edge): вызов несуществующего инструмента и некорректный JSON-RPC-запрос
      возвращают корректную JSON-RPC-ошибку (а не панику и не 501); сервер остаётся работоспособным.
- [ ] AC8 (baseline §1): неаутентифицированный MCP-запрос отвергается на уровне транспорта (401),
      отозванный/неизвестный ключ — тоже (401), сбой/повреждение keystore — отказ (403); ни один из
      этих случаев не доходит до MCP-обработки и не вызывает инструмент.
- [ ] AC9 (baseline §4): каждый MCP-вызов пишет структурированную (JSON) аудит-запись через
      существующий аудит-механизм транспорта: timestamp (UTC), fingerprint ключа через
      `keystore.Fingerprint` (НЕ сам ключ), имя метода/инструмента, результат (success/fail/deny),
      удалённый адрес. Тест: запись содержит fingerprint, имя инструмента и результат.
- [ ] AC10 (baseline §4/§1): в аудит-логе, ответах MCP и сообщениях об ошибках НЕТ тела ключа, его
      хэша, salt и приватного TLS-ключа (тест: предъявленный ключ не встречается подстрокой в
      захваченном логе и в теле MCP-ответов).
- [ ] AC11 (baseline §2): MCP-эндпоинт поднимается тем же `raxd serve` (тот же порт и TLS-серт) —
      отдельного процесса/порта для MCP НЕТ. Тест: после `serve` MCP доступен на
      `https://127.0.0.1:<port>/<mcp-route>`.
- [ ] AC12 (Origin, baseline §2): для браузерных MCP-клиентов валидация заголовка `Origin` работает
      через Origin/Host-проверку транспорта (контракт `tls-transport`); поведение по умолчанию
      (пропуск при отсутствии Origin, 403 при present&invalid) сохраняется. Тест: запрос с невалидным
      `Origin` отвергается до MCP-обработки.
- [ ] AC13 (расширяемость): инструменты `execute_command` и `upload_file` НЕ реализованы, но заложена
      точка расширения для их регистрации, когда появятся задачи `command-exec`/`file-upload`. Их
      отсутствие не ломает `tools/list`.
- [ ] AC14 (baseline §6): все тесты (initialize→capabilities; tools/list набор; ping→pong;
      server_info; неаутентифицированный→401; невалидный JSON-RPC→ошибка; аудит без секретов; Origin)
      зелёные в Docker; сборка/тесты офлайн из `vendor/` (`-mod=vendor`), без `go mod download`.
- [ ] AC15 (подключение): в docs (`tech-writer`) задокументированы параметры подключения MCP-клиента:
      URL `https://127.0.0.1:<port>/<mcp-route>`, заголовок `Authorization: Bearer rax_live_…`,
      оговорка про self-signed TLS (как доверять/обходить проверку серта в dev).

## Требуемые инструменты
- `ping` — проверка живости MCP-цикла; вход пустой, выход «pong».
- `server_info` — версия `raxd` и базовые сведения о демоне (без секретов); закрывает потребность в
  `version` как поле результата, чтобы не плодить тривиальный отдельный tool.

Набор минимален намеренно: оба инструмента честно работают на готовом транспорте и не зависят от
невыполненных `command-exec`/`file-upload`. Этого достаточно, чтобы доказать протокол + транспорт +
аутентификацию + аудит end-to-end; остальные инструменты добавятся отдельными задачами через точку
расширения (AC13).

## Out of Scope
- `execute_command`/`exec` и любое выполнение команд по сети — задача `command-exec` (baseline §3);
  здесь только точка расширения.
- `upload_file`/передача файлов — задача `file-upload`; здесь только точка расширения.
- MCP Resources и Prompts — не входят в эту итерацию (см. Q4); при добавлении — отдельная задача.
- Переписывание/доработка сетевого транспорта (TLS, listener, middleware, rate-limit, аудит) —
  готово в `tls-transport`; здесь только потребление и монтаж MCP-эндпоинта.
- mTLS / клиентские сертификаты — вне scope (как в `tls-transport`); аутентификации по API-ключу
  достаточно для v1.
- Управление ключами (`key create|list|delete`, формат, хранение) — готово в `key-management`; здесь
  только потребление `keystore.Verify`/`Fingerprint`.
- Регистрация системного сервиса, установка, дистрибуция, install.sh, goreleaser — задачи
  `service-install`/`distribution`.
- Финальный визуальный дизайн вывода CLI/статус-блоков — задача `cli-ux`.

## Open Questions
- [ ] Q1 (SDK vs stdlib) — делегировано research-analyst, НЕ решаю как PM. Развилка: официальный Go
      MCP SDK против минимальной реализации MCP JSON-RPC на stdlib. Контекст-ограничение: SDK требует
      `go mod vendor`, а из Docker proxy НЕДОСТУПЕН (vendor делается на ХОСТЕ, ADR-002). Дефолт PM:
      при прочих равных предпочесть официальный SDK ради соответствия протоколу и совместимости с
      `command-exec`/`file-upload`, ЕСЛИ research подтвердит offline/vendor-флоу; иначе — stdlib.
- [ ] Q2 (маршрут эндпоинта) — делегировано architect. Дефолт PM: `/mcp` (как в MCP-INTEGRATION).
- [ ] Q3 (версия протокола MCP) — делегировано research-analyst. Дефолт PM: ориентир `2025-11-25`
      (MCP-INTEGRATION); подтвердить актуальность и точную строку версии в `initialize`.
- [ ] Q4 (Resources/Prompts) — делегировано architect. Дефолт PM: в этой итерации НЕ добавлять;
      пересмотреть, если клиентам нужен `status`/`capabilities` как resources.
- [ ] Q5 (Origin для браузерных MCP-клиентов) — делегировано architect/security. Дефолт PM: сохранить
