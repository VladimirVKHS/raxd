# ADR-002: Инъекция версии/commit/date через `-ldflags -X`

## Контекст
Spec bootstrap-cli (AC: «`raxd version` печатает версию, git-commit и дату сборки; значения
подставляются на этапе сборки (ldflags); при сборке без флагов выводятся осмысленные значения по
умолчанию (напр. `dev`)»). Нужно выбрать канонический, проверяемый механизм подстановки билд-метаданных.

## Решение
Объявить в пакете точки входа три **package-level string**-переменные с осмысленными дефолтами
(например `version` = `dev`, `commit` = `none`, `date` = `unknown`) и подставлять их значения
линкером через флаги вида `-X importpath.name=value` (по одному `-X` на каждую переменную, передаются
в `go build` через `-ldflags`). Значения берутся из git (короткий хэш коммита) и текущей даты сборки.
Требования линкера соблюдаются: переменная типа `string`, уровня пакета, инициализирована
константным строковым литералом (не вызовом функции и не ссылкой на другую переменную).
Конкретную реализацию (код + build-команда) пишет developer; иллюстрация паттерна — в research.md,
раздел Q3 (там она помечена как «не код продукта»).
→ https://pkg.go.dev/cmd/link
→ `specs/bootstrap-cli/research.md` (раздел Q3)
→ https://www.digitalocean.com/community/tutorials/using-ldflags-to-set-version-information-for-go-applications

## Альтернативы
- **`runtime/debug.ReadBuildInfo()` (VCS-стэмпинг Go)**: даёт commit/dirty без ldflags, но семвер-
  версию и произвольную дату так же удобно не задать (особенно при сборке вне tagged-VCS); годится
  как дополнение, не как замена. (Не источник факта об ldflags — отдельный механизм; для AC «версия
  подставляется на этапе сборки» ldflags прямее.)
- **Генерация файла с константами на этапе сборки**: усложняет пайплайн и противоречит «без флагов —
  дефолты»; ldflags решает то же одним флагом.

## Последствия
- Плюсы: канонический, документированный механизм; осмысленные дефолты (`dev`/`none`/`unknown`)
  закрывают сценарий сборки без флагов; имена `version`/`commit`/`date` совместимы с тем, что
  goreleaser инжектит по умолчанию — упростит будущую задачу `distribution`. → https://goreleaser.com
- Минусы/цена: при неверном import-path переменной сборка проходит молча, а значение остаётся
  дефолтным — нужен тест/проверка, что подстановка реально сработала. Имена/пакет переменных надо
  держать в синхроне с build-командой и goreleaser-конфигом.
- Влияние на стек: новых зависимостей нет (stdlib-линкер); согласуется с `CGO_ENABLED=0` и
  goreleaser-матрицей из STACK. → внутренний `.claude/reference/STACK.ru.md`
