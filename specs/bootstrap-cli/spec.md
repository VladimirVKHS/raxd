# Spec: Bootstrap CLI — каркас проекта raxd

## Goal
Создать минимальный компилируемый каркас продукта `raxd`: модуль Go, базовую структуру проекта,
cobra-CLI с корневой командой и деревом подкоманд (часть — рабочие, часть — заглушки), загрузку
конфигурации и путей по XDG, баннер с авторством и `Dockerfile` для воспроизводимой dev/test-среды.
Цель — дать команде рабочий скелет, поверх которого следующие задачи добавляют бизнес-логику, без
самой логики ключей/сети/выполнения команд на этом шаге.

## User Stories
- Как разработчик команды raxd, я хочу собрать и запустить пустой скелет CLI, чтобы наращивать
  функциональность поверх стабильного дерева команд, а не с нуля.
- Как пользователь, я хочу видеть `raxd --help`, `raxd version` и `raxd status`, чтобы понимать
  доступные команды и текущее состояние демона ещё до настройки.
- Как новый контрибьютор, я хочу собирать и тестировать проект одной docker-командой, чтобы не
  запускать потенциально опасный демон на своём хосте.

## Acceptance Criteria
- [ ] Существует `go.mod` с именем модуля `github.com/vladimirvkhs/raxd` и минимальной версией
      `go 1.25`; проект собирается без ошибок командой `go build ./...`.
- [ ] Исходники разнесены по каталогам точки входа и внутренней логики (например `cmd/` + `internal/`);
      внутренние пакеты не импортируются извне модуля.
- [ ] `raxd --help` показывает корневую команду и все подкоманды дерева: `key` (с `create`, `list`,
      `delete`), `config` (с `port`), `serve`, `version`, `status`.
- [ ] Команды-заглушки (`key create|list|delete`, `config port`, `serve`) завершаются с понятным
      сообщением вида «<команда>: not implemented yet» и ненулевым кодом возврата, но присутствуют в
      `--help` и имеют корректные usage-строки. Конкретно `serve` — «честная» заглушка: печатает
      сообщение и завершается с ненулевым кодом, без запуска блокирующего процесса.
- [ ] `raxd version` печатает версию, git-commit и дату сборки; значения подставляются на этапе
      сборки (ldflags), при сборке без флагов выводятся осмысленные значения по умолчанию (напр. `dev`).
- [ ] `raxd status` (заглушка-состояние) печатает статус демона «не запущен» и фактические пути к
      `config.yaml`, будущему `keys.db` и директории TLS; завершается кодом 0.
- [ ] Конфиг и пути вычисляются по XDG (`adrg/xdg` + `viper`) согласно раскладке STACK; на Linux и
      macOS канонический путь конфига — `~/.config/raxd/`; отсутствие `config.yaml` не является
      ошибкой (применяются значения по умолчанию).
- [ ] При первом запуске недостающие директории конфига/состояния/TLS создаются с корректными
      правами по SECURITY-BASELINE (директория состояния и приватные данные — не шире `0700`/`0600`).
- [ ] Вывод содержит баннер с названием продукта и авторством «Vladimir Kovalev, OEM TECH»
      (детальный дизайн вывода — в отдельной задаче cli-ux; здесь фиксируется только наличие и состав).
- [ ] Скелет не содержит секретов и не пишет секреты в лог/вывод (на этом шаге секретов нет, но
      требование выполнимо по построению — нет хардкода ключей/сертификатов).
- [ ] Есть `Dockerfile`, внутри которого успешно проходят `go build ./...` и `go test ./...`;
      команда запуска документирована как docker-команда (SECURITY-BASELINE §6).
- [ ] Юнит-тесты подтверждают: регистрацию всех подкоманд в дереве, ненулевой код у заглушек,
      `raxd version` и `raxd status` завершаются с кодом 0, корректность вычисления XDG-путей.
      Тесты зелёные в Docker.

## Out of Scope
- Реальная логика API-ключей: генерация, хранение, отзыв, сравнение — задача `key-management`.
- TLS, генерация сертификатов, сетевой сервер (TCP/TLS) — задача `tls-transport`.
- Выполнение команд по сети, allowlist, таймауты, аудит-лог — задача `command-exec`.
- MCP-сервер и его транспорт/инструменты — задача `mcp-server`.
- Регистрация системного сервиса (systemd/launchd) и реальный `serve` — задача `service-install`.
- `install.sh` (`curl | sh`), goreleaser, релизы, подпись/нотаризация — задача `distribution`.
- Финальный визуальный дизайн вывода/баннера и таблиц — задача cli-ux (`ux-spec`).
- Rate limiting, аутентификация запросов — относятся к сетевым задачам выше, не к каркасу.

## Resolved Decisions
- D1 (Go-модуль): имя модуля — `github.com/vladimirvkhs/raxd`.
- D2 (версия Go): минимальная версия в `go.mod` — `go 1.25`.
- D3 (macOS-путь конфига): канонический путь — `~/.config/raxd/`, единообразно с Linux.
- D4 (поведение `serve`): «честная» заглушка — печать сообщения и выход с ненулевым кодом, без
  запуска блокирующего процесса.

## Open Questions
- None — все вопросы закрыты пользователем, решения зафиксированы в «Resolved Decisions».
