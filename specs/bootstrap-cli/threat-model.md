# Threat Model: bootstrap-cli — каркас CLI/демона raxd

> Scope: каркас (cobra-дерево с заглушками `key`/`config`/`serve`, `version`/`status`, конфиг + XDG-пути,
> баннер, Dockerfile dev/test). Реальные ключи/TLS/exec/MCP/сервис — в других task-id и здесь
> присутствуют только как заглушки и будущие границы. Модель угроз СОРАЗМЕРНА каркасу: оцениваем то,
> что уже существует на этом шаге, и фиксируем будущие границы baseline как точки расширения,
> которые каркас не должен нарушить.

## Активы

Что реально существует и/или закладывается каркасом, и почему ценно:

- **Каталоги и файлы конфига/состояния/TLS** (`~/.config/raxd/`, `$XDG_STATE_HOME/raxd/`, `tls/`):
  каркас их создаёт (`config.EnsureDirs`). Сами по себе пусты, но это **будущее место хранения
  секретов** (`keys.db`, приватный ключ TLS). Если каркас создаст их со слабыми правами — будущие
  секреты унаследуют дыру.
- **Исполняемый бинарь `raxd` и его сборочная цепочка** (`go build`, Dockerfile, ldflags): целостность
  бинаря и предсказуемость build-метаданных (version/commit/date).
- **Поверхность вывода CLI** (`version`, `status`, баннер, сообщения заглушек): не должна раскрывать
  секреты (их сейчас нет — но требование «нет хардкода/нет утечек» выполнимо по построению) или
  чувствительные пути сверх необходимого.
- **Хост разработчика/контейнера**: каркас не должен открывать порт, принимать соединения или
  исполнять внешние команды (в т.ч. случайно — заглушка `serve`).
- **Конфигурационный ввод** (`config.yaml`, переменные `XDG_*`, `$HOME`): источник путей; парсится viper.

## Поверхность атаки

На этапе каркаса поверхность **минимальна и преимущественно локальна**:

- **CLI** — единственный реальный вход: аргументы команд, `--help`, `version`, `status`.
- **Файловая система** — `config.yaml` (читается viper), переменные окружения `XDG_CONFIG_HOME`,
  `XDG_STATE_HOME`, `$HOME` (влияют на резолв путей в `config.Paths()`).
- **Сборочная среда** — Dockerfile (`golang:1.25`), ldflags при сборке.
- **НЕ присутствует на этом шаге** (явно вне поверхности каркаса): сетевые входы (TCP+TLS),
  HTTP/MCP-эндпоинт, install-скрипт, реальный exec команд, реальные API-ключи. Эти векторы — зоны
  будущих task-id (`tls-transport`, `mcp-server`, `command-exec`, `key-management`, `distribution`,
  `service-install`); ниже зафиксированы как «будущие границы», которые каркас обязан не нарушить.

Кто может атаковать на этапе каркаса:
- **Локальный пользователь / другой процесс на той же машине** — может попытаться прочитать файлы
  конфига/состояния/TLS, если права слишком широки.
- **Тот, кто контролирует окружение сборки/запуска** (CI, образ) — целостность бинаря и build-метаданных.
- Внешний сетевой атакующий, MITM, клиент с/без ключа — **вне scope каркаса** (нет сети), но
  актуальны для будущих задач (зафиксированы в разделе «Остаточные риски / эскалации»).

## Угрозы (STRIDE-ориентир)

- **S**poofing: на каркасе аутентификации/сети нет → прямого вектора нет. Будущая граница: подмена
  клиента / обход аутентификации по ключу — зона `key-management`/`tls-transport`/`mcp-server`.
- **T**ampering: подмена `config.yaml` (битый/вредоносный YAML) → каркас должен корректно падать на
  парсинге, а не вести себя непредсказуемо; подмена бинаря/build-метаданных — зона `distribution`.
- **R**epudiation: аудит-лога на каркасе нет (нет действий, требующих аудита) → не применимо сейчас;
  будущая граница — аудит каждого exec/auth (зона `command-exec`/`mcp-server`).
- **I**nformation disclosure: утечка секретов в выводе CLI/логах/commit (актуально уже сейчас —
  по построению: нет хардкода ключей/сертификатов); раскрытие чувствительных путей в `status` сверх
  необходимого.
- **D**enial of service: на каркасе нет сети/ресурсов под нагрузкой → не применимо; rate-limit —
  будущая граница (зона сетевых задач).
- **E**levation of privilege: создание каталогов/файлов со слишком широкими правами (актуально
  сейчас — `EnsureDirs`/umask); запуск демона от root и shell-инъекция в exec — будущие границы
  (зоны `service-install`/`command-exec`), каркас не должен их предвосхищать небезопасно (заглушка
  `serve` не запускает процесс и не повышает привилегии).

## Риски (риск → вероятность/влияние → смягчение)

Актуальные для каркаса (закрываются на этом шаге):

- [ ] Риск: `config.EnsureDirs` / `MkdirAll` создаёт `ConfigDir`/`StateDir`/`TLSDir` со слишком широкими
      правами (например `0755`), и будущие секреты (`keys.db`, приватный ключ TLS) попадают в
      world-/group-читаемые каталоги → вероятность средняя / влияние высокое (компрометация будущих
      секретов локальным пользователем) → смягчение: каталоги создаются строго `0700`; тест проверяет
      фактический режим после `EnsureDirs`. (baseline §1, §2, раскладка STACK)
- [ ] Риск: реальный режим директорий искажён процессным `umask` (даже при запросе `0700` umask может
      снять биты, но опаснее — при будущем создании файлов `0600` umask не добавит лишнего; основной
      риск — что код полагается на umask вместо явных прав) → вероятность средняя / влияние среднее →
      смягчение: права задаются явным аргументом режима у `MkdirAll`/файловых операций, не зависят от
      umask; требование зафиксировано как контракт для будущих файлов `0600`. (baseline §2)
- [ ] Риск: заглушка `serve` случайно открывает порт / запускает блокирующий процесс / слушает
      соединения → вероятность низкая / влияние высокое (немой сетевой вход вне модели угроз каркаса,
      запуск потенциально опасного демона на хосте) → смягчение: `serve` — «честная» заглушка (D4):
      печать сообщения + ненулевой код, без `net.Listen`, без блокировки; тест проверяет ненулевой код
      и отсутствие блокировки. (baseline §3, §6)
- [ ] Риск: `status` раскрывает чувствительные абсолютные пути / содержимое будущих секретов сверх
      необходимого → вероятность низкая / влияние низкое-среднее → смягчение: `status` печатает только
      пути (`config`, `keys`, `tls`) и состояние `not running`, без чтения/печати содержимого `keys.db`
      или ключей; секретов на каркасе нет, печать путей — осознанный минимум для диагностики. (baseline §4)
- [ ] Риск: секрет/токен/сертификат захардкожен в исходниках, баннере, дефолтах config или попадает в
      вывод/лог → вероятность низкая / влияние высокое → смягчение: по построению нет хардкода ключей/
      сертификатов; дефолтный `config.yaml` не содержит секретов; проверяется grep по репозиторию и по
      выводу `version`/`status`/баннера на отсутствие паттернов секретов. (baseline §4)
- [ ] Риск: битый/вредоносный `config.yaml` приводит к непредсказуемому поведению вместо явной ошибки
      → вероятность низкая / влияние низкое → смягчение: `config.Load()` возвращает ошибку только при
      битом YAML; отсутствие файла НЕ ошибка (дефолты); тест на битый YAML → ошибка, на отсутствие
      файла → дефолты. (baseline §4 — устойчивость)
- [ ] Риск: сборка/тесты/запуск `raxd` выполняются на хосте, а не в Docker, нарушая изоляцию мощного
      инструмента → вероятность средняя / влияние высокое → смягчение: есть `Dockerfile`, внутри
      которого проходят `go build ./...` и `go test ./...`; команда запуска документирована как
      docker-команда; на хосте `raxd` не запускается. (baseline §6)
- [ ] Риск: build-метаданные (version/commit/date) не подставляются (неверный import-path ldflags) и
      молча остаются пустыми, маскируя неизвестную сборку → вероятность низкая / влияние низкое →
      смягчение: осмысленные дефолты (`dev`/`none`/`unknown`), юнит-тест `version.Info()` фиксирует
      формат и дефолты. (косвенно baseline §5 — целостность дистрибуции)
- [ ] Риск: `$HOME` недоступен / `XDG_*` указывает на неожиданный путь → каркас создаёт каталоги в
      неожиданном месте или падает неинформативно → вероятность низкая / влияние низкое → смягчение:
      `config.Paths()` уважает `XDG_CONFIG_HOME`/`XDG_STATE_HOME`, иначе `$HOME/...`; ошибка только при
      недоступном `$HOME`; тест проверяет резолв путей с заданным и пустым `XDG_*`. (baseline §1/§2 раскладка)

## Остаточные риски / эскалации

Пункты baseline, **не реализуемые на каркасе** (нет соответствующих сущностей). Это НЕ нарушения, а
зоны будущих task-id; зафиксированы как границы, чтобы каркас их не предвосхитил небезопасно и чтобы
последующие роли их не пропустили. Эскалации, требующей решения пользователя до релиза каркаса, нет —
все пункты ниже логически отложены вместе со своей функциональностью по решению PM (`Out of Scope` в
spec.md). Каждый помечен компенсирующим контролем на уровне каркаса.

- [ ] baseline §1 (генерация/хранение/сравнение ключей, constant-time): нет логики ключей на каркасе
      (`key *` — заглушки) → компенсирующий контроль: заглушки не реализуют небезопасный суррогат
      (никаких `==` по «ключам», никакого хранения в открытом виде); поля путей (`KeysDB`) заведены
      без логики. Зона будущей задачи `key-management`. Эскалация не требуется (отложено по spec
      `Out of Scope`).
- [ ] baseline §2 (TLS обязателен, `MinVersion`, self-signed серт, валидация Origin, bind): нет сети
      и TLS на каркасе → компенсирующий контроль: `TLSDir` создаётся `0700` заранее, чтобы будущий
      приватный ключ лёг в защищённый каталог; `serve`-заглушка не открывает сокетов. Зона будущих
      задач `tls-transport`/`mcp-server`. Эскалация не требуется (отложено по spec).
- [ ] baseline §3 (exec без shell-интерполяции, таймаут, allowlist, не от root, ограниченное окружение):
      нет выполнения команд на каркасе → компенсирующий контроль: ни одна заглушка не вызывает
      `exec.Command`/`sh -c`; `serve` не запускает процесс. Зона будущих задач `command-exec`/
      `service-install`. Эскалация не требуется (отложено по spec).
- [ ] baseline §4 (аудит-лог, логирование отказов, rate-limit, graceful restart): нет действий,
      требующих аудита, и нет сети → компенсирующий контроль: применяется только подпункт «секреты не
      в логах/выводе/commit» (см. актуальные риски). Зоны `command-exec`/`mcp-server`/`service-install`.
      Эскалация не требуется (отложено по spec).
- [ ] baseline §5 (install-скрипт, SHA256SUMS, подпись/нотаризация macOS): нет дистрибуции на каркасе →
      компенсирующий контроль: build-метаданные осмысленны и тестируемы (основа целостности). Зона
      будущей задачи `distribution`. Эскалация не требуется (отложено по spec).
- [ ] Замечание для STACK-owner (наследовано из plan.md Trade-offs, НЕ блокер bootstrap-cli):
      путь lipgloss в STACK устарел (`github.com/charmbracelet/lipgloss/v2` — beta; стабильный —
      `charm.land/lipgloss/v2`). На безопасность каркаса не влияет (lipgloss в каркасе не используется),
      но фиксируется как открытый вопрос синхронизации STACK для cli-ux. Эскалация — к research/STACK-owner.

None из перечисленного не является молчаливым пропуском обязательного пункта: каждый отложенный пункт
сопровождён компенсирующим контролем уровня каркаса и привязан к конкретному будущему task-id из
`spec.md` (`Out of Scope`).
