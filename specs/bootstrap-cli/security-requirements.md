# Security Requirements: bootstrap-cli — каркас CLI/демона raxd

> Каждое требование проверяемо (можно ответить «выполнено/нет» одним вопросом) и ссылается на пункт
> `SECURITY-BASELINE.ru.md`, а также на AC из `spec.md` и/или модуль/контракт из `plan.md`.
> Эти требования обязаны выполнить **developer** (реализация) и подтвердить **qa** (тесты в Docker).
> Соответствие проверяют **reviewer** и **security-guardian**.
>
> Scope каркаса минимален: ниже разделены **активные требования** (проверяются на этом шаге) и
> **будущие границы** (требования-заглушки, которые каркас обязан не нарушить; полная реализация — в
> других task-id). Привязки: AC = пункт Acceptance Criteria из spec.md; контракт = из plan.md.

## Права и расположение файлов/директорий (baseline §1/§2, раскладка STACK) — АКТИВНО

- [ ] `config.EnsureDirs(p Paths)` создаёт `ConfigDir`, `StateDir`, `TLSDir` с режимом строго `0700`
      (не шире); проверяемо: тест читает `os.Stat(...).Mode().Perm()` каждого каталога после вызова и
      сравнивает с `0700`. (baseline §1/§2, AC «директории создаются с корректными правами», контракт
      `config.EnsureDirs`)
- [ ] `EnsureDirs` идемпотентен (`MkdirAll`): повторный вызов на существующих каталогах не падает и не
      расширяет права; проверяемо: тест вызывает `EnsureDirs` дважды → ошибки нет, права остаются `0700`.
      (baseline §2, контракт `config.EnsureDirs`)
- [ ] Права задаются явным аргументом режима у файловых операций, а не делегируются процессному `umask`;
      проверяемо: тест, выставив нестандартный `umask` перед вызовом, всё равно получает каталоги не
      шире `0700` (или код-ревью подтверждает явный режим в вызове). (baseline §2)
- [ ] Контракт прав для будущих файлов зафиксирован и не нарушается каркасом: файлы внутри
      `StateDir`/`TLSDir` (когда появятся: `keys.db`, приватный ключ TLS) — `0600`; на каркасе таких
      файлов нет → проверяемо: каркас не создаёт в этих каталогах файлов шире `0600` (grep/код-ревью:
      нет создания файлов с режимом > `0600` в `internal/config`). (baseline §1/§2, раскладка STACK)
- [ ] `config.Paths()` резолвит пути по XDG: `ConfigDir = $XDG_CONFIG_HOME/raxd` при заданной
      переменной, иначе `$HOME/.config/raxd`; `StateDir` — через `$XDG_STATE_HOME`/`$HOME/.local/state`;
      ошибка только при недоступном `$HOME`; проверяемо: табличный тест с заданными и пустыми `XDG_*`
      сверяет результат. (baseline §1/§2 раскладка, AC «пути по XDG», контракт `config.Paths`, D3)

## Отсутствие секретов в выводе/логах/конфиге (baseline §4) — АКТИВНО

- [ ] В исходниках нет хардкода ключей/токенов/сертификатов/приватных ключей; проверяемо: grep по
      репозиторию на паттерны секретов (`rax_live_`, `BEGIN PRIVATE KEY`, base64-подобные литералы в
      коде) — пусто. (baseline §4, AC «скелет не содержит секретов»)
- [ ] Дефолтный `config.yaml`/дефолты viper не содержат секретов (только несекретные значения, напр.
      `Port`); проверяемо: ревью дефолтной структуры `Config` и примера config — секретов нет.
      (baseline §4, контракт `config.Load`)
- [ ] Вывод `raxd version`, `raxd status` и баннера не содержит секретов; проверяемо: тест/ревью
      перехватывает stdout+stderr указанных команд и grep-ом подтверждает отсутствие паттернов
      секретов. (baseline §4, AC «не пишет секреты в лог/вывод»)
- [ ] `status` печатает только пути (`config`, `keys`, `tls`) и состояние `not running`, не читает и не
      печатает содержимое будущего `keys.db` или ключей; проверяемо: ревью `status` Run + тест: вывод
      содержит метки `state/config/keys/tls`, не содержит чтения файлов состояния. (baseline §4, AC
      «status печатает пути», контракт `status` Run)
- [ ] Баннер выводится на stderr (не засоряет машиночитаемый stdout) и содержит только название
      продукта, описание и автора `Vladimir Kovalev, OEM TECH`, без чувствительных данных; проверяемо:
      тест проверяет наличие строки автора и канал вывода (stderr). (baseline §4, AC «баннер с
      авторством», контракт `banner.Render`)

## Среда сборки/тестов/запуска — только Docker (baseline §6) — АКТИВНО

- [ ] В репозитории есть `Dockerfile`, внутри которого успешно проходят `go build ./...` и
      `go test ./...`; проверяемо: сборка образа и прогон тестов в контейнере завершаются успешно.
      (baseline §6, AC «есть Dockerfile … go build/go test», контракт `Dockerfile`)
- [ ] Команда запуска тестов/сборки документирована как docker-команда (не как хостовый `go ...`);
      проверяемо: в доке/README/Dockerfile-комментарии присутствует docker-команда запуска.
      (baseline §6, AC «команда запуска документирована как docker-команда»)
- [ ] `raxd` не предполагает запуска на хосте: тесты исполняются в контейнере, заглушка `serve` не
      запускает демон; проверяемо: тесты зелёные в Docker; нет инструкции «запусти raxd на хосте».
      (baseline §6, AC «тесты зелёные в Docker»)

## Заглушки и поведение каркаса (baseline §3) — АКТИВНО

- [ ] Заглушка `serve` не открывает сетевой порт, не вызывает `net.Listen`/`http.ListenAndServe` и не
      блокируется: печатает сообщение и завершается с ненулевым кодом; проверяемо: тест проверяет
      ненулевой exit и отсутствие зависания/прослушивания (нет `net.Listen` в `internal/cli/serve.go`).
      (baseline §3/§6, AC «serve — честная заглушка», контракт `serve.go`, D4)
- [ ] Ни одна команда/заглушка каркаса не вызывает `exec.Command`/`sh -c`/`os/exec`; проверяемо: grep
      по `internal/` на `exec.Command`, `os/exec`, `sh -c` — пусто. (baseline §3, контракт заглушек)
- [ ] Заглушки `key create|list|delete` и `config port` возвращают ошибку `not implemented yet` с
      ненулевым кодом и не реализуют небезопасный суррогат логики ключей (нет хранения «ключей» в
      открытом виде, нет сравнения через `==`); проверяемо: тест на ненулевой код + ревью отсутствия
      какой-либо обработки секретов в заглушках. (baseline §1/§3, AC «команды-заглушки … ненулевой код»,
      контракт `newStub`)
- [ ] Каркас не повышает привилегии и не предполагает запуск от root: нет setuid/CAP-логики, нет
      вызовов, требующих root, в коде каркаса; проверяемо: ревью — отсутствие операций, требующих
      привилегий (всё в пользовательском `$HOME`/`$XDG_*`). (baseline §3)
- [ ] `config.Load()` корректно обрабатывает ввод: отсутствие `config.yaml` → дефолты (не ошибка),
      битый YAML → явная ошибка; проверяемо: два теста (нет файла → дефолты; битый YAML → ошибка).
      (baseline §4 устойчивость, AC «отсутствие config.yaml не ошибка», контракт `config.Load`)

## Будущие границы (baseline §1-§5) — ТРЕБОВАНИЯ-ЗАГЛУШКИ, реализация в других task-id

> Эти пункты НЕ реализуются в bootstrap-cli (соответствующих сущностей нет — см. spec `Out of Scope`).
> Они зафиксированы как контракт-границы: каркас обязан **не противоречить** им и заложить точки
> расширения. Полная реализация и проверка — в указанных task-id. Развёрнутое обоснование и
> компенсирующие контроли — в `threat-model.md`, раздел «Остаточные риски / эскалации».

- [ ] (Граница) Аутентификация ключей: `crypto/rand` ≥128 бит, хранение `sha256(key+salt)`+salt,
      сравнение constant-time (`crypto/subtle`/`hmac.Equal`), показ ключа один раз, мгновенный отзыв.
      Каркас: точка расширения — поле `KeysDB` в `Paths`/`Config`, заглушки `key *` без логики.
      Реализация — task-id `key-management`. (baseline §1)
- [ ] (Граница) TLS обязателен, `MinVersion` ≥ TLS 1.2 (цель 1.3), self-signed серт `crypto/x509`,
      приватный ключ `0600`, валидация `Origin` для HTTP/MCP, осознанный bind. Каркас: `TLSDir`
      создаётся `0700` заранее. Реализация — task-id `tls-transport`/`mcp-server`. (baseline §2)
- [ ] (Граница) Выполнение команд: `exec.Command(bin, args...)` без shell-интерполяции, таймаут через
      `context`, опциональный allowlist, демон не от root, ограниченное окружение. Каркас: ни одной
      реализации exec. Реализация — task-id `command-exec`/`service-install`. (baseline §3)
- [ ] (Граница) Аудит-лог каждого действия (JSON + ротация), логирование отказов аутентификации,
      rate-limit per-key/per-IP, graceful restart. Каркас: только подпункт «нет секретов в логах/выводе»
      применён. Реализация — task-id `command-exec`/`mcp-server`/`service-install`. (baseline §4)
- [ ] (Граница) Дистрибуция: install-скрипт (`set -euo pipefail`, тело в функции, `trap`), проверка
      `SHA256SUMS`, подпись/нотаризация macOS (минимум — снятие quarantine + инструкция). Каркас:
      build-метаданные осмысленны и тестируемы. Реализация — task-id `distribution`. (baseline §5)

## Эскалации

- None. Все пункты baseline, неприменимые на каркасе, логически отложены вместе со своей
  функциональностью (PM-решение в `spec.md`, раздел `Out of Scope`) и сопровождены компенсирующими
  контролями уровня каркаса (см. `threat-model.md`). Эскалации, требующей решения пользователя до
  релиза каркаса, по безопасности нет.
- Замечание (не блокер, не относится к безопасности каркаса): рассинхрон пути lipgloss в STACK —
  открытый вопрос для research/STACK-owner (см. plan.md Trade-offs); на bootstrap-cli не влияет.
